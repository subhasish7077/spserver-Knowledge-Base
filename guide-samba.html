Samba File Sharing

Access spserver files from Windows PC like a network drive — drag, drop, edit directly over the local network at full LAN speed. No internet required, no Cloudflare needed.

01 — Concept
What is Samba

Samba is a free, open-source implementation of the SMB (Server Message Block) protocol for Linux. SMB is the native file sharing protocol that Windows uses — it is what powers network drives, shared folders, and printer sharing in Windows environments. By running Samba on spserver, the server speaks Windows's own language — it shows up in File Explorer just like any other Windows machine on the network.

The result: you can browse spserver's folders, drag files in and out, open documents directly, and edit them — all at full local network speed without going through the internet. A 1GB file transfers in about 8-10 seconds over ethernet, versus minutes over Cloudflare.

ℹ Samba is local network only. SMB ports (139, 445) are blocked by most ISPs and corporate firewalls — you cannot reach Samba from outside your home network. For remote file access, use Filebrowser (fileserver.dustyserver.co.in) instead.
02 — Protocol
What is SMB and why Linux needs Samba to speak it

SMB (Server Message Block) was developed by IBM in the 1980s, later refined by Microsoft, and became the standard for Windows file sharing. Windows understands SMB natively. Linux does not — SMB is not built into the Linux kernel. Samba is the software that bridges this gap, implementing the SMB protocol from scratch on Linux so that Windows machines can talk to it seamlessly.

Windows PC ──── SMB protocol ──── local network ──── spserver running Samba File Explorer sees: \\192.168.31.150\shared Samba presents it as: a Windows-compatible network share Actual files live at: /home/sipu/shared/ on Linux filesystem

Samba handles two ports: 139 (NetBIOS session service, older) and 445 (SMB directly over TCP, modern). Windows uses port 445 when connecting to a modern Samba server.

03 — Architecture
How Samba works on spserver
spserver runs two Samba daemons: smbd ← handles file sharing — ports 139 and 445 reads /etc/samba/smb.conf for share definitions authenticates users against Samba's own password database manages file locks, permissions, sessions nmbd ← NetBIOS name service — port 137/138 advertises spserver name on the local network enables Windows to find the server by name (\\spserver) less important on modern networks — direct IP works fine
ℹ Samba has its own password database separate from Linux system passwords. A user must exist as a Linux user AND have a Samba password set. Same username, but the passwords can be different. Set with sudo smbpasswd -a USERNAME.

05 — Alternatives
Samba vs other file access methods
Samba local
Full network drive. Drag-drop, edit-in-place. LAN speed (~100Mbps). Works offline. Windows native. Local network only.
Filebrowser remote
Browser-based file manager. Upload/download only — no edit-in-place. Works from anywhere via Cloudflare Tunnel. Slower than Samba.
SFTP/SCP remote
SSH-based file transfer. Works via Tailscale from anywhere. Command-line or apps like WinSCP. No native Windows drive letter. Fast and secure.

01 — Install
Step 1 — Install Samba
1
Install and verify
▼

The samba package installs both smbd (file sharing) and nmbd (name resolution) daemons. It also creates the default config file at /etc/samba/smb.conf.

bash
sudo apt install samba -y

# Verify version installed
samba --version
# Should print: Version 4.19.x

# Verify smbd is running
sudo systemctl status smbd

# Enable auto-start on boot
sudo systemctl enable smbd
02 — Folders
Step 2 — Create shared directories
2
Create folders and set permissions
▼

The directories must exist before Samba can share them. Permissions must allow the sipu user to read and write, and Samba to access the files on their behalf.

bash
# Create the three shared directories
mkdir -p /home/sipu/shared
mkdir -p /home/sipu/certs
mkdir -p /home/sipu/projects

# Set ownership to sipu
sudo chown -R sipu:sipu /home/sipu/shared
sudo chown -R sipu:sipu /home/sipu/certs
sudo chown -R sipu:sipu /home/sipu/projects

# Set permissions: owner full access, others read-only
sudo chmod 755 /home/sipu/shared /home/sipu/certs /home/sipu/projects

04 — Authentication
Step 4 — Set Samba password for sipu
4
Create Samba password
▼

Samba maintains its own password database — separate from the Linux system password. The Linux user sipu must already exist (it does), and you run smbpasswd -a to add them to Samba's database with a Samba-specific password. This can be the same as the Linux password or different.

bash
# Add sipu to Samba's password database
# -a flag = add new user
sudo smbpasswd -a sipu
# Enter password: (type your chosen Samba password)
# Retype password:
# Added user sipu.

# Verify sipu is in the Samba database
sudo pdbedit -L
# Should list: sipu:1000:Sipu
05 — Firewall
Step 5 — Open Samba ports in UFW
5
Allow Samba through UFW
▼

The Samba UFW app profile opens all required ports at once — 137 (UDP, NetBIOS name), 138 (UDP, NetBIOS datagram), 139 (TCP, NetBIOS session), and 445 (TCP, SMB). The most important are 445 and 139 — those are what Windows uses to actually connect.

bash
# The 'Samba' app profile opens all required ports
sudo ufw allow samba

# Verify rules were added
sudo ufw status
# Should show: Samba   ALLOW   Anywhere
06 — Connect
Step 6 — Connect from Windows
6
One-time access via File Explorer
▼

Type the server address directly into File Explorer's address bar. Windows sends an SMB connection request to spserver — Samba responds, Windows asks for credentials, and you're in.

Windows File Explorer address bar
\\192.168.31.150

Windows will ask for a username and password — enter sipu and your Samba password. You'll see the three shares: shared, certs, projects.

7
Map as a persistent network drive
▼

Mapping as a persistent drive gives you a drive letter (like Z:) that appears every time you open File Explorer — no need to type the IP again. Check "Reconnect at sign-in" to make it permanent across reboots.

Windows — steps
# Method 1: File Explorer GUI
# Right-click "This PC" → "Map network drive"
# Drive: Z:  (or any letter)
# Folder: \\192.168.31.150\shared
# ✓ Reconnect at sign-in
# ✓ Connect using different credentials
# Enter: sipu / your Samba password

# Method 2: PowerShell (faster)
net use Z: \\192.168.31.150\shared /user:sipu YourSambaPassword /persistent:yes
01 — Full config
Share definitions added to /etc/samba/smb.conf

These are appended at the bottom of the existing smb.conf file — do not replace the whole file, just add these blocks after the existing content.

/etc/samba/smb.conf — append at bottom
# ── spserver shares ──────────────────────

[shared]
   path = /home/sipu/shared
   browseable = yes
   writable = yes
   valid users = sipu
   create mask = 0644
   directory mask = 0755

[certs]
   path = /home/sipu/certs
   browseable = yes
   writable = no
   valid users = sipu

[projects]
   path = /home/sipu/projects
   browseable = yes
   writable = yes
   valid users = sipu
   create mask = 0644
   directory mask = 0755
02 — Line by line
Every directive explained
DirectiveWhat it does
[shared]The share name in square brackets. This is what Windows sees — \\192.168.31.150\shared. The name can be anything but should be short and descriptive.
path = /home/sipu/sharedThe actual filesystem path this share exposes. The directory must exist and be readable by the valid users.
browseable = yesShows this share in the list when you browse to \\192.168.31.150. Set to no to create a hidden share — still accessible by direct path but not shown in listings.
writable = yesAllow users to create, modify, and delete files. Set to no for a read-only share like [certs].
valid users = sipuOnly users in this list can access the share. Must be a Samba user (added with smbpasswd). Comma-separated for multiple: sipu, john.
create mask = 0644Permissions applied to new files created through Samba. 0644 = owner read/write, group and others read-only. Prevents accidentally creating executable files.
directory mask = 0755Permissions applied to new directories created through Samba. 0755 = owner full access, others read and execute (can enter directory).
03 — More options
Other useful smb.conf directives
DirectiveEffect
guest ok = yesAllow access without a password. Anyone on the network can read (and write if writable=yes). Only use on a fully trusted private network.
read only = yesAlternative to writable = no. Makes the share read-only for everyone in valid users.
write list = sipuOnly users in this list can write — others in valid users get read-only. Useful for multi-user setups with different permission levels.
force user = sipuAll file operations happen as this Linux user regardless of who connected. Simplifies permissions on shared directories.
comment = My shareDescription shown in Windows when browsing the server — appears next to the share name in File Explorer.
max connections = 5Limit simultaneous connections to this share.
hide dot files = yesHide Linux hidden files (starting with .) from Windows clients — cleaner view.
vfs objects = recycleEnable a recycle bin for this share — deleted files go to a .recycle folder instead of being permanently deleted immediately.

01 — What went wrong
Blockers hit during Samba setup
⚠ Blocker 1
Windows couldn't see the server — network discovery not finding \\spserver
After setting up Samba, tried browsing to \\spserver in File Explorer. Windows showed "Windows cannot access \\spserver" or nothing appeared in Network. Expected the server to appear automatically.
✓ Fix
Use IP address instead of hostname — \\192.168.31.150
NetBIOS name resolution (what makes \\spserver work) depends on nmbd and network broadcast, which can be unreliable on modern networks. The fix is straightforward — use the static IP directly: \\192.168.31.150. Since we configured a static IP, this always works and is more reliable than hostname-based discovery. Map drives by IP, not hostname.
⚠ Blocker 2
Access denied when trying to write files — permissions error
Could browse and read files through the Samba share but couldn't create new files or edit existing ones. Windows showed "You don't have permission to perform this action" when trying to save or copy files into the share.
✓ Fix
Directory ownership and Samba writable setting
Two things needed fixing: (1) The shared directory was owned by root, not sipu — Samba was acting as sipu but couldn't write to root-owned directories. Fix: sudo chown -R sipu:sipu /home/sipu/shared. (2) Double-checked that writable = yes was set in smb.conf for the [shared] section. After both fixes and an smbd restart, file creation worked immediately.
⚠ Blocker 3
testparm showed errors — smb.conf syntax issue
After editing smb.conf, running testparm showed warnings about unknown parameters. One of the directive names was misspelled — browsable instead of browseable (note the extra 'e'). Samba ignored the unknown directive silently.
✓ Fix
Always run testparm — it catches misspelled directives
Fixed the spelling to browseable = yes and ran testparm again — no warnings. The lesson: always run testparm before restarting smbd. Unlike many config files, Samba doesn't fail to start on unknown directives — it just silently ignores them, which makes bugs hard to find without testparm.
01 — Issues and fixes
Common Samba problems
Windows cannot access \\192.168.31.150 — no response
Samba isn't running or UFW is blocking. Check: (1) sudo systemctl status smbd — is it running? (2) sudo ufw status — is Samba allowed? (3) Test from server itself: smbclient -L localhost -U sipu. (4) Ping the server first: ping 192.168.31.150 — if ping fails, it's a network issue, not Samba.
Logon failure — wrong credentials
The Samba password is different from the Linux password. Common mistake: setting the Samba password then forgetting it. Reset with: sudo smbpasswd sipu. Also check that the user is in the Samba database: sudo pdbedit -L — if sipu isn't listed, run sudo smbpasswd -a sipu to add them.
Can browse but cannot write — permission denied
Check three things: (1) Is writable = yes in smb.conf for this share? (2) Does sipu own the directory? ls -la /home/sipu/ — all shared dirs should show sipu sipu as owner. (3) Do the directory permissions allow writing? chmod 755 is the minimum — or chmod 775 if group write is needed.
Share not visible when browsing \\192.168.31.150
Check: (1) browseable = yes in the share config. (2) Run testparm — is the share listed in the output? (3) Did you restart smbd after editing smb.conf? (4) Is the sipu user valid for that share? valid users = sipu must be set or the share won't be shown to that user.
Slow file transfer speeds
Expected ~100Mbps over ethernet, getting much less. Common causes: (1) WiFi instead of ethernet — WiFi introduces latency and variable speeds. Use ethernet for spserver. (2) USB ethernet adapter bottleneck — USB 2.0 max is ~40MB/s, USB 3.0 is faster. (3) SMB signing overhead — add server signing = disabled to the [global] section in smb.conf (slightly less secure but faster on trusted LAN).
Access Samba remotely via Tailscale
SMB ports (139, 445) are blocked by the internet and most firewalls. But Tailscale creates a virtual LAN — over Tailscale, spserver is at 100.x.x.x. Connect from any Tailscale device using the Tailscale IP: \\100.x.x.x\shared in File Explorer. Works from mobile, work PC, anywhere with Tailscale installed.
01 — What's next
Future scope
⟶ Things to explore from here
Tailscale + Samba remote access — install Tailscale on your phone or work laptop, then access \\100.x.x.x\shared from anywhere. Works over mobile data. No VPN setup required beyond Tailscale. Full Samba functionality remotely.
Samba recycle bin — add vfs objects = recycle and configure a recycle bin for the shared folder. Deleted files go to a hidden .recycle directory instead of being gone immediately. Safety net for accidental deletions.
Multiple Samba users — create additional Linux users (sudo adduser john) and add them to Samba (sudo smbpasswd -a john). Give each user access to only their own share by setting valid users = john on specific shares.
Time Machine over Samba — macOS supports Time Machine backups to Samba shares. Add vfs objects = catia fruit streams_xattr and fruit:time machine = yes to a share config. Your Mac can then back up directly to spserver.
Samba as a domain controller — Samba can act as an Active Directory domain controller for a small office network. Centralised user management, login credentials, group policies. Massive scope expansion — relevant if spserver grows beyond personal use.
NFS as alternative — NFS (Network File System) is Linux's native file sharing protocol. Faster than Samba for Linux-to-Linux transfers, no authentication overhead. Set up NFS if you ever add more Linux machines to the home network.
