CG-NAT Explained

Why Airtel makes it physically impossible to host services using port forwarding — and the two tools that bypass it entirely: Tailscale for private access and Cloudflare Tunnel for public services.

01 — Foundation
What is NAT

NAT stands for Network Address Translation. IPv4 only has about 4 billion addresses, but the world has far more internet-connected devices. NAT allows multiple devices to share a single public IP address.

Your home router performs NAT. Every device (PC, phone, spserver) has a private IP (192.168.31.x). When a device makes an outgoing request, the router translates the private IP to its public IP, sends the request, and routes the response back to the right device.

Your PC (192.168.31.7) ──► visits google.com Router (192.168.31.1) translates: 192.168.31.7 → 106.215.x.x (public IP) Google responds to: 106.215.x.x Router delivers back to: 192.168.31.7 Multiple devices share one public IP — outgoing works fine Problem: unsolicited INCOMING connections have no NAT table entry → dropped

Port forwarding solves the incoming problem by telling the router "forward any packet arriving on port 22 to 192.168.31.150". This works — but only if you control the router that holds the real public IP. With CG-NAT, you don't.

02 — The problem
What is CG-NAT

CG-NAT stands for Carrier Grade NAT. It is NAT performed by Airtel's own infrastructure — before traffic even reaches your router. Airtel shares a single real public IP among hundreds of customers. Your router's WAN IP is in the 100.64.0.0/10 range — IANA-reserved for CG-NAT. Our router shows 100.220.x.x.

Normal NAT: Internet ──► 106.215.x.x (YOUR router) ──► 192.168.31.150 You control this ↑ Port forwarding works here CG-NAT (what Airtel gives): Internet ──► 106.215.x.x (AIRTEL servers) ──► 100.220.x.x (your router WAN) ──► spserver Airtel controls this ↑ Traffic dropped here ↑ Port forwarding here is useless
✗ When your router WAN IP starts with 100.x.x.x (range 100.64.0.0–100.127.255.255) — you have CG-NAT. No router configuration changes this. The first NAT layer is Airtel's infrastructure, which you cannot touch.

Why? IPv4 addresses ran out. ISPs use CG-NAT to share their limited IPv4 pool across many customers — it reduces Airtel's costs significantly. The downside: customers cannot accept incoming connections from the internet.

03 — Why it fails
Why port forwarding cannot work
External device ──► sends packet to 106.215.x.x:22 ↓ Arrives at Airtel CG-NAT — no mapping for this Packet dropped ✗ ↓ Your router ← never reached (port 22 → 192.168.31.150 rule exists but useless) spserver ← never reached (sshd running and ready — but unreachable)

The packet never gets past Airtel's layer. Your router's port forwarding rules, your UFW rules, sshd being perfectly configured — none of it matters. The packet is dropped before your hardware sees it.

How we confirmed CG-NAT

Two checks confirmed CG-NAT definitively — no special tools needed.

# Returned: 106.215.x.x  ← Airtel's shared public IP
Router admin panel — Check 2: WAN IP

✗ Key signal: curl ifconfig.me returned 106.215.x.x but router WAN showed 100.220.x.x. Two different IPs = a NAT layer between them that you don't control = CG-NAT.

✓ Root cause found
CG-NAT — Airtel's infrastructure, not fixable by us

No configuration on our routers can change this. The solution must use only outgoing connections — which both Tailscale and Cloudflare Tunnel do.

01 — Diagnostics
Commands to detect and diagnose CG-NAT
curl ifconfig.me
Show public IP as seen from internet. If different from router WAN IP → CG-NAT confirmed.
curl -4 ifconfig.me
Force IPv4 public IP check
curl -6 ifconfig.me
Check if IPv6 is available — if this returns an address, you have a real public IPv6 address and can accept incoming IPv6 connections
ip route show
Show routing table — default gateway should be 192.168.31.1 (backup router)
traceroute 8.8.8.8
Show network path — reveals how many hops exist before reaching the internet. Multiple RFC1918/CG-NAT hops = double/triple NAT.
ping -c 3 192.168.31.1
Ping backup router — confirms local network is working
ping -c 3 8.8.8.8
Ping internet — confirms outgoing internet works (CG-NAT always allows outgoing)
ℹ One-check CG-NAT detection: Run curl ifconfig.me then check your router admin WAN IP. If they differ — CG-NAT. If router WAN is in 100.64.0.0–100.127.255.255 — CG-NAT definitively confirmed.
01 — Questions and answers
Common CG-NAT questions
How do I confirm I have CG-NAT?
Run curl ifconfig.me on spserver, then check your router admin WAN IP. If the two IPs are different — you have a NAT layer you don't control. If the router WAN is in the 100.64–100.127 range — CG-NAT confirmed. Also try: ping your public IP from a mobile hotspot — if no response, it's CG-NAT.
Can I bypass CG-NAT for free?
Yes — Tailscale and Cloudflare Tunnel both bypass CG-NAT for free. You cannot remove CG-NAT (it's Airtel's infrastructure), but these tools make it irrelevant for every use case: SSH, public web services, file access. In many ways these solutions are better than a simple public IP — free SSL, DDoS protection, no exposed home IP.
Does CG-NAT affect download/upload speed?
No — CG-NAT only blocks unsolicited incoming connections. All outgoing traffic (browsing, streaming, downloading, apt update, cloudflared, Tailscale) works normally at full Airtel connection speed. spserver can reach the internet without restriction.
Port forwarding is configured but SSH still fails from outside
First confirm whether you have CG-NAT: compare curl ifconfig.me with your router WAN IP. If different — port forwarding cannot work regardless of configuration. Switch to Tailscale for SSH access. If IPs match (no CG-NAT), check UFW rules on spserver and whether sshd is actually listening on port 22.
Will upgrading to Airtel fibre remove CG-NAT?
Not automatically. Airtel fibre can still use CG-NAT. After upgrading, run the same two checks. If you want a guaranteed real IP, call Airtel support and request a static IP add-on — this specifically removes CG-NAT. Without explicitly requesting it, most Airtel plans use CG-NAT regardless of connection type.
