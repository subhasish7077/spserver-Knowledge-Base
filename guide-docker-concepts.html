Docker Concepts

The mental model behind Docker — images, containers, layers, registries, networks, volumes, and ports. Understanding these makes every docker command and docker-compose file obvious.

01 — Fundamental distinction
Image vs Container — the most important concept

Everything in Docker starts with understanding this distinction. Confusing images and containers is the most common source of Docker confusion.

Image — the blueprint
A read-only template. Defines what the container will contain — OS, runtime, libraries, code, environment variables. Built from a Dockerfile or pulled from a registry. Stored on disk. An image on its own does nothing — it just exists as a file.
Container — the running instance
A live, running process created from an image. Has its own process space, filesystem layer, and network interface. Multiple containers can run from the same image simultaneously — each fully isolated from the others.
Python analogy: Image = class definition (blueprint, stored on disk, not running) Container = object instance (running, has state, uses resources) Docker reality: python:3.11-slim (image — on disk, ~125MB) ↓ docker run container-1 (running Flask app on port 5000) container-2 (running a different Flask app — same image, different container) container-3 (running a bash session — same image again) All three from the same image, all isolated from each other
ℹ When you change code in your app and want the container to pick it up — you must rebuild the image and create a new container from it. The running container doesn't see changes to source files unless you're using a bind mount (covered in the Storage tab).
02 — How images are built
Image layers — why Docker builds are fast

Docker images are not monolithic blobs — they are built in layers. Each instruction in a Dockerfile creates one layer. Layers are read-only and stacked on top of each other. When a container runs, Docker adds a thin writable layer on top — changes go there, the image layers remain unchanged.

The power of layers is caching. Docker only rebuilds layers that have changed, plus all layers after them. If you change app.py but not requirements.txt, Docker skips rebuilding the pip install layer (the slow one) and only rebuilds the COPY layer. This is why copying requirements.txt before copying the rest of your code is a deliberate optimisation.

Change app.py only — rebuild time: ~2 seconds Layer 0: FROM python → ✓ cached Layer 1: WORKDIR → ✓ cached Layer 2: COPY requirements.txt → ✓ cached (file unchanged) Layer 3: RUN pip install → ✓ cached (dependencies unchanged) Layer 4: COPY . . → ✗ rebuild (app.py changed) Change requirements.txt — rebuild time: ~60 seconds Layer 0–1: ✓ cached Layer 2–4: ✗ all rebuilt (requirements.txt changed → pip install reruns)
03 — Where images come from
Registries and Docker Hub

A registry is a server that stores and distributes Docker images — like PyPI for Python packages, but for container images. Docker Hub is the default public registry. When you write image: nginx:latest in a docker-compose file, Docker pulls it from Docker Hub automatically.

🌐
Docker Hub
Default registry at hub.docker.com. Millions of public images. Official images from Nginx, MySQL, Python, Node etc. Free for public images.
📦
Official images
Maintained by the software's own team. Verified, regularly updated, security-patched. Examples: nginx, python, mysql, postgres.
👤
User images
Published by individuals or organisations. Format: username/image:tag. Verify before using — quality varies.
🏷️
Image tags
Version identifiers. nginx:latest = newest. python:3.11-slim = specific version + variant. Always pin versions in production.
bash — image naming examples
# Official image — no username prefix
docker pull nginx              # = nginx:latest from Docker Hub
docker pull python:3.11-slim   # specific version + variant
docker pull mysql:8.0          # pinned to MySQL 8.0.x

# User/org image
docker pull filebrowser/filebrowser

# Your own locally-built image (no pull needed)
docker build -t myflaskapp:latest .
# myflaskapp is only on this machine until you push it to a registry
04 — Container lifecycle
States a container goes through
Image on disk ↓ docker run / docker compose up Running ← process is active, using CPU/RAM ↓ docker stop (graceful) or docker kill (immediate) Stopped ← process ended, container still exists on disk ↓ docker start (restart) or docker rm (delete) Removed ← container gone, image still on disk ↓ docker rmi (remove image) Image gone ← nothing left (unless pushed to registry) Key insight: stopped containers still exist — they take disk space Run 'docker ps -a' to see them, 'docker rm NAME' to delete
ℹ Use docker run --rm to automatically delete the container when it exits — perfect for one-off commands. Use docker compose up -d for services — Compose manages restarts and keeps containers around between runs.
01 — The most important distinction
ports vs expose — critical difference

This is the most misunderstood Docker networking concept. ports and expose look similar but have fundamentally different effects. Getting this wrong either exposes services you shouldn't, or makes services unreachable.

ports:
"8082:80"
host_port:container_port — maps to HOST, publicly accessible
expose:
"5000"
internal only — only containers on same network can reach it
nothing:
—
container is completely isolated from everything
ports: "8082:80"
Maps host port 8082 to container port 80. Anyone who can reach the host can reach this port — your browser, Cloudflare Tunnel, other machines on the network. Use only for entry-point services like Nginx. The format is HOST:CONTAINER.
expose: "5000"
Documents that the container listens on port 5000, but does NOT map it to the host. Only other containers on the same Docker network can reach it by container name. Perfect for Flask, databases, backend services that should not be publicly reachable.
✓ Correct security pattern for our setup: Nginx container uses ports: "8082:80" (public). Flask container uses expose: "5000" (internal). Database containers use expose too. Only Nginx is publicly reachable — everything else is internal.
docker-compose.yml — correct pattern
services:
  nginx:
    ports:
      - "8082:80"    # PUBLIC — Cloudflare Tunnel hits this

  flask:
    expose:
      - "5000"      # INTERNAL — only nginx can reach this

  mysql:
    expose:
      - "3306"      # INTERNAL — only flask can reach this
02 — Container communication
Docker networks — how containers find each other

By default, containers are isolated from each other even on the same host. To allow containers to communicate, they must be on the same Docker network. Docker creates a virtual network where containers can reach each other using their service names as hostnames — Docker has built-in DNS that resolves container names to their internal IPs automatically.

docker-compose creates network: nginx-proxy_appnet nginx container internal IP: 172.18.0.2 hostname: nginx flask container internal IP: 172.18.0.3 hostname: flask mysql container internal IP: 172.18.0.4 hostname: mysql From inside nginx container: proxy_pass http://flask:5000 ← Docker DNS resolves "flask" → 172.18.0.3 From inside flask container: host="mysql", port=3306 ← Docker DNS resolves "mysql" → 172.18.0.4 These IPs are internal to Docker — not reachable from outside
ℹ The service name in docker-compose.yml is the hostname. If your service is named flask, containers on the same network reach it at http://flask:PORT. This is why Nginx proxy_pass uses the service name — Docker's internal DNS handles resolution automatically.
03 — Name resolution
Docker's built-in DNS

Docker runs an embedded DNS server at IP 127.0.0.11 inside every container. This server resolves container names to their internal IPs within a Docker network. It also forwards external DNS queries (like google.com) to the host's DNS resolver.

bash — test DNS from inside a container
# Open a shell inside the nginx container
docker compose exec nginx bash

# Inside the container — test DNS resolution
ping flask         # resolves to flask container's internal IP
ping google.com    # resolves via external DNS (forwarded by Docker)

# See which DNS server the container uses
cat /etc/resolv.conf
# nameserver 127.0.0.11  ← Docker's embedded DNS

# Test connectivity to flask by name
curl http://flask:5000
⚠ Docker DNS only works between containers on the same user-defined network. The default bridge network does NOT support container name resolution — only IP addresses work there. Always define a named network in docker-compose.yml and attach services to it.

01 — Persistent data
Docker volumes — the correct way to persist data

By default, any data written inside a container is lost when the container is removed. Docker volumes solve this — they are storage areas managed by Docker, stored outside the container's filesystem, and persist independently of the container's lifecycle.

Without volume: MySQL container writes /var/lib/mysql/ docker rm mysql_container → database data gone ✗ With named volume: MySQL container writes /var/lib/mysql/ ↕ Docker mounts mysql-data volume ← stored at /var/lib/docker/volumes/mysql-data/ docker rm mysql_container → volume survives ✓ docker run mysql again → database data restored ✓
docker-compose.yml — named volume example
services:
  mysql:
    image: mysql:8.0
    volumes:
      - mysql-data:/var/lib/mysql   # named volume
    environment:
      MYSQL_ROOT_PASSWORD: secret

volumes:
  mysql-data:   # declare the volume — Docker manages it
02 — Bind mounts
Bind mounts — map host directory into container

A bind mount maps a specific directory on the host filesystem directly into the container. Unlike volumes (which Docker manages), bind mounts point to an exact path you control. Changes on either side (host or container) are immediately visible to the other.

Bind mount: /home/sipu/projects/flaskapp (host — your actual files) ↕ mounted as /app (inside container) Edit app.py on host → instantly visible inside container at /app/app.py Flask restarts (with debug=True) → changes live immediately
docker-compose.yml — bind mount for dev
services:
  flask:
    build: .
    volumes:
      - .:/app   # . = current host dir, /app = container dir
    # Changes to any file in . appear instantly in /app inside container
    # Still need debug=True in Flask for auto-reload on code changes
docker-compose.yml — bind mount for config
services:
  nginx:
    image: nginx:latest
    volumes:
      - ./nginx/nginx.conf:/etc/nginx/nginx.conf   # single file mount
      - ./html:/usr/share/nginx/html              # directory mount
    # Edit nginx.conf on host → change is live (after nginx reload)
03 — Choosing the right storage
Volume vs bind mount vs no mount
Storage typeManaged byPath controlBest for
Named volume Docker Docker chooses path in /var/lib/docker/volumes/ Database data, persistent app state. Survives container removal. Portable across machines. Backup-friendly.
Bind mount You You specify exact host path Development code, config files. Edit on host, see changes in container. Not suitable for databases — ownership/permission issues.
No mount N/A Container filesystem only Stateless services, one-off containers. Data lost on container removal — fine for stateless apps.
tmpfs mount Host RAM In-memory only Sensitive temporary data. Stored in host RAM, never written to disk. Lost on container stop. Useful for secrets or temporary caches.
⚠ Common mistake: Using a bind mount for a database (MySQL, PostgreSQL). The container's database process writes files with specific ownership (mysql:mysql user inside container). On the host, these files may have wrong permissions or UIDs. Always use named volumes for databases.
01 — Images
Image management commands
docker images
List all images on disk — shows repository, tag, image ID, size
docker pull nginx:latest
Download an image from Docker Hub without running it
docker build -t myapp:latest .
Build image from Dockerfile in current directory, tag as myapp:latest
docker build --no-cache -t myapp .
Build without layer cache — force full rebuild of all layers
docker rmi nginx:latest
Remove an image from disk — must stop all containers using it first
docker image prune
Remove dangling images (untagged, not used by any container)
docker history myapp:latest
Show all layers in an image with their sizes and commands
docker inspect myapp:latest
Full JSON details about an image — config, layers, environment variables
docker tag myapp:latest myapp:v1.0
Add another tag to an existing image — useful for versioning
docker save myapp:latest | gzip > myapp.tar.gz
Export image to a file — useful for transferring to another machine
02 — Containers
Container management commands
docker run -d -p 8080:80 nginx
Run nginx container in background (-d), map port 8080 to container port 80
docker run --rm -it python:3.11 bash
Run interactively (-it), remove when done (--rm) — great for one-off experiments
docker ps
List running containers
docker ps -a
List all containers including stopped ones
docker stop CONTAINER
Gracefully stop a container — sends SIGTERM, waits 10s, then SIGKILL
docker start CONTAINER
Start a stopped container
docker rm CONTAINER
Remove a stopped container — container must be stopped first
docker exec -it CONTAINER bash
Open an interactive shell inside a running container — essential for debugging
docker exec CONTAINER cat /app/app.py
Run a single command inside a running container without opening full shell
docker logs CONTAINER
Show container stdout/stderr output
docker logs -f CONTAINER
Follow container logs live — Ctrl+C to stop
docker inspect CONTAINER
Full JSON details — network settings, mounts, environment, restart count
docker stats
Live stream of CPU, memory, network, and disk I/O for all running containers
docker cp CONTAINER:/path/file .
Copy a file from inside a container to the host
03 — Networks and volumes
Network and volume commands
docker network ls
List all Docker networks — shows name, driver, scope
docker network inspect NETWORK
Show full network details including which containers are connected and their IPs
docker network create mynet
Create a user-defined bridge network manually
docker volume ls
List all Docker volumes — shows name and driver
docker volume inspect VOLUME
Show volume details including the host path where data is stored
docker volume prune
Remove all volumes not used by any container — careful, deletes data
01 — What's next
Future scope
⟶ Things to explore from here
Multi-stage builds — use multiple FROM statements in a Dockerfile to build in one stage and copy only the final artifact to a minimal runtime image. Dramatically reduces image size — a Go binary built in a 1GB build image can run in a 10MB scratch image.
Docker layer caching in CI/CD — in GitHub Actions, save and restore the Docker layer cache between runs using cache-from and cache-to. Turns 5-minute builds into 30-second builds when only code changes.
Docker networking deep dive — understand how Docker creates virtual ethernet pairs (veth), how iptables rules forward port traffic, how the embedded DNS works at the packet level. Highly relevant for debugging complex networking issues.
Docker secrets — store sensitive data (passwords, API keys) as Docker secrets instead of environment variables. Secrets are encrypted in the Swarm, only accessible to containers that need them, not visible in docker inspect.
Container health checks — add a HEALTHCHECK instruction to Dockerfiles. Docker periodically runs the check and marks containers as healthy or unhealthy. docker-compose restarts unhealthy containers automatically. Essential for production reliability.
Kubernetes mental model mapping — once Docker concepts are solid: Image → same in K8s. Container → same. Pod → group of containers. Deployment → manages Pod replicas. Service → networking like docker-compose networks. The concepts translate directly.
