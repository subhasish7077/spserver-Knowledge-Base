Interface Rename with udev

Renaming the USB ethernet adapter from its auto-generated MAC-based name to a clean, human-readable ethusb0 — so every config file, firewall rule, and Netplan entry uses a consistent, predictable name.

01 — The problem
Why the default interface name is a problem

spserver uses a USB ethernet adapter instead of the built-in NIC. On modern Linux, USB ethernet adapters get auto-generated names based on their MAC address — producing names like enx00e04c680bc2. This is technically unique and predictable, but completely impractical to work with.

auto-generated name
enx00e04c680bc2
18 characters. Encodes the MAC address. Hard to type. Easy to mistype. Changes if you swap adapters. Clutters every config file.
after udev rename
ethusb0
7 characters. Meaningful — "ethernet USB adapter 0". Easy to type. Never changes. Consistent across all config files.

Every place the interface name appears — Netplan config, UFW rules, ip commands, network diagnostics — would require typing that 18-character string. A single typo in the Netplan config breaks networking and locks you out of SSH. Renaming to ethusb0 makes everything cleaner and less error-prone.

⚠ The interface rename must be done before writing the Netplan static IP config. The name in the Netplan YAML must exactly match the actual interface name. If you write Netplan with the long MAC-based name and then rename — the Netplan config breaks.
02 — Concept
What is udev

udev is the Linux device manager — a system service that handles what happens when hardware is connected or disconnected. When you plug in a USB drive, keyboard, ethernet adapter, or any peripheral, udev detects the event and applies rules to configure the device: what name to assign it, what permissions to set, what scripts to run.

udev rules live in /etc/udev/rules.d/ — a directory of rule files that are processed in alphabetical-numerical order. Each rule is one line describing: match conditions (what device to match) and actions (what to do when matched).

Hardware event
USB ethernet adapter connects at boot (or is plugged in)
Kernel detects device
Kernel creates a device entry and sends a uevent to udev
udev processes rules
udev reads /etc/udev/rules.d/ files in order, checks each rule against the device attributes
Rule matches
Our rule matches: SUBSYSTEM=="net", ACTION=="add", ATTR{address}=="00:e0:4c:68:0b:c2"
Action applied
udev assigns NAME="ethusb0" — the interface appears as ethusb0 instead of enx00e04c680bc2
03 — Context
Why Linux uses these long names by default

The long MAC-based names are part of Linux's Predictable Network Interface Names scheme introduced in systemd. Before this, interfaces were named eth0, eth1 etc. — but those names were assigned in the order hardware was detected at boot, which could change between reboots. A server with two NICs might see them as eth0/eth1 on one boot and eth1/eth0 on the next. For servers with bonded NICs or specific routing rules, this was a serious problem.

The solution: name interfaces based on their physical characteristics so the same hardware always gets the same name. For PCI NICs this produces names like eno1 (onboard) or enp2s0 (PCI slot 2). For USB NICs, the MAC address is the most stable identifier — hence enx + MAC address.

Predictable naming — the problem it solves
Same physical NIC always gets the same name. Bonding, routing rules, and firewall rules survive reboots without re-checking which eth number is which.
udev rename — best of both worlds
We keep predictability (rule matches by MAC, always same adapter) but get a clean, human-readable name. ethusb0 is both stable and easy to type.
04 — Important distinction
udev rules vs systemd .link files — what works on spserver

There are two common methods for renaming network interfaces on modern Ubuntu — and they are not interchangeable. The correct one depends on your specific setup. We discovered this through trial and error.

MethodFile locationWorks on spserver?Notes
udev rule what we use /etc/udev/rules.d/10-network.rules Yes ✓ Reliable on Ubuntu 24.04 with USB ethernet on spserver. Confirmed working after reboot.
systemd .link file /etc/systemd/network/10-rename.link Not confirmed The alternative method. Works on some setups where udev alone doesn't apply before systemd-networkd initialises the interface. Not needed on spserver.
✓ On spserver: udev rule at /etc/udev/rules.d/10-network.rules — rebooted after creating the rule — rename applies correctly and permanently. This is the authoritative method for our setup.
01 — Find the adapter
Step 1 — Find the MAC address of the USB ethernet adapter
1
Identify the USB ethernet adapter and its MAC address
▼

The udev rule matches the adapter by its MAC address — a hardware-burned identifier that never changes for a specific adapter. We need the exact MAC address to write an accurate rule. The MAC is visible in the output of ip link show.

bash — run on spserver before the rename
# List all network interfaces
ip link show

# Example output:
# 1: lo: <LOOPBACK> ...
# 2: enp2s0: <NO-CARRIER> ... (built-in NIC — not connected)
# 3: enx00e04c680bc2: <BROADCAST,MULTICAST,UP,LOWER_UP>
#       link/ether 00:e0:4c:68:0b:c2 brd ff:ff:ff:ff:ff:ff
#                  ↑ this is the MAC address

# The USB adapter is identifiable by:
# - Name starts with "enx" (USB ethernet uses "enx" + MAC)
# - State shows UP (it's connected and active)
# - Has an assigned IP if Netplan is already working

# Alternatively — list USB devices to find the adapter
lsusb
# Look for an ethernet/network adapter in the list
ℹ Our adapter MAC address: 00:e0:4c:68:0b:c2 — this is what the udev rule matches against. Yours will be different. Use the MAC from your own ip link show output.
02 — Write the rule
Step 2 — Create the udev rule file
2
Write the rule — replace MAC with your own
▼

Rule files in /etc/udev/rules.d/ are numbered — lower numbers are processed first. We use 10-network.rules (prefix 10) to ensure it runs early, before other network-related rules. Each line in the file is one rule — we only need one.

bash
# Create the udev rules file
sudo nano /etc/udev/rules.d/10-network.rules

Paste this single line — replacing the MAC address with your adapter's MAC from step 1:

/etc/udev/rules.d/10-network.rules
SUBSYSTEM=="net", ACTION=="add", ATTR{address}=="00:e0:4c:68:0b:c2", NAME="ethusb0"
⚠ This is a single line — no line breaks. The MAC address must be lowercase with colons exactly as shown by ip link show. The NAME value is what your interface will be called after rename.
03 — Apply
Step 3 — Reload rules and reboot
3
Reload udev and reboot to apply the rename
▼

Running udevadm control --reload-rules tells udev to re-read all rule files. However, for a network interface rename, a full reboot is the most reliable way to ensure the new name is applied cleanly — the rename happens when the device is first detected during boot, before networking starts.

bash
# Tell udev to reload all rule files
sudo udevadm control --reload-rules

# Trigger udev events (may not fully apply network rename without reboot)
sudo udevadm trigger

# Reboot — the rename will be applied on next boot
sudo reboot

# Note: SSH will disconnect during reboot — reconnect after ~1 minute
# ssh sipu@192.168.31.150
ℹ After reboot, SSH back in using the static IP (192.168.31.150) — same as before. The rename only changes the interface name, not the IP address. If the static IP was already configured in Netplan using the old name, update Netplan to use ethusb0 first (see Step 5).
04 — Verify
Step 4 — Confirm the rename worked
bash — run after reboot
# Check interface list — should now show ethusb0, not enx00e04c...
ip link show

# Specifically check ethusb0 exists and is UP
ip link show ethusb0
# Should show: ethusb0: <BROADCAST,MULTICAST,UP,LOWER_UP>

# Confirm IP address is still assigned
ip addr show ethusb0
# Should show: inet 192.168.31.150/24

# Confirm networking still works
ping -c 3 8.8.8.8

# Check udev saw the rule was applied
udevadm info /sys/class/net/ethusb0
# Shows all udev properties for the interface
✓ If ip link show ethusb0 shows the interface as UP and ip addr show ethusb0 shows 192.168.31.150 — the rename is working correctly and permanently. The MAC address is hardware-burned, so this rule matches every time the adapter is connected.
05 — Update Netplan
Step 5 — Make sure Netplan uses the new name
5
Update Netplan config to reference ethusb0
▼

If Netplan was already written using the old MAC-based name, it now refers to an interface that no longer exists. Update the YAML to use ethusb0. If Netplan hasn't been written yet — write it with ethusb0 from the start (see the Netplan guide).

/etc/netplan/01-netcfg.yaml — use ethusb0 not the long name
network:
  version: 2
  renderer: networkd
  ethernets:
    ethusb0:          # ← use the new name here, not enx00e04c680bc2
      dhcp4: false
      addresses:
        - 192.168.31.150/24
      routes:
        - to: default
          via: 192.168.31.1
      nameservers:
        addresses: [8.8.8.8, 8.8.4.4]
bash — apply the updated Netplan
sudo netplan try   # test — auto-reverts in 120s if broken
sudo netplan apply # apply permanently after confirming it works
01 — The rule file
Complete rule file — every part explained
/etc/udev/rules.d/10-network.rules
SUBSYSTEM=="net", ACTION=="add", ATTR{address}=="00:e0:4c:68:0b:c2", NAME="ethusb0"
PartWhat it does
SUBSYSTEM=="net"Match only network devices. udev handles all hardware — USB drives, input devices, sensors. This filter limits the rule to network interfaces only so it doesn't accidentally match other hardware.
ACTION=="add"Fire only when a device is added (connected or detected at boot). Not on remove or change events. Renaming only makes sense when a device first appears.
ATTR{address}=="00:e0:4c:68:0b:c2"Match by MAC address — the hardware-burned identifier unique to this specific adapter. MAC addresses never change for a given piece of hardware. This is why the rename is permanent — the same adapter always matches.
NAME="ethusb0"The action: assign this name to the matching interface. udev applies this before the interface is registered with the kernel, so the interface appears under this name from the very beginning.
ℹ udev rules use == for matching conditions and = for setting values. The comma between conditions means AND — all conditions must match for the rule to fire. Order within a rule line does not matter — all matches are checked before any actions are applied.
02 — Other match options
Other ways to match a network interface in udev

We match by MAC address because it is the most stable identifier for our USB adapter. Other attributes can be used depending on the use case:

AttributeMatches onWhen to use
ATTR{address}=="MAC" Hardware MAC address Our method. Best for USB adapters — MAC is hardware-burned, never changes for this specific adapter.
DRIVERS=="r8152" Driver name Useful if you want to rename all adapters using the same driver. Less specific than MAC — all adapters of the same chipset would match.
ATTR{idVendor}=="0bda" USB vendor ID Match by USB manufacturer ID (Realtek = 0bda). Combine with idProduct for more specificity.
KERNELS=="1-1.2" USB bus/port path Match by physical USB port. If the adapter is always in the same port — consistent name tied to port not device. Breaks if you move the adapter to a different port.
bash — find all udev attributes for your adapter
# List all udev attributes for the interface — use these in rules
udevadm info --attribute-walk /sys/class/net/ethusb0

# Or by device path (before rename, use the long name)
udevadm info --attribute-walk /sys/class/net/enx00e04c680bc2

01 — What went wrong
Blockers hit during udev setup
⚠ Blocker 1
udevadm trigger didn't apply the rename — interface still had the old name
After writing the rule file and running sudo udevadm control --reload-rules && sudo udevadm trigger, the interface still showed as enx00e04c680bc2. The rule appeared correct but wasn't being applied.
✓ Fix
Reboot — udev rename for active interfaces requires reboot
udev can rename devices as they are added, but cannot rename an already-active network interface — the kernel holds a reference to the existing name. The rename only takes effect when the device is first detected, which happens at boot. After sudo reboot, the interface appeared as ethusb0 immediately. The rule itself was correct from the start — the reload-trigger approach simply doesn't work for already-active interfaces.
⚠ Blocker 2
Lost network after reboot — Netplan config still referenced old name
After rebooting with the udev rule in place, the interface was renamed to ethusb0 correctly — but SSH no longer connected. The server had no IP address. The Netplan config still referenced enx00e04c680bc2 — an interface that no longer existed under that name.
✓ Fix
Update Netplan to reference ethusb0 BEFORE rebooting
Connected physically (keyboard + monitor) to access the server. Edited /etc/netplan/01-netcfg.yaml to replace enx00e04c680bc2 with ethusb0. Ran sudo netplan apply. IP address came back immediately. Lesson: always update Netplan to use the new interface name before rebooting after a rename. The correct order is: (1) write udev rule, (2) update Netplan to use new name, (3) reload rules, (4) reboot.
01 — Issues and fixes
Common udev rename problems
No network after reboot — SSH not connecting
The most common issue — Netplan still references the old interface name. The interface was renamed but Netplan doesn't know its new name, so no IP is assigned. Recovery options: (1) Physical access — connect keyboard and monitor, edit /etc/netplan/01-netcfg.yaml, change enx00e04c680bc2 to ethusb0, run sudo netplan apply. (2) Tailscale — if it was set up, SSH in via Tailscale IP. Prevention: always update Netplan before rebooting.
Interface shows old name even after reboot
The udev rule didn't match. Check: (1) MAC address in the rule — run udevadm info /sys/class/net/OLDNAME | grep ADDRESS and compare with what's in the rule file. MAC must be lowercase with colons. (2) Rule file syntax — no quotes missing, no extra spaces. (3) Rule file name — must end in .rules. (4) Rule file location — must be in /etc/udev/rules.d/ not /etc/udev/.
Rule matches but NAME is ignored — interface keeps old name
On some Ubuntu 22.04+ setups, systemd-networkd takes over interface naming before udev can apply. In this case, a systemd .link file is needed alongside or instead of the udev rule. Create: /etc/systemd/network/10-rename.link with [Match] MACAddress=00:e0:4c:68:0b:c2 and [Link] Name=ethusb0. Run sudo systemctl restart systemd-networkd and reboot. This was not needed on spserver but may be required on different setups.
How to find the old interface name if it no longer exists
After a successful rename, the old name is gone. If you need to find the MAC address again: udevadm info /sys/class/net/ethusb0 | grep ID_NET_NAME_MAC — shows the original MAC-based name which contains the MAC address. Or simply: ip link show ethusb0 — the link/ether line shows the MAC address directly.
Verifying the rule is active and matched
Run udevadm info /sys/class/net/ethusb0 and look for the property ID_NET_NAME_FROM_USER_SPACE=ethusb0 — this confirms udev applied the rename via a user-space rule. Also check: udevadm test /sys/class/net/ethusb0 2>&1 | grep NAME — shows what NAME udev would assign when processing rules for this device.
01 — What's next
Future scope
⟶ Things to explore from here
Rename other devices with udev — the same technique works for USB storage drives (/dev/sdb → /dev/usb-backup), serial ports, and other hardware. Write rules matching by SUBSYSTEM=="block" or SUBSYSTEM=="tty" with SYMLINK+= to create stable device names.
udev automation rules — trigger scripts when devices connect. For example: when a USB drive with a specific serial is plugged in, automatically mount it and start a backup. udev supports RUN+="/path/to/script" to execute arbitrary commands on device events.
Multiple ethernet adapters — if adding a second USB ethernet adapter, add a second rule with a different MAC and different name (ethusb1). Both adapters get stable names. Could configure bonding or load balancing across them with Netplan.
systemd .link files — the alternative to udev rules for interface renaming. Managed by systemd-networkd directly. Worth understanding both methods and knowing when each is needed. Documented at man systemd.link.
Network bonding — after mastering interface naming, bond two ethernet adapters together for redundancy or increased bandwidth using Netplan's bonding configuration. If the USB adapter fails, traffic automatically fails over to the built-in NIC.
