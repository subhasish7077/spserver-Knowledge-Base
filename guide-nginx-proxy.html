Nginx Reverse Proxy

Running Nginx on the host as the single entry point for all spserver services — domain-based routing to Docker backends, proxy headers, and HTTPS via Certbot and Let's Encrypt.

01 — What & Why
What a reverse proxy is — and why Nginx

A reverse proxy sits in front of one or more backend services and forwards incoming requests to them. The client talks to Nginx on port 80 or 443 — it never sees the backend directly. Nginx decides which backend to forward each request to, based on the hostname, path, or other rules.

Without a reverse proxy
Every service exposes its own port on the host. Jenkins on :8090, Flask on :5000, Gitea on :3000. Users must remember different ports. No SSL — everything is plain HTTP. No single place to add auth, rate limiting, or logging. UFW needs a hole per service.
With Nginx as reverse proxy
All traffic enters on port 80 and 443. Nginx routes by subdomain: jenkins.dustyserver.co.in → Jenkins, dustyserver.co.in → Flask. SSL is terminated at Nginx — one cert, all services. Backend services bind to localhost only and are never directly exposed.

Nginx is installed directly on the Ubuntu host — not inside Docker. This gives it access to all host ports and makes it the single point of entry for every service on spserver, whether Dockerised or running natively.

ℹ Why host Nginx, not containerised Nginx? Running Nginx on the host gives it access to all services — Docker containers, systemd services, anything listening on localhost — without extra network config. A containerised Nginx would need additional configuration to reach services outside its Compose network. For a multi-purpose server like spserver, host Nginx is simpler and more flexible.

03 — Proxy headers
Why proxy headers matter

When Nginx forwards a request to a backend, the backend sees the connection as coming from 127.0.0.1 — Nginx itself — not the real client. This breaks IP-based logging, rate limiting, and security checks. Proxy headers carry the original request details through to the backend.

HeaderWhat it carriesWhy backends need it
X-Real-IP The client's original IP address Lets the backend log or rate-limit by real client IP. Without it, every request looks like it came from 127.0.0.1.
X-Forwarded-For Chain of IPs through all proxies Standard header read by Flask, Django, and most logging tools. Format: client, proxy1, proxy2.
X-Forwarded-Proto The original protocol — http or https Lets the backend know if the original request was HTTPS even though Nginx forwarded it as HTTP internally. Needed for redirect logic.
Host The original Host header from the client Without this, the backend sees the proxy address as the host. Needed for virtual hosting and URL generation inside the app.
ℹ Include all four headers in every proxy_pass block as a matter of habit. Jenkins in particular reads X-Forwarded-For and X-Forwarded-Proto to validate its "Jenkins URL" config — missing headers cause an admin warning that can't be dismissed until they are set correctly.
04 — Routing strategy
Subdomain-based routing — why not port-based
ApproachExampleTradeoffs
Port-based avoid dustyserver.co.in:8090 Users must remember port numbers. Non-standard ports are blocked by many corporate firewalls. No easy SSL. Cannot put two services on port 443.
Subdomain-based chosen jenkins.dustyserver.co.in Clean, memorable URLs. All traffic on port 443. One SSL cert covers multiple subdomains. Each service gets its own Nginx server block — easy to add, modify, or remove independently.

Each subdomain requires a DNS A record at the registrar pointing to spserver's public IP. All subdomains of dustyserver.co.in resolve to the same IP — Nginx uses the server_name directive to route each request to the correct backend.

01 — Install
Step 1 — Install Nginx on the host
1
Install and verify Nginx
▼

Nginx from Ubuntu's official repo is current enough for everything we need and receives security updates via apt upgrade. We enable it as a systemd service so it starts automatically on every server reboot.

bash
sudo apt update
sudo apt install -y nginx

sudo systemctl enable nginx
sudo systemctl start nginx

# Confirm it is running
sudo systemctl status nginx
# Should show: Active: active (running)

# Confirm Nginx responds on port 80
curl -s http://localhost | head -5
# Returns the default Nginx welcome page HTML

nginx -v
# nginx version: nginx/1.24.x
2
Allow ports 80 and 443 through UFW
▼

Ports 80 and 443 must be open for Nginx to receive external traffic — and for Certbot's HTTP-01 challenge to work when obtaining SSL certificates. 'Nginx Full' is a pre-defined UFW app profile that opens both at once.

bash
sudo ufw allow 'Nginx Full'
# Equivalent to: sudo ufw allow 80 && sudo ufw allow 443

sudo ufw status
# Should show Nginx Full as ALLOW
02 — Server block
Step 2 — Write server blocks for each service

Each service gets its own config file in /etc/nginx/sites-available/. The file is not active until it is symlinked into /etc/nginx/sites-enabled/. The pattern — server_name, proxy_pass, proxy headers — is identical for every service. Only the domain and upstream address change.

3
Jenkins server block — jenkins.dustyserver.co.in → 127.0.0.1:8090
▼
/etc/nginx/sites-available/jenkins
server {
    # Listen on port 80 for HTTP
    # Certbot will add a :443 block and a redirect here after it runs
    listen 80;
    listen [::]:80;

    # Nginx matches the incoming Host header against this value
    server_name jenkins.dustyserver.co.in;

    location / {
        # Forward all requests to Jenkins on localhost:8090
        # Jenkins Docker container has ports: "8090:8080" in compose
        proxy_pass http://127.0.0.1:8090;

        # Real client IP — without this Jenkins sees every request from 127.0.0.1
        proxy_set_header X-Real-IP        $remote_addr;

        # Standard proxy chain header — frameworks and logging tools read this
        proxy_set_header X-Forwarded-For  $proxy_add_x_forwarded_for;

        # Original protocol — Jenkins needs this to generate correct redirect URLs
        proxy_set_header X-Forwarded-Proto $scheme;

        # Original Host header — Jenkins uses this to build its own absolute URLs
        proxy_set_header Host             $host;

        # Increase timeouts — Jenkins builds and SCM polling can take time
        proxy_read_timeout    90s;
        proxy_connect_timeout 90s;

        # Websocket support — required for Jenkins live log streaming and terminal
        proxy_http_version 1.1;
        proxy_set_header Upgrade    $http_upgrade;
        proxy_set_header Connection "upgrade";
    }
}
4
Flask app server block — dustyserver.co.in → 127.0.0.1:5000
▼
/etc/nginx/sites-available/flask-app
server {
    listen 80;
    listen [::]:80;

    server_name dustyserver.co.in;

    location / {
        proxy_pass http://127.0.0.1:5000;

        proxy_set_header X-Real-IP         $remote_addr;
        proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header Host              $host;
    }
}
ℹ Websocket headers are omitted here — Flask doesn't use websockets by default. If the Flask app is extended with Flask-SocketIO, add the same Upgrade and Connection headers as the Jenkins block.
03 — Enable
Step 3 — Enable configs with symlinks

Nginx reads configs from /etc/nginx/sites-enabled/. Files in sites-available/ are not active until symlinked there. This pattern lets you keep configs ready but disabled — useful for quickly toggling a service offline.

bash
# Symlink format: ln -s <absolute-target> <link-path>
sudo ln -s /etc/nginx/sites-available/jenkins   /etc/nginx/sites-enabled/jenkins
sudo ln -s /etc/nginx/sites-available/flask-app /etc/nginx/sites-enabled/flask-app

# Confirm symlinks are correct
ls -la /etc/nginx/sites-enabled/
# jenkins   -> /etc/nginx/sites-available/jenkins
# flask-app -> /etc/nginx/sites-available/flask-app
# default   -> /etc/nginx/sites-available/default  (pre-existing)

# To disable a config without deleting it — remove only the symlink
# sudo rm /etc/nginx/sites-enabled/flask-app
# The file in sites-available/ stays intact and can be re-enabled anytime
⚠ Use absolute paths in the symlink — /etc/nginx/sites-available/jenkins, not ../sites-available/jenkins. A relative symlink is resolved from the directory where ln -s was run, not from the symlink's location. Get it wrong and the symlink silently points nowhere — no error, but the config is never loaded.
04 — Test & reload
Step 4 — Validate config and reload Nginx

Always test before reloading. A syntax error in a new config file will make Nginx refuse to reload — but if Nginx is restarted (not reloaded) with a bad config, it won't start at all, taking every service behind it offline. nginx -t validates the full configuration without touching the running process.

5
Test, reload, and verify routing
▼
bash
# Validate ALL Nginx config files — including new ones
sudo nginx -t
# nginx: the configuration file /etc/nginx/nginx.conf syntax is ok
# nginx: configuration file /etc/nginx/nginx.conf test is successful
#
# If you see errors — fix them before continuing. Do NOT reload with a failing -t.

# Reload: apply config changes without dropping existing connections
sudo nginx -s reload
# Or equivalently:
sudo systemctl reload nginx

# Test routing before DNS is configured — override the Host header with curl
curl -H "Host: jenkins.dustyserver.co.in" http://localhost/
# Should return Jenkins HTML (typically a redirect to /login)

curl -H "Host: dustyserver.co.in" http://localhost/
# Should return Flask JSON response
✓ If nginx -t reports "test is successful" and the curl Host-header tests return the expected backend responses — Nginx is routing correctly. DNS records and HTTPS are the next steps.
05 — HTTPS
Step 5 — Add HTTPS with Certbot and Let's Encrypt

Certbot is the official Let's Encrypt client. It obtains a free SSL/TLS certificate, installs it into Nginx, and edits the server block automatically to redirect HTTP → HTTPS. Certificates are valid for 90 days — Certbot installs a systemd timer to auto-renew them before expiry.

The HTTP-01 challenge requires that Let's Encrypt can reach port 80 on spserver from the public internet to verify domain ownership. UFW must allow port 80, and the domain's DNS must be resolvable externally via Tailscale.

6
Install Certbot and obtain certificates
▼
bash
# Install Certbot and its Nginx plugin
sudo apt install -y certbot python3-certbot-nginx

# Obtain and install a certificate for Jenkins
# Certbot reads server_name from the Nginx config and edits it automatically
sudo certbot --nginx -d jenkins.dustyserver.co.in

# Obtain a certificate for the main domain
sudo certbot --nginx -d dustyserver.co.in

# Or obtain one certificate covering both (Subject Alternative Names)
sudo certbot --nginx -d dustyserver.co.in -d jenkins.dustyserver.co.in

# What Certbot does automatically:
#  1. Verifies domain ownership via HTTP-01 challenge on port 80
#  2. Downloads the certificate from Let's Encrypt
#  3. Edits the Nginx server block — adds ssl directives and 80 → 443 redirect
#  4. Reloads Nginx
#  5. Installs a systemd timer for auto-renewal

# Confirm the auto-renewal timer is active
sudo systemctl status certbot.timer

# Test renewal without actually renewing
sudo certbot renew --dry-run
# Should show: Congratulations, all simulated renewals succeeded
What the config looks like after Certbot
/etc/nginx/sites-available/jenkins — after Certbot edits
server {
    # Certbot adds this: redirect all HTTP traffic to HTTPS
    listen 80;
    listen [::]:80;
    server_name jenkins.dustyserver.co.in;
    return 301 https://$host$request_uri;
}

server {
    listen 443 ssl;
    listen [::]:443 ssl;
    server_name jenkins.dustyserver.co.in;

    # Certificate paths — managed by Certbot, do not edit manually
    ssl_certificate     /etc/letsencrypt/live/jenkins.dustyserver.co.in/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/jenkins.dustyserver.co.in/privkey.pem;
    include             /etc/letsencrypt/options-ssl-nginx.conf;
    ssl_dhparam         /etc/letsencrypt/ssl-dhparams.pem;

    location / {
        proxy_pass              http://127.0.0.1:8090;
        proxy_set_header X-Real-IP         $remote_addr;
        proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header Host              $host;
        proxy_read_timeout    90s;
        proxy_connect_timeout 90s;
        proxy_http_version 1.1;
        proxy_set_header Upgrade    $http_upgrade;
        proxy_set_header Connection "upgrade";
    }
}
ℹ After Certbot runs, always verify with sudo nginx -t — Certbot edits the config in place and occasionally introduces whitespace issues on some Ubuntu versions. If the test fails, check sudo cat /etc/nginx/sites-available/jenkins to inspect what Certbot wrote.

01 — What went wrong
Blockers hit while setting up Nginx reverse proxy
⚠ Blocker 1
nginx -s reload failed — syntax error in new config
After writing the Jenkins server block, ran sudo nginx -s reload directly without testing first. Nginx logged: "nginx: [emerg] unexpected end of file, expecting ';' or '}'". The new block wasn't active — Nginx kept serving the old config.
✓ Fix
Always run nginx -t before reload — it reports the exact line and file
Ran sudo nginx -t which reported the exact file and line number — a missing semicolon at the end of the proxy_pass line. Fixed it, ran nginx -t again, confirmed "test is successful", then reloaded. The block became active immediately. Lesson learned: sudo nginx -t && sudo nginx -s reload as a single habit — the && ensures reload only runs if the test passes.
⚠ Blocker 2
Certbot failed — HTTP-01 challenge connection timeout
Running sudo certbot --nginx -d jenkins.dustyserver.co.in failed with a timeout during the HTTP-01 challenge. Let's Encrypt's servers couldn't reach port 80 on spserver to verify domain ownership.
✓ Fix
Two issues: UFW blocking port 80, and DNS pointing to unreachable address
(1) UFW was blocking port 80 — fixed with sudo ufw allow 'Nginx Full'. (2) The domain's DNS A record was pointing to the Tailscale IP rather than the externally reachable public IP. Fixed the DNS record at the registrar to point to the correct address. After both fixes, Certbot succeeded. Key check before running Certbot: curl -4 ifconfig.io to see your public IP, then confirm the DNS A record matches it with nslookup jenkins.dustyserver.co.in.
⚠ Blocker 3
Symlink in sites-enabled pointed nowhere — config silently ignored
Created the symlink with a relative path: sudo ln -s ../sites-available/jenkins /etc/nginx/sites-enabled/jenkins. The symlink appeared in ls output, but nginx -t didn't include the Jenkins config and the proxy wasn't working — no errors, just no routing.
✓ Fix
Symlinks must use absolute paths — relative paths resolve from the wrong directory
A relative symlink path is resolved from the directory where ln -s was run — not from the symlink's own location. Since the command was run from the home directory, ../sites-available/jenkins resolved to a path that doesn't exist. Verified with readlink -f /etc/nginx/sites-enabled/jenkins — showed a non-existent path. Deleted the broken symlink and recreated: sudo ln -s /etc/nginx/sites-available/jenkins /etc/nginx/sites-enabled/jenkins. Config loaded on next nginx -t.
⚠ Blocker 4
Jenkins showed "reverse proxy set up is broken" warning
Jenkins loaded through Nginx but displayed a persistent admin warning: "It appears that your reverse proxy set up is broken". Jenkins was generating URLs as http://127.0.0.1:8090/... instead of https://jenkins.dustyserver.co.in/....
✓ Fix
Set Jenkins URL in admin settings + confirm proxy headers are present
Two-part fix: (1) Jenkins → Manage Jenkins → System → Jenkins URL — set it to https://jenkins.dustyserver.co.in. Jenkins uses this value to generate webhook and redirect URLs. (2) Confirmed proxy_set_header Host $host and X-Forwarded-Proto $scheme were set in the Nginx location block — Jenkins reads these to confirm it is behind a proxy and to know the externally facing scheme. After both changes and an Nginx reload, the warning disappeared and all URLs were correct.
01 — Issues and fixes
Common Nginx reverse proxy problems and fixes
502 Bad Gateway
Nginx reached the port in proxy_pass but the backend refused the connection or isn't running. Check: (1) Is the backend running? docker compose ps — look for the service as "Up". (2) Is it listening on the right port? ss -tlnp | grep 8090. (3) Is the port bound to 127.0.0.1 (accessible to Nginx on the host) not just the Docker network? Check docker compose ps — the Ports column shows the bind address. (4) sudo tail /var/log/nginx/error.log — the error message explains exactly what failed upstream.
504 Gateway Timeout
Nginx connected to the backend but it didn't respond in time. Common with Jenkins during heavy builds. Fix: increase timeouts in the server block — proxy_read_timeout 300s; proxy_connect_timeout 300s;. Check if the backend is overloaded: docker stats. For Jenkins: also ensure the Jenkins URL is set correctly in admin settings — a misconfigured URL can cause redirect loops that exhaust the timeout.
nginx: [emerg] error on reload or restart
Config syntax error — Nginx won't reload with a broken config. Run sudo nginx -t to see the exact file, line, and error. Common mistakes: missing semicolons, unmatched braces, wrong directive names (e.g. proxy-pass vs proxy_pass), duplicate server_name values across blocks. Fix the error and confirm nginx -t reports "test is successful" before reloading.
Server block not loading — nginx -t doesn't include it
Config file is not symlinked into sites-enabled/ or the symlink is broken. Check: ls -la /etc/nginx/sites-enabled/ — look for the expected entry. If it exists, verify the target: readlink -f /etc/nginx/sites-enabled/jenkins — should print the absolute path to the file in sites-available/. If the path doesn't exist, the symlink is broken — delete and recreate with an absolute path.
Certbot fails — challenge timed out or connection refused
Let's Encrypt cannot reach port 80 on spserver. Check: (1) UFW: sudo ufw status — port 80 must be ALLOW. (2) DNS: nslookup jenkins.dustyserver.co.in — must resolve to spserver's public IP. (3) Is Nginx running and responding on port 80 before Certbot runs? Certbot temporarily serves a challenge file through Nginx — Nginx must be up. (4) Is Tailscale routing port 80 traffic to spserver from the public internet?
Backend sees all requests as 127.0.0.1 — real IP lost
Proxy headers are missing or the backend isn't reading them. Check the Nginx config: proxy_set_header X-Real-IP $remote_addr; must be inside the location block. In Flask, add ProxyFix middleware: from werkzeug.middleware.proxy_fix import ProxyFix and app.wsgi_app = ProxyFix(app.wsgi_app, x_for=1, x_proto=1, x_host=1). Without ProxyFix, Flask ignores X-Forwarded-* headers by default as a security measure.
How to test routing before DNS is set up
Simulate a real request using a custom Host header: curl -H "Host: jenkins.dustyserver.co.in" http://localhost/. Nginx matches on the Host header — this exercises the full routing logic without needing actual DNS. If you get the expected backend response, the Nginx config is correct. DNS and Certbot can then be configured with confidence.
01 — What's next
Future scope
⟶ Things to explore from here
Wildcard certificate — instead of one certificate per subdomain, obtain a wildcard: sudo certbot certonly --manual --preferred-challenges dns -d "*.dustyserver.co.in". One cert covers all current and future subdomains. Requires DNS validation (add a TXT record at the registrar) rather than HTTP-01. Renewal also requires DNS validation, so full automation needs registrar API integration.
IP allowlist for admin services — restrict Jenkins to the Tailscale network range only: inside the Jenkins location block add allow 100.64.0.0/10; (Tailscale CGNAT range) and deny all;. Jenkins is then only reachable over Tailscale, even if the domain resolves publicly. A simple, high-impact security improvement.
Rate limiting — protect the Flask API from abuse: define a zone in nginx.conf — limit_req_zone $binary_remote_addr zone=api:10m rate=10r/s; — then add limit_req zone=api burst=20 nodelay; to the Flask location block. Requests beyond the rate get a 429 response instead of hitting the backend.
Flask ProxyFix middleware — so Flask reads the proxy headers Nginx sets: from werkzeug.middleware.proxy_fix import ProxyFix and app.wsgi_app = ProxyFix(app.wsgi_app, x_for=1, x_proto=1, x_host=1) in app.py. This makes request.remote_addr return the real client IP and request.url use https:// correctly, even though Flask internally sees HTTP from Nginx.
Nginx static file serving — instead of sending static file requests through Gunicorn/Flask, serve them directly from Nginx: location /static/ { alias /var/www/flask-static/; expires 30d; }. Nginx is much faster at serving static files than Python. Use a bind mount or a shared Docker volume to make the static directory accessible to both the Flask container and host Nginx.
Containerised Nginx (nginx-proxy + acme-companion) — for a fully Dockerised setup, replace host Nginx with the nginx-proxy container. It watches the Docker socket and automatically generates Nginx configs for any container with a VIRTUAL_HOST environment variable. The acme-companion sidecar handles SSL via Let's Encrypt automatically — no manual Certbot commands needed. The entire proxy layer becomes part of the Compose stack.
