Ubuntu Server Install

Converting a spare laptop into a dedicated headless home server — the foundation everything else is built on.

01 — Concept
What is Ubuntu Server

Ubuntu Server is a stripped-down version of Ubuntu Linux — no graphical interface, no desktop apps, just the operating system and a terminal. It is designed to run continuously in the background, serving requests, managing files, and running services without anyone sitting at a screen.

It is the same Ubuntu you may have used on a desktop, but with everything visual removed. You control it entirely through a terminal — either physically connected or, more commonly, over SSH from another machine.

ℹ Ubuntu Server powers millions of web servers, cloud instances, home labs, and production systems worldwide. Learning it here on a spare laptop gives you the same skills used in real DevOps work.
02 — Decision
Why Server edition over Desktop

This is the first decision made in the project. The spare laptop could have run Ubuntu Desktop — but Server was the right choice for several concrete reasons.

Ubuntu Desktop — not used
Runs a full graphical interface (GNOME). Uses ~1-2GB RAM just for the display. Includes apps like a file manager, browser, office suite. All of this consumes resources that should go to your actual services. It was built for humans sitting at a screen, not for a machine running unattended.
Ubuntu Server — chosen
No GUI at all. Uses ~300MB RAM at idle. Every resource goes toward your services — SSH, Nginx, Docker, Samba. Terminal-only means less attack surface, less to break, and faster boot. This is how every real production server works.
Ubuntu Desktop idle: ~1.5GB RAM used ← GUI, apps, display server Ubuntu Server idle: ~300MB RAM used ← just the OS and your services That's 1.2GB freed up for Docker containers, databases, and apps
03 — Decision
Why 24.04 LTS specifically

Ubuntu releases two types of versions. LTS (Long Term Support) versions are released every two years and receive security updates for 5 years. Non-LTS versions are released every 6 months and supported for only 9 months.

For a server you set up once and leave running, this matters enormously. A non-LTS version forces you to upgrade the OS every 9 months. An LTS version lets you set it up and forget it until 2029.

Version Type Supported until Best for
24.04 LTS chosen LTS April 2029 Servers, production systems, anything long-running
24.10 Non-LTS July 2025 Cutting-edge desktop users who want latest features
22.04 LTS LTS April 2027 Still valid — older but stable alternative
20.04 LTS LTS April 2025 End of life — avoid for new installs
04 — Concept
What is a headless server

A headless server is a machine that runs without a monitor, keyboard, or mouse permanently attached. You control it entirely over the network — via SSH from your PC, or via a browser terminal like ttyd.

Traditional setup: Monitor + Keyboard + Mouse ──── physically connected ──── Server Headless setup (what we have): Your PC anywhere ──── SSH over network ──── spserver ──── browser terminal ── spserver spserver sits on a shelf with only power + ethernet. No screen ever needed.
✓ Once SSH is set up (next guide), you never need to physically touch spserver again. Everything is done remotely from your Windows PC or phone.

01 — What we did
The setup in summary

A spare laptop was repurposed as a dedicated server. Ubuntu Server 24.04 LTS was installed on it directly — wiping whatever was previously on the laptop. After install, the machine was configured with a hostname, user account, and basic system settings. From that point forward, all interaction happens over SSH — the laptop is closed and sits somewhere with power and ethernet.

ℹ The Ubuntu Server installer is straightforward — download the ISO, flash to a USB drive using Rufus or Balena Etcher, boot from USB, follow the prompts. During install you set the hostname (spserver) and the first user account (sipu).
02 — First boot
Steps after first login
1
Update the system
▼

The first thing to do on any fresh Ubuntu install is update. The ISO image you downloaded may be weeks or months old — there will be security patches and software updates waiting.

bash
# Refresh the list of available packages from Ubuntu's servers
sudo apt update

# Download and install all available updates (-y = yes to all prompts)
sudo apt upgrade -y

# Remove packages that were installed as dependencies but no longer needed
sudo apt autoremove -y
2
Verify hostname and user
▼

Confirm the hostname and username were set correctly during install. These matter — the hostname appears in your terminal prompt and SSH connections, and the username is what you log in with everywhere.

bash
hostname           # should print: spserver
whoami             # should print: sipu
lsb_release -a     # confirms: Ubuntu 24.04 LTS
uname -r           # shows current Linux kernel version
3
Check system resources
▼

Good to know your baseline — how much RAM, disk, and CPU you have available before adding any services.

bash
free -h            # RAM — total, used, available
df -h              # Disk — total, used, free per partition
nproc              # Number of CPU cores
lscpu              # Detailed CPU info
htop               # Interactive real-time resource monitor (install if not present)
4
Install essential tools
▼

A fresh Ubuntu Server install is minimal — some useful tools are not included by default. These are safe to install immediately and make everything easier going forward.

bash
sudo apt install -y \
  htop \            # interactive process monitor
  curl \            # transfer data from URLs (used constantly)
  wget \            # download files from internet
  git \             # version control
  nano \            # simple terminal text editor
  net-tools \       # includes ifconfig, netstat
  unzip             # extract zip archives
5
Enable automatic security updates
▼

Since this server runs 24/7 and is exposed to the internet via Cloudflare Tunnel, keeping it patched matters. Ubuntu's unattended-upgrades package applies security patches automatically without you having to manually update.

bash
sudo apt install unattended-upgrades -y
sudo dpkg-reconfigure --priority=low unattended-upgrades
# Select "Yes" when prompted

02 — Package management
apt commands
sudo apt update
Refresh package list from Ubuntu repositories — always run before installing
sudo apt upgrade -y
Install all available updates for installed packages
sudo apt install PACKAGE -y
Install a package. Replace PACKAGE with the name (e.g. nginx, git, curl)
sudo apt remove PACKAGE
Remove a package (keeps config files)
sudo apt purge PACKAGE
Remove a package and its config files
sudo apt autoremove -y
Remove packages installed as dependencies that are no longer needed
apt list --installed
List all installed packages
apt search KEYWORD
Search for packages by name or description
apt show PACKAGE
Show details about a package before installing
03 — Service management
systemctl commands

systemctl manages all background services (daemons) on Ubuntu. Services you add — SSH, Nginx, Docker, ttyd, Filebrowser, cloudflared — are all managed through systemctl.

sudo systemctl status SERVICE
Check if a service is running — green active = good
sudo systemctl start SERVICE
Start a service now
sudo systemctl stop SERVICE
Stop a running service
sudo systemctl restart SERVICE
Stop then start a service — use after config changes
sudo systemctl reload SERVICE
Reload config without restarting (Nginx supports this)
sudo systemctl enable SERVICE
Mark service to auto-start on every boot
sudo systemctl disable SERVICE
Stop service from auto-starting on boot
sudo systemctl list-units --type=service
List all currently active services
sudo journalctl -u SERVICE -f
Follow live logs for a specific service
sudo journalctl -xe
View recent system logs including errors
ℹ The d at the end of service names stands for daemon — a background process. sshd = SSH daemon, nginx = Nginx web server, dockerd = Docker daemon. Enable makes them start at boot. Start starts them right now. Always do both.
01 — Issues
Common problems and fixes
apt update fails — cannot reach repository
Network not configured yet, or DNS not working. Check internet: ping 8.8.8.8. If ping works but apt fails, DNS issue — check /etc/resolv.conf. If ping fails, network is not up — see the Netplan guide for static IP setup.
Cannot sudo — sipu is not in the sudoers file
During install you may have created a user without sudo rights. Boot into recovery mode, mount filesystem read-write, and run usermod -aG sudo sipu. Reboot normally. Alternatively reinstall — it is faster on a fresh machine.
hostname shows localhost instead of spserver
The hostname was not set during install. Fix it now: sudo hostnamectl set-hostname spserver. Also update /etc/hosts — replace the old name with spserver on the 127.0.1.1 line. Takes effect immediately.
Server does not boot after install — stuck at GRUB or blank screen
Likely a UEFI/BIOS boot order issue. Enter BIOS (usually F2, F12, or Del at startup), find boot order, and set the internal drive (not USB) as first. Also check if Secure Boot is enabled — some older Ubuntu versions need it disabled.
Terminal shows weird characters or garbled text over SSH
Terminal encoding mismatch. Set locale: sudo locale-gen en_US.UTF-8 then sudo update-locale LANG=en_US.UTF-8. Logout and SSH back in. Also make sure your SSH client (Windows Terminal, PuTTY) is set to UTF-8.
apt upgrade asks about config files — what to choose?
When upgrading, if a config file was modified by you and a new version exists, Ubuntu asks what to do. On a fresh install — choose Y (keep the package maintainer's version). On an existing configured server — choose N (keep your version). Read the diff shown before deciding.
01 — What's next
Future scope
⟶ Things to explore from here
Proxmox VE — install Proxmox on spserver to run multiple isolated virtual machines on the same hardware. Great Phase 4+ project.
Ansible — automate the entire server setup with Ansible playbooks. Run one command and get a fresh Ubuntu install fully configured. Eliminates manual steps.
Cloud-init — auto-configure a server on first boot using a YAML file. Used by every major cloud provider (AWS, GCP, Azure) for provisioning instances. Same concept as Ansible but at boot time.
Automated OS upgrades — set up fully unattended upgrades including kernel updates. Requires careful testing but keeps the server secure without manual intervention.
Monitoring — install Prometheus + Grafana to monitor CPU, RAM, disk, and network usage over time. Visualise your server's health on a dashboard at a subdomain.
Second server — repurpose another spare machine and set up high availability, failover, or distributed services between the two.
