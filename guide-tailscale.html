Tailscale VPN

Secure SSH access to spserver from any network in the world — phone, work laptop, a cafe, anywhere — without exposing a single port to the internet. The emergency recovery lifeline for the entire stack.

01 — Concept
What is Tailscale

Tailscale is a zero-config VPN that connects your devices into a private encrypted network — called a tailnet. Every device gets a stable IP address in the 100.x.x.x range. These devices can reach each other directly, as if they were on the same local network, regardless of where they physically are in the world.

You install Tailscale on spserver and your Windows PC. Both login with the same Tailscale account. They automatically find each other and establish an encrypted tunnel. From that point, ssh sipu@100.x.x.x works from any network — mobile data, work WiFi, a hotel — without any router configuration, port forwarding, or firewall rules.

✓ Tailscale is free for personal use — up to 100 devices, unlimited bandwidth, no time limits. No credit card required. Just a Google, GitHub, or Microsoft account to sign in.
02 — Why we need it
The problem Tailscale solves

Airtel uses CG-NAT — Carrier Grade NAT. Your router's WAN IP is 100.220.x.x, which is Airtel's internal address space, not a real public IP. Incoming connections from the internet never reach your router — Airtel's NAT layer drops them before they get there.

This means standard port forwarding cannot work. SSH on port 22 is unreachable from outside the home network by direct connection. We need a method that doesn't rely on incoming connections at all.

Without Tailscale — external SSH fails: Your phone ──► ssh sipu@106.215.x.x ──► Airtel CG-NAT ← packet dropped here never reaches router or spserver With Tailscale — works from anywhere: Your phone ──► ssh sipu@100.x.x.x ──► Tailscale tunnel ──► spserver No incoming connections needed — all outgoing ↑
03 — How it works
How Tailscale bypasses CG-NAT

CG-NAT only blocks unsolicited incoming connections. Outgoing connections always work. Tailscale exploits this — both devices reach out to Tailscale's coordination servers simultaneously. These servers help the devices discover each other and broker a direct connection using a technique called NAT traversal.

Phase 1 — Coordination (both devices reach OUT) spserver ──── outgoing HTTPS ────► login.tailscale.com Your Windows PC ──── outgoing HTTPS ────► login.tailscale.com CG-NAT allows both ↑ (they are outgoing) Coordination server exchanges IP/port info between devices Phase 2 — Direct tunnel established (STUN/DERP) spserver ◄═══════ encrypted WireGuard tunnel ═══════► Your Windows PC spserver IP on tailnet: 100.78.211.x (example) Direct P2P when possible, DERP relay server otherwise

When direct P2P connection is not possible (some strict NAT configurations), Tailscale uses DERP relay servers — Tailscale's own servers that relay encrypted traffic between devices. Even in relay mode, Tailscale is fast enough for SSH, Samba, and most use cases.

ℹ Run tailscale netcheck to see your connection type. "Direct" means P2P (fastest). "Relay" means traffic goes through a DERP server (still encrypted, slightly higher latency). With Airtel CG-NAT, you usually get direct connection.
04 — Under the hood
WireGuard — what Tailscale is built on

Tailscale is built entirely on top of WireGuard — a modern, extremely fast VPN protocol built into the Linux kernel since version 5.6. WireGuard is simpler, faster, and more secure than older protocols like OpenVPN or IPsec.

The difference: raw WireGuard requires manually generating key pairs for every device, manually exchanging public keys, and manually configuring peers. For 2 devices that's manageable. For 10+ it becomes painful. Tailscale takes WireGuard and adds automatic key management, peer discovery, NAT traversal, and a coordination layer so you never touch any of that manually.

Raw WireGuard
You manage key pairs, configure each peer manually, set up a server, handle NAT yourself. Powerful but complex. Better when you need full control or can't use Tailscale's infrastructure.
Tailscale on WireGuard
Install, login, done. Keys managed automatically. Peers discovered automatically. NAT traversal handled automatically. Same WireGuard encryption and performance — zero manual configuration.
05 — Use cases
How we use Tailscale on spserver
🔑
Remote SSH
Primary use. ssh sipu@100.x.x.x works from any network worldwide. Mobile hotspot, work WiFi, hotel — always connects.
🚨
Emergency recovery
If Cloudflare Tunnel breaks, Nginx crashes, or UFW locks out SSH — Tailscale still works independently. The recovery path when everything else fails.
📁
Remote Samba
Connect to \\100.x.x.x\shared from any Tailscale device. Full network drive access anywhere — not just the home network.
🔍
Debug local services
Access services running on local ports (not on Cloudflare Tunnel) remotely. http://100.x.x.x:PORT reaches any service on spserver.
✗ Never remove Tailscale from spserver. It is the last resort access method. If cloudflared goes down, if Nginx breaks, if UFW gets misconfigured — Tailscale is how you SSH in and fix it. It operates completely independently of everything else.

1
Sign up at tailscale.com
▼

02 — Install on server
Step 2 — Install Tailscale on spserver
2
Install using the official script
▼

Tailscale provides a one-line install script that detects your Linux distro (Ubuntu 24.04 in our case), adds their apt repository, and installs the latest stable version. It also sets up the systemd service to auto-start on boot.

bash — run on spserver
# Official one-line install script
curl -fsSL https://tailscale.com/install.sh | sh

# Verify installation
tailscale version
# Should print: 1.x.x

# Connect to your Tailscale account
# This prints a URL — open it in your browser
sudo tailscale up

# Visit the URL shown, login with your account
# Click "Connect" — spserver joins your tailnet

# Get spserver's Tailscale IP — save this
tailscale ip
# Example output: 100.78.211.34
# This IP is stable — it never changes for this device
3
Verify service is enabled and running
▼

The install script should enable auto-start automatically, but worth verifying. If spserver reboots and Tailscale doesn't start, you lose the ability to SSH in remotely — the install script should handle this but check to be sure.

bash
# Check Tailscale daemon status
sudo systemctl status tailscaled
# Look for: Active: active (running)

# Verify it is set to auto-start on boot
sudo systemctl is-enabled tailscaled
# Should print: enabled

# If not enabled, enable it
sudo systemctl enable tailscaled

# Check connection status
tailscale status
# Shows all devices in your tailnet and their status

04 — Test
Step 4 — Verify SSH via Tailscale works
5
SSH into spserver using Tailscale IP
▼

The real test is connecting from a network that is NOT your home — use your mobile hotspot. Disconnect your PC from home WiFi, enable mobile hotspot, then try SSH. If it works, Tailscale is correctly bypassing CG-NAT.

Windows Terminal / PowerShell
# SSH using Tailscale IP (works from any network)
ssh sipu@100.x.x.x   # replace with your spserver Tailscale IP

# Add a shortcut to ~/.ssh/config on Windows
# C:\Users\subha\.ssh\config
Host spserver-remote
    HostName 100.x.x.x    # spserver Tailscale IP
    User sipu

# Then connect with:
ssh spserver-remote
✓ Once confirmed working on mobile hotspot, Tailscale SSH works from anywhere — office, travel, cafe. The Tailscale IP never changes so the SSH config entry stays valid forever.
05 — Key expiry
Step 5 — Disable key expiry for spserver
6
Disable expiry in Tailscale admin
▼

By default, Tailscale auth keys expire after 180 days. After expiry, tailscale up asks you to re-authenticate — which means opening a browser URL on the server. For a headless server this is annoying. Disabling expiry means Tailscale stays connected indefinitely without re-auth.

Tailscale admin — browser steps
# 1. Go to: https://login.tailscale.com/admin/machines
# 2. Find "spserver" in the list
# 3. Click the three-dot menu (⋯) next to spserver
# 4. Select "Disable key expiry"
# 5. Confirm
# spserver now stays connected indefinitely

# Alternatively via CLI:
sudo tailscale up --accept-routes
01 — Tailscale commands
All useful Tailscale commands
tailscale status
List all devices in your tailnet and their connection status (online/offline, IPs)
tailscale ip
Show this device's Tailscale IP — the stable 100.x.x.x address
tailscale ip -4
Show only the IPv4 Tailscale address
sudo tailscale up
Connect to Tailscale / re-authenticate if expired
sudo tailscale down
Disconnect from Tailscale (device stays registered but goes offline)
tailscale ping 100.x.x.x
Ping another Tailscale device — confirms connectivity and shows latency
tailscale netcheck
Diagnose network — shows NAT type, UDP availability, latency to DERP servers, connection type (direct/relay)
tailscale version
Show installed Tailscale version
tailscale logout
Log out and remove this device from the tailnet
sudo tailscale up --reset
Re-authenticate — generates a new auth URL to login again
sudo systemctl status tailscaled
Check Tailscale daemon status — should show active (running)
sudo systemctl restart tailscaled
Restart Tailscale daemon if connection drops
sudo journalctl -u tailscaled -f
Follow Tailscale daemon logs live — useful for debugging connection issues

01 — What went wrong
Blockers hit during Tailscale setup
⚠ Blocker 1
tailscale up showed a URL but no browser on server
Running sudo tailscale up on the headless server printed a long authentication URL in the terminal. The server has no browser — couldn't just click it. Wasn't immediately obvious how to authenticate.
✓ Fix
Copy the URL and open it on any device
The URL printed by tailscale up can be opened on any device — your Windows PC, phone, tablet. Copy the full URL from the terminal output (it starts with https://login.tailscale.com/a/...), paste it into your Windows PC's browser, login with your account, and click Connect. spserver is then authenticated. You don't need to open the URL on the server itself.
⚠ Blocker 2
Tailscale stopped working after 180 days — re-auth required
After ~6 months, SSH via Tailscale stopped connecting. Running tailscale status showed spserver as offline. On the server, tailscale status said authentication was expired and re-authentication was needed.
✓ Fix
Disable key expiry in Tailscale admin dashboard
Default Tailscale keys expire after 180 days. Re-authenticate: sudo tailscale up → copy the URL → open on Windows browser → login. Then permanently disable expiry in the admin dashboard (login.tailscale.com/admin/machines → spserver → ⋯ → Disable key expiry). This was the moment we learned to do this immediately after setup — it's now in the setup steps.
01 — Issues and fixes
Common Tailscale problems
SSH via Tailscale IP times out or refuses connection
Check in order: (1) Is Tailscale running on spserver? sudo systemctl status tailscaled. (2) Is spserver showing as online in Tailscale admin? Visit login.tailscale.com/admin/machines. (3) Is sshd running on spserver? Tailscale provides the network path — sshd must still be running for SSH to work: sudo systemctl status ssh. (4) Has the auth key expired? Run tailscale status on spserver — if it shows "Needs reauthentication", run sudo tailscale up.
spserver shows as offline in Tailscale dashboard
spserver rebooted and Tailscale didn't reconnect, or the key expired. If you can access spserver locally: sudo tailscale up. If key expired, this generates a new URL — open it on any device. If you can't reach spserver at all, you need physical access or to wait until someone is on the local network to SSH in via 192.168.31.150.
Tailscale connection works but is slow / high latency
Run tailscale netcheck to check connection type. "relay" mode means traffic goes through Tailscale's DERP servers, adding latency. This is normal for some network configurations. For SSH it's fast enough. For large file transfers via Samba it may feel slower than local network. Direct mode (P2P) is usually achievable — try restarting tailscaled: sudo systemctl restart tailscaled.
Tailscale installed but can't find the IP
Run tailscale ip — if it says "Tailscale is stopped" or returns nothing, Tailscale isn't connected. Run sudo tailscale up to connect. If it prints an IP, you're good. The IP is also visible in the Tailscale admin dashboard next to each device name.
Can I use Tailscale and Cloudflare Tunnel at the same time?
Yes — they are completely independent and do not interfere. Cloudflare Tunnel handles public web traffic. Tailscale handles private SSH and admin access. Both run simultaneously on spserver. This is the recommended setup — public services via Cloudflare, private access via Tailscale.
01 — What's next
Future scope
⟶ Things to explore from here
Tailscale SSH — enable Tailscale's built-in SSH server (sudo tailscale up --ssh). Handles SSH authentication through Tailscale identity — no sshd config needed, no SSH keys to manage per device. Experimental but powerful.
Tailscale ACLs — in the admin dashboard, write access control rules to define exactly which devices can reach which others and on which ports. Useful if you add more devices and want to restrict what your phone can access vs your PC.
Tailscale Funnel — expose a local service publicly via Tailscale's infrastructure without Cloudflare Tunnel. Limited to Tailscale's domain (not your custom domain) but useful for quick sharing or testing.
Headscale — self-hosted open-source replacement for Tailscale's coordination server. Run your own Tailscale backend on a cheap VPS. All device coordination happens through your server — no data through Tailscale's infrastructure. Good Phase 4+ project.
WireGuard directly — learn raw WireGuard to understand what Tailscale is doing underneath. Highly relevant for DevOps and network security roles. Generate key pairs, configure peers, manage routing tables manually. Builds deep networking understanding.
Mobile access — install Tailscale on your Android or iOS phone. SSH into spserver from your phone using a terminal app like Termius or JuiceSSH. Full server access from anywhere with just a phone.
