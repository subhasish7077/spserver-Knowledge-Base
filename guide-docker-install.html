Docker Install

Installing the official Docker Engine on spserver — the full stack including the daemon, CLI, Compose plugin, and Buildx. The foundation for all containerised services.

01 — Why Docker
What Docker gives us

Docker packages each application into a container — a self-contained, isolated environment that includes everything the app needs: the code, runtime, libraries, environment variables, and config. Containers share the host's Linux kernel but are completely isolated from each other and from the host OS.

Without Docker — bare metal installs
Install Flask directly on Ubuntu. Install MySQL directly. Install Node directly. Over time: conflicting library versions, hard-to-reproduce environments, messy upgrades, no easy cleanup. Moving to another server means manually recreating everything.
With Docker — containers
Each service runs in an isolated container with its own dependencies. No conflicts. Start, stop, update, and remove cleanly. Move to any Docker host and it runs identically. Industry standard for every modern DevOps workflow.
Without Docker: Flask app needs Python 3.9 + MySQL needs libssl 1.0 + Node app needs OpenSSL 3 All installed on same Ubuntu → version conflicts → pain With Docker: Flask container ← Python 3.9, its own libs, isolated MySQL container ← libssl 1.0, its own libs, isolated Node container ← OpenSSL 3, its own libs, isolated All on same Ubuntu, zero conflicts, clean separation
ℹ Containers are not virtual machines. They share the host kernel — so no hardware emulation overhead, no separate OS boot, minimal resource usage. A Docker container typically starts in under a second and uses only a few MB of RAM on top of the application itself.
02 — What gets installed
The Docker components we install

Docker is not a single monolithic program — it is split into separate components that can be updated independently. The official install method installs all of them:

docker-ce
Docker Engine (daemon)
The core background service — dockerd. Manages containers, images, networks, and volumes. Everything else talks to this.
docker-ce-cli
Command line interface
The docker command you type. Sends instructions to the Docker daemon via a Unix socket. Could run on a different machine from the daemon.
containerd.io
Container runtime
The actual container runtime that runs containers at the OS level. Docker Engine delegates to containerd. Kubernetes uses containerd directly without Docker Engine.
docker-buildx-plugin
Extended build system
Modern image building with BuildKit. Faster builds, better caching, multi-platform support. Required for docker build on modern Docker.
docker-compose-plugin
Multi-container orchestration
The docker compose command (note: no hyphen). Manages multi-container applications from a YAML file. Essential for everything we build in Phase 2.
ℹ Note the difference: docker-compose-plugin (what we install) provides docker compose as a Docker CLI plugin. The old standalone docker-compose (with hyphen, version 1.x) is a separate Python tool — deprecated and no longer maintained. Always use the plugin version.
03 — Installation method
Why we use the official Docker repository — not apt

Ubuntu's default package repositories contain a package called docker — but this is not Docker. It is an entirely unrelated system tray application. Installing it won't give you containers at all. Even the docker.io package in Ubuntu's repos is an older community-maintained version that lags behind the official release by months or more.

❌ sudo apt install docker
Installs a system tray application called "docker" — has nothing to do with containers. Complete wrong package.
⚠ sudo apt install docker.io
Installs the community-maintained Debian/Ubuntu Docker package. Older version, fewer features, slower updates. Does not include Compose plugin.
⚠ snap install docker
Snap-packaged Docker. Works but has filesystem restrictions (snaps run in a sandbox) that can cause issues with volume mounts and bind mounts.
✓ Official Docker repo
Latest stable version directly from Docker Inc. All components included. Verified GPG signatures. Automatic security updates. Industry standard method.

01 — Clean slate
Step 1 — Remove any conflicting packages
1
Remove old or wrong Docker packages
▼

If any version of Docker was previously installed — via apt, snap, or any other method — it must be removed first. Conflicting packages cause the official install to fail or behave unexpectedly. On a fresh Ubuntu Server install these likely don't exist but the removal command does nothing if they aren't installed — safe to run either way.

bash
sudo apt remove docker docker-engine docker.io containerd runc -y
# These may not be installed — that's fine, apt will just say "not installed"
# What matters is they are absent before we add the official repo
02 — Add repository
Step 2 — Add Docker's official apt repository
2
Install dependencies and add Docker GPG key
▼

Before apt can use Docker's repository, it needs to verify that packages are genuinely from Docker Inc. — not tampered with. The GPG key is Docker's cryptographic signature that apt uses to verify every package before installing. This is why we download and save the key first.

bash
# Install packages needed to use apt over HTTPS
sudo apt update
sudo apt install -y ca-certificates curl gnupg lsb-release

# Create directory for apt keyrings
sudo install -m 0755 -d /etc/apt/keyrings

# Download Docker's official GPG key and save it
curl -fsSL https://download.docker.com/linux/ubuntu/gpg | sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg
sudo chmod a+r /etc/apt/keyrings/docker.gpg
3
Add the Docker apt repository
▼

This adds Docker's repository to apt's sources list, pointing to the correct architecture and Ubuntu release. The $(dpkg --print-architecture) resolves to amd64 on spserver, and $(lsb_release -cs) resolves to noble (Ubuntu 24.04's codename). These auto-detect your system so the command works on any Ubuntu version.

bash
# Add Docker repository to apt sources
echo \
  "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.gpg] https://download.docker.com/linux/ubuntu \
  $(lsb_release -cs) stable" | sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

# Refresh apt package list — now includes Docker packages
sudo apt update

# Confirm Docker packages are now available from the official repo
apt-cache policy docker-ce
# Should show: Candidate: 5:27.x.x-1~ubuntu.24.04~noble
#              download.docker.com as the source
03 — Install
Step 3 — Install Docker Engine and all components
4
Install all Docker components in one command
▼

One apt command installs the complete Docker stack — engine, CLI, runtime, build tools, and Compose. All from Docker's official repository, all signed with their GPG key, all the latest stable versions.

bash
sudo apt install -y \
  docker-ce \
  docker-ce-cli \
  containerd.io \
  docker-buildx-plugin \
  docker-compose-plugin

# Enable Docker daemon to start on boot
sudo systemctl enable docker

# Start Docker daemon now
sudo systemctl start docker

# Verify Docker daemon is running
sudo systemctl status docker
# Look for: Active: active (running)
04 — Permissions
Step 4 — Add sipu to the docker group
5
Grant sipu permission to run Docker without sudo
▼

By default, the Docker socket (/var/run/docker.sock) is only accessible by root and members of the docker group. Without this, every docker command requires sudo. Adding sipu to the docker group gives direct access to the Docker socket — meaning docker ps works without sudo.

bash
# Add sipu to the docker group
sudo usermod -aG docker sipu

# IMPORTANT: Log out and back in for the group change to take effect
# The group membership is only applied at login time
exit

# SSH back in
# ssh sipu@192.168.31.150

# Verify you are now in the docker group
groups
# Should include: docker
⚠ Security note: members of the docker group have effective root access to the host — a container can be configured to mount the host filesystem. Only add trusted users. In production environments, consider using rootless Docker (Podman) instead for stronger isolation.
05 — Verify
Step 5 — Confirm everything works
bash — run after logging back in
# Check Docker version — installed version: 29.x.x
docker --version
# Docker version 29.x.x, build xxxxxxx

# Check Docker Compose version
docker compose version
# Docker Compose version v2.x.x

# Run the hello-world container — ultimate verification
# Pulls image from Docker Hub, runs it, prints output
docker run hello-world
# Should print: "Hello from Docker!"
# This confirms: pull works, run works, networking works

# Verify no sudo needed
docker ps
# Should work without sudo and show empty container list

# Check Docker daemon info
docker info
# Shows storage driver, runtime, server info
✓ If docker run hello-world prints "Hello from Docker!" without using sudo — Docker is fully installed, the daemon is running, hub access works, and permissions are correct. You're ready for Phase 2.
01 — Docker commands
Essential post-install Docker commands
docker --version
Show Docker Engine version
docker compose version
Show Docker Compose plugin version
docker info
Full Docker daemon info — storage driver, runtime, OS, kernel version, total containers/images
docker run hello-world
Pull and run the minimal hello-world container — confirms Docker is working end to end
docker ps
List running containers — should work without sudo after adding sipu to docker group
docker ps -a
List all containers including stopped ones
docker images
List all downloaded images on disk
docker system df
Show Docker disk usage — how much space images, containers, volumes and build cache are using
docker system prune
Remove stopped containers, unused networks, dangling images — free disk space safely
docker system prune -a
Remove everything unused — including images not referenced by any container. More aggressive cleanup.

01 — What went wrong
Blockers hit during Docker install
⚠ Blocker 1
docker: command not found after install — wrong package installed
Initially ran sudo apt install docker without checking the package name carefully. The command succeeded but docker --version returned "command not found". The docker package in Ubuntu's repos is a completely different application — a system management tool, not Docker the container platform.
✓ Fix
Remove wrong package, use official Docker repository method
Removed the wrong package: sudo apt remove docker. Then followed the official Docker documentation to add Docker's own repository and install docker-ce, docker-ce-cli, containerd.io, and the plugins. This installs the real Docker Engine from Docker Inc. directly.
⚠ Blocker 2
Permission denied — docker commands require sudo even after group add
After running sudo usermod -aG docker sipu, tried docker ps immediately. Got "permission denied while trying to connect to the Docker daemon socket". The group change wasn't taking effect.
✓ Fix
Must log out and log back in — group changes apply at login
Linux group membership changes only take effect when you start a new login session. The current SSH session still has the old group list. Fix: exit to close the SSH session, then ssh sipu@192.168.31.150 to reconnect. After reconnecting, groups showed docker in the list and docker ps worked without sudo. Alternatively, newgrp docker applies the group change in the current session without re-logging in.
01 — Issues and fixes
Common Docker install problems
docker: command not found
Wrong package installed or Docker not in PATH. Check: (1) Which Docker package is installed: apt list --installed | grep docker. If it shows docker (not docker-ce) — wrong package. Remove it and install from official repo. (2) If docker-ce is installed but command not found — PATH issue: check which docker, should show /usr/bin/docker. Try relogging.
Cannot connect to Docker daemon — permission denied
Running docker commands without sudo and not in docker group, or group change not applied yet. Fix: (1) Check groups: groups — if docker not listed, log out and back in. (2) If still not working: sudo usermod -aG docker sipu then log out/in again. (3) Verify socket permissions: ls -la /var/run/docker.sock — should show group docker. (4) As a temporary workaround: use sudo docker until group is applied.
docker run hello-world fails — cannot pull image
Docker cannot reach Docker Hub. Check internet connectivity: curl -I https://hub.docker.com. If that fails — DNS issue or firewall. Check: ping 8.8.8.8 (internet) and ping hub.docker.com (DNS). If DNS fails, check /etc/resolv.conf. Docker daemon also needs outgoing internet access — UFW only blocks incoming by default so this is usually not a UFW issue.
apt update fails after adding Docker repo — NO_PUBKEY error
The GPG key was not saved correctly. Re-download the key: curl -fsSL https://download.docker.com/linux/ubuntu/gpg | sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg then ensure permissions are correct: sudo chmod a+r /etc/apt/keyrings/docker.gpg. Then run sudo apt update again.
How to check which Docker version is installed
docker --version shows the CLI version. docker info | grep "Server Version" shows the daemon version (should match). docker compose version shows Compose plugin version. Our installed version: Docker Engine 29.x.x. To update later: sudo apt update && sudo apt upgrade docker-ce docker-ce-cli containerd.io docker-buildx-plugin docker-compose-plugin.
01 — What's next
Future scope
⟶ Things to explore from here
Rootless Docker — run Docker as a non-root user without the docker group. Provides stronger security isolation — a container escape doesn't give root access to the host. Enabled with dockerd-rootless-setuptool.sh install. Recommended for production.
Docker resource limits — constrain how much CPU and RAM each container can use. Prevents a runaway container from taking down the whole server. Set in docker-compose.yml: deploy.resources.limits.memory: 512m. Important as more services are added.
Private Docker registry — host your own Docker image registry on spserver instead of pushing images to Docker Hub. Install registry:2 container. Useful for storing custom images privately without cloud dependency.
Docker daemon configuration — edit /etc/docker/daemon.json to configure log rotation (prevents logs from filling disk), change storage driver, configure DNS, set default ulimits. Important for a long-running server.
Podman as an alternative — daemonless, rootless-by-default container tool from Red Hat. Docker-compatible CLI — most docker commands work unchanged. No central daemon means no single point of failure. Worth learning alongside Docker.
k3s — Kubernetes on spserver — Phase 4 goal. k3s is a lightweight Kubernetes distribution that runs on spserver's hardware. Once Docker is comfortable, k3s is the natural next step — uses containerd directly, Docker knowledge transfers fully.
