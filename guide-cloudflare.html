Cloudflare Tunnel

Making spserver's services publicly accessible on the internet — despite CG-NAT, without port forwarding, with free SSL, DDoS protection, and zero exposed home IP.

01 — Concept
What is Cloudflare Tunnel

Cloudflare Tunnel is a service that creates a secure outbound connection from your server to Cloudflare's global network. Once connected, Cloudflare forwards public internet traffic through this tunnel to your server — without your server ever needing to accept unsolicited incoming connections or expose a port to the internet.

This is the solution to our CG-NAT problem for public-facing services. Where Tailscale handles private SSH access, Cloudflare Tunnel handles everything that needs to be publicly accessible — the website, browser terminal, file server, Flask app.

Without Cloudflare Tunnel
CG-NAT means nobody can reach spserver from the internet. Port forwarding doesn't work. Nginx runs but nobody can visit the website. All services are trapped behind Airtel's NAT.
With Cloudflare Tunnel
dustyserver.co.in is publicly accessible worldwide. Free SSL. DDoS protection. Your home IP is never exposed. Adding a new service takes 3 steps. Zero router configuration.
ℹ Cloudflare Tunnel is completely free. There are no connection limits, no bandwidth caps, and no time restrictions on the free tier. The only requirement is a Cloudflare account and a domain with DNS managed by Cloudflare.
02 — Architecture
How the tunnel works end to end

The key insight is the same as Tailscale — outgoing connections always work behind CG-NAT. cloudflared on spserver initiates an outgoing connection to Cloudflare's edge servers and keeps it alive permanently. When a visitor requests your domain, Cloudflare receives the request at their edge and forwards it through the existing persistent connection to your server.

Setup phase (done once): cloudflared on spserver ──── outgoing connection ────► Cloudflare edge CG-NAT allows this ↑ persistent tunnel maintained Request phase (every visitor): Visitor browser ──► dustyserver.co.in ↓ DNS resolves to Cloudflare IP (104.21.x.x) Cloudflare edge receives HTTPS request ↓ forwards through existing tunnel cloudflared on spserver ↓ forwards to local service localhost:80 (Nginx) or :7681 (ttyd) or :8080 (Filebrowser) ↓ response travels back the same path Visitor sees the page

Notice what the visitor never sees: your home IP address. Their DNS query resolves to Cloudflare's IP (104.21.x.x or 172.67.x.x) — Cloudflare's servers — not your server. This is a significant privacy and security benefit. Even a DDoS attack hits Cloudflare's massive infrastructure, not your home server.

03 — Component
What is cloudflared

cloudflared is the daemon (background service) that runs on spserver and maintains the tunnel connection. It is a single binary downloaded from Cloudflare, installed at /usr/local/bin/cloudflared, and managed by systemd just like sshd or Nginx.

cloudflared reads a config file (/etc/cloudflared/config.yml) that tells it which tunnel to connect to and how to route incoming requests to local services. It maintains multiple connections to Cloudflare's edge for redundancy — if one connection drops, the others keep the tunnel alive.

cloudflared process ← started by systemd at boot reads: /etc/cloudflared/config.yml connects to: Cloudflare edge (4 redundant connections) routes incoming requests based on hostname: terminal.dustyserver.co.in ──► http://localhost:7681 fileserver.dustyserver.co.in ──► http://localhost:8080 flask.dustyserver.co.in ──► http://localhost:5000 dustyserver.co.in ──► http://localhost:80 www.dustyserver.co.in ──► http://localhost:80 anything else ──► 404
04 — Routing
How ingress rules work

Ingress rules are the routing table for your tunnel. When a request arrives, cloudflared checks the hostname of the request against each rule in order — top to bottom, first match wins. The matched rule's service is where the request gets forwarded.

⚠ Three important rules: (1) Order matters — put more specific hostnames before general ones. (2) The catch-all http_status:404 must always be the last rule. (3) Use second-level subdomains only (flask.dustyserver.co.in) — third-level subdomains (flask.test.dustyserver.co.in) caused DNS issues in our setup.
RuleWhat it means
hostname: terminal.dustyserver.co.inMatch requests for this exact hostname.
service: http://localhost:7681Forward matching requests to ttyd running on port 7681. localhost here means the server itself — not the internet.
service: http_status:404Catch-all — must be last. Any hostname not matched by a rule above returns a 404 error. Without this, cloudflared will refuse to start.

Every time you add a new service — a new Docker container, a new app, a new tool — you follow the same pattern: add an ingress rule, add a DNS record, restart cloudflared. The tunnel itself never changes. You are just adding new routing rules.

01 — Install
Step 1 — Install cloudflared
1
Download and install the cloudflared binary
▼

cloudflared is distributed as a single binary — no package manager required. Download the latest release, make it executable, and move it to a system PATH location so it's available to all users including root.

bash — on spserver
# Download the latest cloudflared binary for Linux amd64
curl -L https://github.com/cloudflare/cloudflared/releases/latest/download/cloudflared-linux-amd64 -o cloudflared

# Make it executable
chmod +x cloudflared

# Move to system PATH — accessible by all users and root
sudo mv cloudflared /usr/local/bin/

# Verify installation
cloudflared version
# Should show: cloudflared version 2026.x.x
02 — Authenticate
Step 2 — Authenticate with Cloudflare
2
Login to Cloudflare account
▼

This command opens a URL. Open it in your browser, login to your Cloudflare account, and select which domain to associate with this tunnel. Cloudflare then creates a certificate file in ~/.cloudflared/cert.pem that proves cloudflared is authorised to act on behalf of your account.

bash
cloudflared tunnel login

# A URL is shown — open it in your browser
# Login to Cloudflare → select dustyserver.co.in → Authorize
# Certificate saved to: /root/.cloudflared/cert.pem (or ~/.cloudflared/cert.pem)
03 — Create tunnel
Step 3 — Create the tunnel
3
Create a named tunnel
▼

The tunnel is a named entity in Cloudflare's system. We named it spserver after the machine. The create command generates a unique tunnel ID and a credentials JSON file — both are needed for the config.

bash
# Create a tunnel named "spserver"
cloudflared tunnel create spserver

# Output shows the tunnel UUID — save this, you need it
# Example: Created tunnel spserver with id b72cf76b-0820-44cb-90df-f4c58ed87184

# Credentials file saved to:
# ~/.cloudflared/b72cf76b-0820-44cb-90df-f4c58ed87184.json

# Verify the tunnel was created
cloudflared tunnel list
ℹ Our tunnel details: Name: spserver — ID: b72cf76b-0820-44cb-90df-f4c58ed87184. The credentials JSON is the private key for this tunnel — keep it safe and never commit it to version control.
04 — Configure
Step 4 — Create the config file
4
Create /etc/cloudflared/ and write config.yml
▼

Why /etc/cloudflared/ and not ~/.cloudflared/? When cloudflared runs as a systemd service, it runs as root. The ~ for root is /root/ — not your user's home directory. If you put config in /home/sipu/.cloudflared/, the service cannot find it. /etc/cloudflared/ is always accessible by root regardless of which user created the files.

bash
# Create the system-wide config directory
sudo mkdir -p /etc/cloudflared

# Copy credentials JSON to system location
sudo cp ~/.cloudflared/b72cf76b-0820-44cb-90df-f4c58ed87184.json /etc/cloudflared/

# Create the config file
sudo nano /etc/cloudflared/config.yml

Paste this into config.yml:

/etc/cloudflared/config.yml
tunnel: b72cf76b-0820-44cb-90df-f4c58ed87184
credentials-file: /etc/cloudflared/b72cf76b-0820-44cb-90df-f4c58ed87184.json

ingress:
  - hostname: terminal.dustyserver.co.in
    service: http://localhost:7681
  - hostname: fileserver.dustyserver.co.in
    service: http://localhost:8080
  - hostname: flask.dustyserver.co.in
    service: http://localhost:5000
  - hostname: dustyserver.co.in
    service: http://localhost:80
  - hostname: www.dustyserver.co.in
    service: http://localhost:80
  - service: http_status:404
05 — DNS
Step 5 — Create DNS CNAME records
5
Route DNS records to the tunnel
▼

Each hostname in the ingress rules needs a DNS CNAME record pointing to the tunnel. The tunnel route dns command creates these automatically in Cloudflare — no need to touch the Cloudflare dashboard.

bash — create DNS for each hostname
cloudflared tunnel route dns spserver dustyserver.co.in
cloudflared tunnel route dns spserver www.dustyserver.co.in
cloudflared tunnel route dns spserver terminal.dustyserver.co.in
cloudflared tunnel route dns spserver fileserver.dustyserver.co.in
cloudflared tunnel route dns spserver flask.dustyserver.co.in

# Each creates a CNAME record like:
# terminal.dustyserver.co.in → b72cf76b-....cfargotunnel.com

# Verify DNS is set up correctly
nslookup dustyserver.co.in
# Should return Cloudflare IPs: 104.21.x.x and 172.67.x.x
ℹ DNS propagation takes 1-5 minutes after creating records. If a subdomain isn't working immediately — wait and try again. Check with nslookup subdomain.dustyserver.co.in — when it returns Cloudflare IPs, it's ready.
06 — Service
Step 6 — Install as systemd service
6
Test manually then install as service
▼

Always test the tunnel manually first — if it works, you know the config is correct. Then install as a systemd service so it runs automatically on every boot. The service uses the config at /etc/cloudflared/config.yml.

bash
# Test manually first — Ctrl+C to stop after confirming it works
cloudflared tunnel --config /etc/cloudflared/config.yml run

# Then visit https://dustyserver.co.in in browser to confirm
# Ctrl+C to stop the manual test

# Install as systemd service
sudo cloudflared service install

# Enable auto-start on boot
sudo systemctl enable cloudflared

# Start it now
sudo systemctl start cloudflared

# Verify it's running
sudo systemctl status cloudflared
# Look for: Active: active (running)
# Look for: Registered tunnel connection connIndex=0,1,2,3
# 4 connections = tunnel is fully connected and redundant
07 — Pattern
Adding a new service — the repeatable pattern

Every time a new service is added to spserver — a new Docker container, a new app, a new tool — the same three steps are followed. The tunnel itself never changes. Only the config and DNS records are updated.

→ Pattern — add new service to tunnel
bash
# Step 1: Add ingress rule to config.yml
sudo nano /etc/cloudflared/config.yml
# Add before the catch-all line:
#   - hostname: newapp.dustyserver.co.in
#     service: http://localhost:PORT

# Step 2: Create DNS record for the new subdomain
cloudflared tunnel route dns spserver newapp.dustyserver.co.in

# Step 3: Restart cloudflared to pick up config changes
sudo systemctl restart cloudflared

# Step 4: Wait 1-2 minutes for DNS propagation
# Step 5: Test — open https://newapp.dustyserver.co.in
✓ This pattern has been used for every service: dustyserver.co.in (Nginx), terminal (ttyd), fileserver (Filebrowser), flask (Flask Docker). The same three steps every time — no router config, no port forwarding, no firewall changes needed.

Every line explained
LineWhat it does

credentials-file: /etc/cloudflared/...Path to the JSON credentials file. This is the private key — it proves cloudflared is authorised to maintain this tunnel. Must be accessible by root.

03 — Ingress deep dive
Ingress rule options and service types

The service field accepts different formats depending on what you want to forward to:

Service formatUse case
http://localhost:PORTStandard HTTP service on the same server. Most common — used for Nginx, ttyd, Filebrowser, Flask.
https://localhost:PORTHTTPS service running locally. cloudflared connects with TLS to the local service. Use if local service requires HTTPS.
ssh://localhost:22SSH service. Enables SSH via browser using cloudflared client on client machine. Requires extra client setup.
tcp://localhost:PORTRaw TCP tunnelling. Forward any TCP service — databases, custom protocols. Requires cloudflared on client too.
http_status:404Return an HTTP status code directly — no backend service. Used for catch-all.
http_status:503Return 503 Service Unavailable — useful for maintenance mode on a specific subdomain.

Additional ingress options you can add per rule:

config.yml — advanced ingress options
ingress:
  - hostname: flask.dustyserver.co.in
    service: http://localhost:5000
    originRequest:
      connectTimeout: 10s         # timeout connecting to local service
      noTLSVerify: false           # verify TLS cert (for https:// services)
      disableChunkedEncoding: false

01 — What went wrong
Blockers hit during setup
⚠ Blocker 1
sudo cloudflared service install — "No file config.yml found"
After creating the tunnel and config file in ~/.cloudflared/, running sudo cloudflared service install gave the error: "Cannot determine default configuration path. No file [config.yml config.yaml] in [~/.cloudflared /etc/cloudflared ...]". The config file existed — but in the wrong location.
✓ Root cause and fix
~ means /root/ when using sudo — copy config to /etc/cloudflared/
When sudo runs a command, it runs as root. The tilde ~ expands to the home directory of the current user — for root, that is /root/, not /home/sipu/. So sudo cloudflared service install looked in /root/.cloudflared/ — which was empty. Fix: copy both the config.yml and the credentials JSON to /etc/cloudflared/ — a system-wide path accessible by root from any context. Update the credentials-file path in config.yml to reflect the new location.
⚠ Blocker 2
502 Bad Gateway — typo in config.yml service URL
After setting up ttyd, opening terminal.dustyserver.co.in showed "502 Bad Gateway". cloudflared logs showed "Unable to reach the origin service". ttyd was running and responding to curl http://localhost:7681. The tunnel config looked correct at first glance.
✓ Fix
Typo: "locakhost" instead of "localhost"
Careful inspection of config.yml revealed: service: http://locakhost:7681 — "locakhost" instead of "localhost". A single transposed character made the service URL completely wrong. cloudflared was trying to connect to a hostname called "locakhost" which doesn't exist. Fixed the typo, restarted cloudflared, terminal immediately accessible.
⚠ Blocker 3
Third-level subdomain (flask.test.dustyserver.co.in) not working
Initially used flask.test.dustyserver.co.in as the hostname for the Flask app — a third-level subdomain. The DNS record was created but the subdomain was not resolving correctly and the site showed errors.
✓ Fix
Use second-level subdomains only
Cloudflare's free plan has limitations around multi-level wildcard DNS. Third-level subdomains like flask.test.dustyserver.co.in are unreliable on the free tier. The fix was to use a direct second-level subdomain: flask.dustyserver.co.in. Additionally, the ingress rule needed to be moved to the correct position — before the main domain rules.
01 — Issues and fixes
Common Cloudflare Tunnel problems
502 Bad Gateway
The tunnel is reaching the server but cloudflared can't connect to the local service. Diagnostic steps: (1) Is the service running? sudo systemctl status SERVICE. (2) Is it on the right port? sudo ss -tlnp | grep PORT. (3) Test locally: curl http://localhost:PORT — if this fails, the service isn't running or is on a different port. (4) Check config.yml for typos — especially "locakhost" vs "localhost". (5) Restart cloudflared after fixing: sudo systemctl restart cloudflared.
cloudflared service won't start — "config.yml not found"
Config is in the wrong location. Check: ls /etc/cloudflared/ — both config.yml and the credentials JSON should be there. If missing: sudo mkdir -p /etc/cloudflared, copy config and JSON from ~/.cloudflared/. Update the credentials-file path in config.yml to /etc/cloudflared/UUID.json. Then reinstall service: sudo cloudflared service install.
DNS not propagated — subdomain not resolving
After running cloudflared tunnel route dns, wait 1-5 minutes. Test with nslookup subdomain.dustyserver.co.in — when it returns Cloudflare IPs (104.21.x.x or 172.67.x.x), it's ready. Open in incognito window to bypass browser DNS cache. If nslookup still shows nothing after 10 minutes, check Cloudflare dashboard DNS tab to verify the CNAME was created.
Tunnel shows only 1-2 connections instead of 4
cloudflared maintains 4 connections to Cloudflare edge for redundancy. Fewer connections means some failed to establish — usually a temporary issue. Check sudo journalctl -u cloudflared -f for error messages. Usually self-resolves. If persistent: sudo systemctl restart cloudflared. The tunnel still works with fewer connections — it's just less redundant.
Config changes not taking effect after editing config.yml
Must restart cloudflared after every config change — it doesn't hot-reload. Run: sudo systemctl restart cloudflared. Then check status: sudo systemctl status cloudflared. If cloudflared fails to start after a config change, there's likely a YAML syntax error — check with cloudflared tunnel --config /etc/cloudflared/config.yml ingress validate.
Validate ingress rules without restarting
Before restarting cloudflared after editing config, validate the config first: cloudflared tunnel --config /etc/cloudflared/config.yml ingress validate. This checks all ingress rules for syntax errors and confirms the catch-all is present. Saves you from a restart that fails.
How to check which tunnel connection is handling a request
Check cloudflared logs: sudo journalctl -u cloudflared -f. When a request comes in, cloudflared logs the hostname matched and the service it forwarded to. This is the fastest way to confirm routing is working correctly and which ingress rule matched.
