Dockerfile

Writing a Dockerfile to build a custom image for the spserver Flask app. Covers every instruction, layer caching, .dockerignore, and how to go from source code to a running container.

01 — What & Why
What a Dockerfile is — and why it exists

A Dockerfile is a plain text file that contains a series of instructions for building a Docker image. It is a recipe — a repeatable, version-controlled description of exactly what goes into the image: which OS, which runtime, which files, which dependencies, how the app starts.

Without a Dockerfile you would manually set up a server, install dependencies, copy files, and configure the app — every time, on every machine. A Dockerfile makes this process deterministic. The same Dockerfile produces the same image on any machine, every time.

Without a Dockerfile
SSH into the server. Install Python 3.11. Install pip packages. Copy your files. Set environment variables. Write a systemd service. Repeat for staging, repeat for a colleague's machine. Any drift and things break silently.
With a Dockerfile
Write the steps once in a Dockerfile. Run docker build — Docker executes every step reproducibly and packages the result into an image. Run the image anywhere. The environment is always identical.
ℹ A Dockerfile is always named exactly Dockerfile — no extension, capital D. Docker automatically finds it when you run docker build in the same directory. You can use a different name with -f but this is rarely needed.
02 — Core concepts
Image vs Container — the critical distinction

These two terms are used constantly and are often confused at first. The relationship is like a class and an instance in programming — or a recipe and a dish.

ConceptWhat it isAnalogyCommand
Image A read-only, layered snapshot of a filesystem — the built result of a Dockerfile. Stored on disk. Does not run on its own. A recipe / blueprint docker build
Container A running instance of an image. Docker adds a thin writable layer on top of the image and starts the process. Multiple containers can run from the same image simultaneously. A cooked dish / running process docker run
Dockerfile → docker build → Image → docker run → Container One image can spawn many containers: flask-app:latest → container A (port 5000) → container B (port 5001) — e.g. for staging → container C (port 5002) — e.g. for testing
ℹ Images are immutable — a running container cannot change the image it came from. Anything written inside the container (logs, temp files) lives only in that container's writable layer and is lost when the container is removed. Use Docker volumes for persistent data.
03 — Instruction reference
Dockerfile instructions — what each one does

Each line in a Dockerfile is an instruction. Every instruction (except FROM and ARG) creates a new layer in the image.

FROM
Base image
Every Dockerfile must start with FROM. It sets the starting point — the OS, runtime, and tools your image is built on. FROM python:3.11-slim gives you a Debian-based OS with Python 3.11 pre-installed. This determines available tools, image size, and security posture.
WORKDIR
Working directory
Sets the current directory inside the container for all subsequent instructions. If the directory does not exist, Docker creates it. All COPY, RUN, and CMD instructions that follow operate relative to this path. We use WORKDIR /app.
COPY
Copy files into image
Copies files or directories from the build context (the directory you run docker build from) into the image filesystem. COPY requirements.txt . copies requirements.txt into WORKDIR. COPY . . copies everything.
RUN
Execute commands at build time
Runs a shell command during the build and commits the result as a new layer. Used to install packages, run scripts, create directories. Happens once at build time — not when the container starts. RUN pip install -r requirements.txt.
EXPOSE
Document the port
Documents which port the container's application listens on. Does NOT actually publish the port — that is done at docker run -p. It is metadata for humans and tools. EXPOSE 5000 for Flask. Think of it as a label, not a firewall rule.
CMD
Default start command
The command that runs when the container starts. This is runtime — not build time. Only the last CMD in a Dockerfile takes effect. Can be overridden at docker run. Use exec form (JSON array) to avoid shell wrapping issues: CMD ["python", "app.py"].
ENV
Environment variables
Sets environment variables inside the container — available both during build (in subsequent RUN instructions) and at runtime. ENV FLASK_ENV=production. Use for config values that should be baked into the image. Sensitive values (passwords) should be passed at runtime with docker run -e instead.
ARG
Build-time variable
Like ENV but only available during the build — not in the running container. Pass at build time: docker build --build-arg VERSION=1.2. Useful for parameterising builds without baking values into the final image. Does not create a layer.
⚠ RUN vs CMD — the most common confusion. RUN executes during docker build to modify the image (install packages, compile code). CMD executes when you start a container from the image. RUN is build time. CMD is run time.

02 — Write the Dockerfile
Step 1 — Create the Dockerfile for the Flask app
1
Create Dockerfile in the project root
▼

The Dockerfile lives in the project root — the same directory as app.py and requirements.txt. Docker uses this location as the build context — the set of files it can access during the build. Every COPY instruction pulls files relative to this directory.

spserver-flask/Dockerfile
# Base image — Debian slim with Python 3.11 pre-installed
# "slim" omits optional packages → smaller image (~150 MB vs ~1 GB for full)
FROM python:3.11-slim

# Set working directory inside the container
# All subsequent commands run relative to /app
WORKDIR /app

# Copy requirements FIRST — before copying the rest of the code
# This is intentional: requirements change less often than code
# Docker can cache this layer and skip re-installing on code-only changes
COPY requirements.txt .

# Install Python dependencies
# --no-cache-dir: don't store pip's download cache in the image
# Both flags shrink the final image size
RUN pip install --no-cache-dir -r requirements.txt

# Copy the rest of the application code into /app
# Done AFTER pip install — so code changes don't invalidate the pip cache layer
COPY . .

# Document that the Flask app listens on port 5000
# This is metadata — it does not publish the port
EXPOSE 5000

# Start the Flask app when a container is created from this image
# Exec form (JSON array) is preferred — avoids shell signal handling issues
CMD ["python", "app.py"]
ℹ We use python:3.11-slim and not python:3.11 or python:latest. The slim variant omits many optional Debian packages and produces a much smaller image — typically around 150 MB vs 1 GB for the full variant. Always pin the version (3.11) rather than using latest — so builds are reproducible and a future Python release doesn't break your app unexpectedly.
03 — Build
Step 2 — Build the image with docker build
2
Build and tag the image
▼

Running docker build executes every instruction in the Dockerfile in order. The -t flag tags the resulting image with a name so you can refer to it later. The . at the end tells Docker to use the current directory as the build context — the set of files available to COPY instructions.

bash — run from spserver-flask/
# Build the image and tag it as flask-app:latest
# -t flask-app:latest   → name:tag (tag defaults to 'latest' if omitted)
# .                     → build context is the current directory
docker build -t flask-app:latest .

# You will see each step execute:
# [1/5] FROM python:3.11-slim  ← pulls base image (first time only)
# [2/5] WORKDIR /app
# [3/5] COPY requirements.txt .
# [4/5] RUN pip install ...    ← installs packages (takes a moment)
# [5/5] COPY . .

# Confirm the image was created
docker images
# REPOSITORY   TAG       IMAGE ID       CREATED        SIZE
# flask-app    latest    a1b2c3d4e5f6   3 seconds ago  180MB

# Tag the image with a version number (optional but good practice)
docker tag flask-app:latest flask-app:v1.0
docker images
# Both tags now point to the same image ID
04 — Run
Step 3 — Run a container from the image
3
Start a container and test the Flask app
▼

docker run creates and starts a container from the image. The -p flag maps a port on the host to a port in the container — without it, the app is unreachable from outside the container even though EXPOSE is set. -d runs the container in the background so the terminal is free.

bash
# Run container in the background (-d detached mode)
# -p 5000:5000 → host port 5000 maps to container port 5000
# --name flask-app → give the container a name (easier to manage)
docker run -d -p 5000:5000 --name flask-app flask-app:latest

# Confirm the container is running
docker ps
# CONTAINER ID   IMAGE              COMMAND           STATUS
# abc123def456   flask-app:latest   "python app.py"   Up 3 seconds

# Test the app is responding
curl http://localhost:5000
# Or from another machine on the LAN:
# curl http://192.168.31.150:5000

# View container logs
docker logs flask-app

# Follow logs live
docker logs -f flask-app

# Stop and remove the container when done
docker stop flask-app
docker rm flask-app
✓ If curl http://localhost:5000 returns your Flask app's response — the Dockerfile works. The app is running in a container with its own isolated filesystem, and port 5000 is mapped to the host.
05 — Layer caching
How Docker caches layers — and why order matters

Every instruction in a Dockerfile creates a layer — an immutable snapshot of the filesystem at that point. Docker caches each layer. On the next build, if an instruction (and everything above it) has not changed, Docker reuses the cached layer instead of re-running it. This makes rebuilds dramatically faster.

The key rule: once a layer is invalidated, all layers below it are also rebuilt. Cache invalidation cascades downward. This is why instruction order is critical.

1
FROM python:3.11-slim
CACHED
2
WORKDIR /app
CACHED
3
COPY requirements.txt .
CACHED — requirements unchanged
4
RUN pip install -r requirements.txt
CACHED — skipped pip install
5
COPY . . ← only app.py changed
REBUILT
6
CMD ["python", "app.py"]
REBUILT

This is why we COPY requirements.txt . and run pip install before COPY . . — requirements change far less often than application code. If you did it the other way (copy all files first, then pip install), every code change would invalidate the pip layer and force a full re-install of all packages.

Wrong order — slow rebuilds
COPY . . ← copies all files
RUN pip install ... ← runs after every code change

Every time app.py changes, Docker sees the COPY layer invalidated, and re-runs pip install from scratch. Dependencies haven't changed but Docker doesn't know that.
Correct order — fast rebuilds
COPY requirements.txt . ← only the dep list
RUN pip install ... ← cached until requirements change
COPY . . ← code goes last

Code changes only invalidate the final COPY layer. Pip install stays cached. Rebuild takes seconds instead of minutes.
ℹ General rule: put instructions that change rarely at the top, and instructions that change often at the bottom. The cache is invalidated from the first changed line downward.
06 — .dockerignore
.dockerignore — keep the build context clean

When you run docker build . Docker sends the entire current directory (the build context) to the Docker daemon before it processes any instructions. If your project contains a large virtual environment (venv/), a .git/ history, or compiled __pycache__ files — all of that gets sent and potentially copied into the image.

A .dockerignore file (placed in the same directory as the Dockerfile) lists files and directories to exclude from the build context. It works exactly like .gitignore.

4
Create .dockerignore for the Flask project
▼
spserver-flask/.dockerignore
# Python byte-code cache — auto-generated, not needed in image
__pycache__/
*.pyc
*.pyo
*.pyd

# Virtual environment — has platform-specific binaries, pip installs inside container instead
venv/
.venv/
env/

# Git history — not needed in the image, just adds size
.git/
.gitignore

# Development editor files
.vscode/
.idea/
*.swp

# Test files — not needed in production image
tests/
*.test.py

# Documentation
*.md
docs/

# Environment files with secrets — never copy into image
.env
.env.*
⚠ The most important exclusion is venv/. A Python virtual environment contains platform-specific compiled binaries. Copying it into a Linux container image is not only wasteful (100s of MB) — it is also likely to break because the binaries may be compiled for a different platform. Always install dependencies fresh with pip install inside the container.
Without .dockerignore
Build context includes venv/ (300 MB), .git/ (50 MB), __pycache__ (10 MB). These are sent to the daemon on every build, COPY . . copies them into the image, image size bloats to 600 MB+ and build is slow.
With .dockerignore
Build context contains only app.py, requirements.txt, templates/, static/. Build context transfer is nearly instant. Image only contains what the app needs. Clean 180 MB image instead of 600 MB.
01 — Build commands
docker build — building images
docker build -t flask-app:latest .
Build image from Dockerfile in current directory, tag as flask-app:latest
docker build -t flask-app:v1.0 .
Build with a specific version tag instead of latest
docker build --no-cache -t flask-app:latest .
Build without using cached layers — forces all steps to re-run. Useful when cache seems stale

Use a differently named Dockerfile with the -f flag

Show full build output including all RUN command output — helpful for debugging
docker tag flask-app:latest flask-app:v1.0
Add a new tag to an existing image without rebuilding
02 — Image commands
Managing images
docker images
List all images on disk — shows repository, tag, image ID, created date, size
docker images flask-app
List only images with the name flask-app — shows all tags
docker inspect flask-app:latest
Show full metadata for an image — layers, environment variables, exposed ports, entrypoint
docker history flask-app:latest
Show the layer history of an image — each layer's instruction, size, and creation time
docker rmi flask-app:latest
Delete an image — fails if a container (even stopped) is using it
docker rmi flask-app:latest --force
Force delete image even if containers reference it (containers lose their image reference)
docker image prune
Remove all dangling images — untagged intermediate layers left over from builds

01 — What went wrong
Blockers hit while writing the Dockerfile
⚠ Blocker 1
COPY failed — no such file or directory
Build failed at the COPY requirements.txt . step with: ERROR: failed to solve: failed to read dockerfile: open /tmp/buildkit-.../requirements.txt: no such file or directory. The file existed on disk but Docker couldn't find it.
✓ Fix
COPY paths are relative to the build context, not the Dockerfile location
COPY paths must be relative to the build context directory — the directory you pass to docker build (the . at the end). In this case, the build was being run from a parent directory while the Dockerfile was in a subdirectory: docker build -f spserver-flask/Dockerfile . — so requirements.txt was not at the context root. Fix: always run docker build from inside the project directory: cd spserver-flask && docker build -t flask-app:latest .
⚠ Blocker 2
Container exited immediately — CMD vs RUN confusion
The container started and immediately exited. docker ps showed nothing. docker ps -a showed the container with status Exited (2) 1 second ago. The Dockerfile had: RUN python app.py instead of CMD ["python", "app.py"].
✓ Fix
Use CMD for the start command — RUN is build time only
RUN python app.py during the build tries to start the Flask server during image construction — which fails because there is no network, no ports mapped, no proper environment. Even if it ran, the command would block the build forever. The correct instruction is CMD ["python", "app.py"] — this tells Docker to run the command when a container is started, not when the image is built.
⚠ Blocker 3
Image size was 600 MB — venv copied into image
After building, docker images showed the flask-app image at 623 MB. Expected around 180 MB. The local venv/ directory (containing platform-specific compiled packages) was being copied into the image by COPY . . because there was no .dockerignore.
✓ Fix
Add .dockerignore with venv/ excluded
Created .dockerignore in the project root with venv/ listed. Docker then excluded the virtual environment from the build context entirely. Rebuilt the image: size dropped to 178 MB. The packages are installed fresh inside the container via pip install -r requirements.txt — which is exactly the right approach, since the pip cache is not included (--no-cache-dir) and the packages are compiled for the container's OS.
01 — Issues and fixes
Common Dockerfile problems and fixes
COPY fails — no such file or directory
The path given to COPY doesn't exist relative to the build context. Check: (1) Are you running docker build . from the right directory? (2) Does the file exist in that directory — ls to confirm. (3) Is the file excluded by .dockerignore? Check cat .dockerignore. (4) Check for typos in the filename — COPY is case-sensitive on Linux. Rule: COPY paths are always relative to the build context directory, not to the Dockerfile location.
Container exits immediately after docker run
The CMD process exits or crashes — Docker stops the container when the main process exits. Check logs: docker logs <container-name>. Common causes: (1) Wrong command in CMD — the file doesn't exist or has a path issue. (2) RUN used instead of CMD — app tried to start at build time. (3) App crashes on startup — check for missing env vars, missing files, import errors. Try: docker run --rm -it flask-app:latest /bin/bash to inspect the container filesystem interactively.
Port not reachable — connection refused on localhost:5000
The container is running but curl localhost:5000 fails. Check: (1) Was -p 5000:5000 included in the docker run command? EXPOSE alone does not publish ports. (2) Is the Flask app binding to 0.0.0.0 and not 127.0.0.1? If Flask binds to localhost inside the container, it's unreachable from outside. Fix: app.run(host='0.0.0.0', port=5000) in app.py.
pip install is slow on every rebuild — cache not working
Layer caching is being invalidated before the pip install step. Most likely cause: COPY . . appears before RUN pip install in the Dockerfile. Fix: reorder so COPY requirements.txt . and RUN pip install come before COPY . .. Also check: if you use --no-cache flag in the build command, all caching is disabled.
Image is much larger than expected
Inspect the layers: docker history flask-app:latest — look for unusually large layers. Common causes: (1) venv/ copied in — add to .dockerignore. (2) .git/ copied in — add to .dockerignore. (3) apt package cache not cleaned — add && rm -rf /var/lib/apt/lists/* after apt-get install. (4) Using the full base image instead of slim — switch from python:3.11 to python:3.11-slim.
How to open a shell inside a running container
To inspect the container filesystem or debug from inside: docker exec -it flask-app /bin/bash (if bash is available) or docker exec -it flask-app /bin/sh (always available). For a one-off inspection without starting the main app: docker run --rm -it flask-app:latest /bin/bash. Useful for checking if files were copied correctly, if packages installed, or what environment variables are set.
01 — What's next
Future scope
⟶ Things to explore from here
Multi-stage builds — use multiple FROM instructions in a single Dockerfile to separate the build environment from the runtime environment. Stage 1 installs build tools and compiles the app. Stage 2 is a minimal runtime image that only copies the compiled output. Result: dramatically smaller production images with no build toolchain included.
Gunicorn instead of Flask dev server — CMD ["python", "app.py"] uses Flask's built-in development server which is not production-ready (single-threaded, no auto-restart). Replace with CMD ["gunicorn", "--bind", "0.0.0.0:5000", "app:app"] for a production WSGI server. Add gunicorn to requirements.txt.
HEALTHCHECK instruction — add HEALTHCHECK CMD curl -f http://localhost:5000/health || exit 1 to the Dockerfile. Docker polls this periodically and marks containers as healthy or unhealthy. Docker Compose and Kubernetes use this to decide when a container is ready to receive traffic.
Non-root user inside the container — by default, the process inside the container runs as root. Add RUN adduser --disabled-password appuser && USER appuser before CMD. Running as a non-root user limits damage if the app is compromised — a container escape gives attacker a non-privileged user instead of root.
Docker Compose — next topic — the Dockerfile builds a single image. Docker Compose orchestrates multiple containers together (Flask app + database + reverse proxy) from a single YAML file. The build: key in docker-compose.yml points to the Dockerfile — Compose builds and manages the image automatically.
Pushing to Docker Hub — once the image is working on spserver, it can be pushed to Docker Hub for sharing or backup: docker tag flask-app:latest username/flask-app:latest && docker push username/flask-app:latest. Then pull and run on any other machine with a single docker run command.
