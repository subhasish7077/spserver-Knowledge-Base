CG-NAT & Tailscale

Understanding why Airtel makes port forwarding impossible — and how Tailscale bypasses it entirely to give SSH access from anywhere in the world.

01 — Concept
What is NAT

NAT stands for Network Address Translation. It solves a fundamental problem — IPv4 addresses are limited to about 4 billion, but the world has far more than 4 billion internet-connected devices. NAT lets multiple devices share a single public IP address.

Your home router performs NAT. Every device on your network (PC, phone, spserver) has a private IP (192.168.31.x). When any of them make an outgoing request, the router translates the private IP to the single public IP it was given by Airtel, sends the request, then when the response arrives, translates it back and sends it to the right device.

Your PC (192.168.31.7) ─── wants to visit google.com ───► │ Router (192.168.31.1) translates source IP ──────────► NAT table: 192.168.31.7 → 106.215.x.x 192.168.31.7 ↔ port 54321 192.168.31.150 ↔ port 54322 │ Google sees request from: 106.215.x.x (router's public IP) Google responds to: 106.215.x.x │ Router receives response ──► looks up NAT table ──► sends to 192.168.31.7

This works perfectly for outgoing connections. The problem arises with incoming connections. If someone on the internet wants to connect to spserver directly, they only know the public IP (106.215.x.x). When the packet arrives at the router, the router has no entry in its NAT table for this unsolicited connection — so it doesn't know which device to forward it to. Port forwarding is the solution to this — you tell the router "any incoming packet on port 22 goes to 192.168.31.150". But this only works if you actually control the router that holds the public IP.

02 — The problem
What is CG-NAT and why Airtel uses it

CG-NAT stands for Carrier Grade NAT. It is NAT performed by the ISP itself — before traffic even reaches your router. Airtel runs a massive NAT layer on their network infrastructure, sharing a single real public IP among potentially hundreds of customers.

The result: your router's WAN IP (the IP it sees on its internet-facing side) is not a real public IP. It is an address in the 100.64.0.0/10 range — a range specifically reserved by IANA (Internet Assigned Numbers Authority) for carrier-grade NAT. This is Airtel's internal network address space, not the public internet.

Normal NAT (what you expect): Internet ──► 106.215.x.x (YOUR router's public IP) ──► 192.168.31.150 (spserver) You control this router ↑ You control this ↑ Port forwarding works here CG-NAT (what Airtel actually gives you): Internet ──► 106.215.x.x (AIRTEL's servers) ──► 100.220.x.x (your router's WAN) ──► 192.168.31.150 Airtel controls this ↑ You control this ↑ Port forwarding impossible here Port forwarding set here (useless)
✗ When your router's WAN IP starts with 100.x.x.x, you have CG-NAT. No amount of port forwarding configuration on your router will fix this — you do not own the first NAT layer. The traffic never reaches your router from the internet.

Why do ISPs use CG-NAT? IPv4 addresses are a finite resource and they ran out years ago. Rather than forcing everyone to IPv6 (which would solve this), ISPs like Airtel use CG-NAT to stretch their IPv4 address pool by sharing addresses across customers. It reduces costs. The downside is that customers cannot host services or accept incoming connections.

03 — Why it fails
Why port forwarding is impossible with CG-NAT

Port forwarding works by telling your router: "when a packet arrives on port 22, forward it to 192.168.31.150". This requires the packet to reach your router first. With CG-NAT, the packet arrives at Airtel's servers, not your router. Airtel's NAT layer has no idea which of their hundreds of customers the packet is meant for — so it drops it.

Someone tries to SSH into spserver from outside: Attacker/User PC ──► sends packet to ──► 106.215.x.x:22 ↓ Arrives at Airtel's CG-NAT servers Airtel has NO mapping for this Packet dropped silently ↓ Your backup router (192.168.31.1) ← never reached Port 22 → 192.168.31.150 ✓ ← this rule is set but useless ↓ spserver (192.168.31.150) ← never reached sshd listening on port 22 ✓ ← SSH is ready but unreachable
04 — Investigation
How we confirmed CG-NAT on Airtel

After SSH timed out from a mobile hotspot even with port forwarding configured, we investigated the actual network topology. Two checks confirmed CG-NAT:

bash — run on spserver
# Check 1: What IP does the internet see?
curl ifconfig.me
# Returned: 106.215.x.x  ← This is Airtel's real public IP

# Check 2: What IP does the router show as its WAN IP?
# (Checked in router admin panel at 192.168.31.1)
# Router WAN IP: 100.220.x.x  ← 100.x.x.x = CG-NAT range confirmed

# These two IPs being different AND the WAN being 100.x.x.x
# = definitive CG-NAT confirmation
curl ifconfig.me → 106.215.x.x
This is what the internet sees when spserver makes outgoing connections. It is Airtel's shared public IP — used by many customers simultaneously.
Router WAN → 100.220.x.x
The 100.64.0.0/10 range (100.64.x.x to 100.127.x.x) is IANA-reserved exclusively for CG-NAT. Seeing this on your router WAN is definitive confirmation.
05 — Discovery
Double NAT — the network we actually have

During the investigation we also discovered the home network has two routers stacked — meaning there are actually three NAT layers between spserver and the internet. This was discovered after port forwarding on the backup router did nothing.

Internet ↕ Airtel CG-NAT ← Layer 1 (Airtel's infrastructure — not your hardware) ↕ Airtel Modem 192.168.1.1 ← Layer 2 (your hardware — modem/router combo) ↕ Backup Router 192.168.31.1 ← Layer 3 (your hardware — second router) ↕ ↕ ↕ spserver 192.168.31.150 Your PC 192.168.31.7 Phone, etc. Port forwarding on backup router only → traffic still blocked at Airtel modem Port forwarding on both routers → traffic still blocked at Airtel CG-NAT No configuration possible → Airtel CG-NAT is outside your control
06 — Solutions
All solutions to CG-NAT — compared
Tailscale chosen — SSH
Free VPN. Both devices reach OUT to Tailscale servers — outgoing always works behind CG-NAT. Direct encrypted P2P tunnel for SSH. Zero config on routers. 100 devices free.
Cloudflare Tunnel chosen — web
Free. Server reaches OUT to Cloudflare edge. Public website/services accessible at your domain. No port forwarding needed. Handles SSL. See Cloudflare Tunnel guide.
ZeroTier free
Similar to Tailscale. Creates a virtual network between devices. Free up to 25 devices. Open source — can self-host the controller. Slightly more complex setup.
WireGuard VPS paid
Rent a cheap VPS (~₹200-300/month) with a real public IP. Set up WireGuard tunnel from spserver to VPS. VPS forwards traffic to spserver. More control, more complexity.
Airtel Static IP paid
Call Airtel and pay for a dedicated static public IP. Removes CG-NAT entirely. Port forwarding works again. Costs extra monthly. Simplest solution if you want traditional setup.
IPv6 free
IPv6 has enough addresses for every device to have a real public IP — no NAT needed. Airtel provides IPv6 in some areas. If available, devices get real public IPv6 addresses and are directly reachable.
✓ We use Tailscale for SSH (private access) and Cloudflare Tunnel for public services (website, ttyd, Filebrowser). Both are free, both bypass CG-NAT, and together they cover all use cases without paying Airtel extra.
01 — How it works
Tailscale's approach to CG-NAT

The key insight behind Tailscale: outgoing connections always work behind CG-NAT. CG-NAT only blocks unsolicited incoming connections. So instead of waiting for connections to come in, Tailscale has both devices reach OUT simultaneously to its coordination servers. Those servers broker a direct connection between the two devices.

Step 1: Both devices connect OUT to Tailscale coordination servers spserver ──── outgoing ────► Tailscale coordination ◄──── outgoing ──── Your PC CG-NAT allows this ↑ (login.tailscale.com) CG-NAT allows this ↑ Step 2: Tailscale brokers direct connection using DERP/STUN spserver ◄════════════ direct encrypted WireGuard tunnel ════════════► Your PC Each device gets a stable Tailscale IP (100.x.x.x) P2P when possible, relayed through DERP servers otherwise

Tailscale is built on WireGuard — a modern, fast, lightweight VPN protocol built into the Linux kernel. Tailscale takes WireGuard and adds automatic key exchange, NAT traversal, and a coordination layer so you never have to manage keys or configure peers manually.

WireGuard (underneath)
The actual VPN protocol. Extremely fast — ~2x faster than OpenVPN. Built into Linux kernel since 5.6. Tailscale handles all WireGuard configuration automatically.
Tailscale (on top)
Handles key distribution, NAT traversal, peer discovery. You login once — devices find each other automatically. Free for up to 100 devices with unlimited bandwidth.
02 — Install
Install Tailscale on spserver
1
Install Tailscale on spserver
▼

Tailscale provides a one-line install script that detects your Linux distribution and installs the correct package. It sets up the Tailscale service to auto-start on boot.

bash — run on spserver
# One-line install script from Tailscale
curl -fsSL https://tailscale.com/install.sh | sh

# Verify installation
tailscale version
2
Connect spserver to your Tailscale account
▼

Running tailscale up connects the device to Tailscale and gives you a URL to authenticate in the browser. Login with your Tailscale account (Google, GitHub, or email). After authenticating, spserver joins your Tailscale network.

bash — run on spserver
# Connect to Tailscale — opens a URL in output
sudo tailscale up

# Copy the URL shown, open it in your browser
# Login with your Tailscale account
# Click "Connect" — spserver is now in your network

# Get spserver's Tailscale IP
tailscale ip
# Returns something like: 100.78.211.x
# This IP is stable — it never changes for this device
3
Install Tailscale on your Windows PC
▼

Both devices must be on the same Tailscale account to talk to each other. Download and install the Windows app, login with the same account, and both devices appear in your Tailscale admin panel.

Windows
# Download Tailscale for Windows from:
# https://tailscale.com/download/windows

# Install → login with same account as spserver
# Both devices now appear in Tailscale admin at:
# https://login.tailscale.com/admin/machines
4
Ensure Tailscale starts on boot
▼

If spserver reboots and Tailscale doesn't start, you lose the ability to SSH in remotely. Verify it is enabled as a systemd service.

bash — on spserver
# Check Tailscale service status
sudo systemctl status tailscaled

# Enable if not already (usually done automatically by install script)
sudo systemctl enable tailscaled

# Verify Tailscale is connected after reboot
tailscale status

04 — Emergency backup
Tailscale as the emergency recovery tool

Tailscale serves a critical secondary purpose — it is the emergency backup access method if everything else breaks. If cloudflared goes down, if Nginx fails, if you misconfigure UFW and lock yourself out via normal SSH — Tailscale is how you get back in.

Normal operation: Public users ──► Cloudflare Tunnel → spserver Your SSH ──► 192.168.31.150 (local) or Tailscale (remote) Emergency — something broke: Cloudflare down? → SSH in via Tailscale → fix cloudflared UFW locked SSH? → SSH in via Tailscale → fix UFW rules Nginx broken? → SSH in via Tailscale → fix Nginx config Can't reach 150? → SSH in via Tailscale → check network config Tailscale works independently of all other services on spserver As long as spserver has power and internet — Tailscale works
⚠ Never remove or disable Tailscale from spserver. Even if you add other remote access methods — keep Tailscale running as the last-resort recovery tool. It has saved the session more than once.

01 — What went wrong
The full investigation story
⚠ Blocker 1
SSH timed out from mobile hotspot — port forwarding set but not working
SSH worked perfectly on the local network (192.168.31.150). Tried from a mobile hotspot using the public IP returned by curl ifconfig.me. Got "Connection timed out". Port 22 forwarding had been configured on the backup router (192.168.31.1). Connection still timed out.
→ Investigation step 1
Discovered Double NAT — port forwarding on wrong router
Realised the backup router (192.168.31.1) gets its internet from the Airtel modem (192.168.1.1). Traffic from the internet hits the Airtel modem first. Modem had no port forwarding rule — dropped all traffic before it reached the backup router. Configured port forwarding on both routers. Still timed out.
⚠ Blocker 2
Port forwarding on both routers — still timed out
Port 22 forwarding configured on Airtel modem (192.168.1.1) AND backup router (192.168.31.1). SSH still timed out from outside. Checked router WAN IP in the modem admin panel — showed 100.220.x.x.
→ Root cause found
CG-NAT confirmed — Airtel never gave a real public IP
The 100.220.x.x WAN IP confirmed CG-NAT. This is Airtel's internal IP — not a real public IP. The traffic never leaves Airtel's network to reach the modem. Port forwarding on our hardware is irrelevant because Airtel's CG-NAT is the first barrier. No configuration on our equipment can fix this — it is Airtel's infrastructure.
⚠ Blocker 3
Port forwarding rules removed — now lost remote access entirely
After confirming CG-NAT makes port forwarding useless, all port forwarding rules were removed from both routers. Before Tailscale was set up, this briefly meant there was no way to SSH into spserver from outside the home network at all.
✓ Fix
Tailscale installed — SSH from anywhere immediately
Installed Tailscale on spserver with curl -fsSL https://tailscale.com/install.sh | sh and sudo tailscale up. Installed on Windows PC. Both devices appeared in Tailscale admin. Immediately SSHed from mobile hotspot using the Tailscale IP. Worked instantly — no router configuration, no port forwarding, no ISP cooperation needed.
01 — Issues and fixes
Common Tailscale and CG-NAT issues
Cannot SSH via Tailscale IP — connection refused or timed out
Check in order: (1) Is Tailscale running on spserver? sudo systemctl status tailscaled — if not active, start it: sudo systemctl start tailscaled. (2) Is spserver online in Tailscale? Run tailscale status on your PC — spserver should show as connected. (3) Is sshd running on spserver? It needs to be — Tailscale just provides the network path. (4) Is UFW allowing port 22? Check: sudo ufw status. Note: Tailscale traffic on the local Tailscale interface (tailscale0) is usually not subject to UFW rules, but verify if UFW logging shows blocks.
Tailscale shows device as offline / disconnected
spserver may have rebooted and Tailscale didn't reconnect. Solutions: (1) If you can reach spserver locally — run sudo tailscale up. (2) Restart the daemon: sudo systemctl restart tailscaled. (3) Check internet connectivity on spserver: ping 8.8.8.8. Tailscale needs outgoing internet to reach its coordination servers. (4) Check if auth expired — keys expire after 180 days by default. Re-authenticate with sudo tailscale up --reset.
Tailscale connection is slow or high latency
Tailscale uses direct P2P connection when possible but falls back to DERP relay servers if NAT traversal fails. Run tailscale netcheck to see connection type. "Direct" means P2P (fast). "Relay" means going through a DERP server (adds latency). With CG-NAT, sometimes only relay is possible. This is expected — usually still fast enough for SSH.
How do I check if my ISP uses CG-NAT?
Run curl ifconfig.me on your server — note the public IP. Then check your router admin panel for the WAN IP. If they are different AND the router WAN shows an IP in the 100.64.0.0/10 range (100.64.x.x to 100.127.x.x), you have CG-NAT. Also: try to ping your public IP from your mobile hotspot — if it doesn't respond or times out, likely CG-NAT.
Tailscale works but I want a custom domain for SSH (ssh.dustyserver.co.in)
This requires Cloudflare Tunnel SSH configuration. Add an ingress rule to config.yml pointing to ssh://localhost:22 and create a DNS record for ssh.dustyserver.co.in. Clients need cloudflared installed locally and a ProxyCommand in their SSH config. More complex than Tailscale — only useful when Tailscale isn't an option on the client device.
tailscale up asks to re-authenticate every time after reboot
This means the auth key expired or the session isn't persisting. Fix: in Tailscale admin (login.tailscale.com/admin/machines), find spserver and click "Disable key expiry" — this makes the auth persistent indefinitely. Alternatively, use an auth key: sudo tailscale up --authkey=tskey-... (generate from admin panel → Settings → Keys).
01 — What's next
Future scope
⟶ Things to explore from here
WireGuard directly — Tailscale uses WireGuard under the hood. Learning raw WireGuard teaches you how VPNs actually work at the protocol level. Highly relevant for DevOps and network security roles at companies like Zscaler.
Headscale — self-hosted Tailscale coordination server. Open source reimplementation of Tailscale's backend. You run your own server so no data goes through Tailscale's infrastructure. Requires a VPS with a real public IP. Good Phase 4+ project.
Tailscale ACLs — access control lists define which devices can talk to which. In the Tailscale admin panel, write ACL rules to restrict what your phone can access vs what your PC can access on spserver. Relevant for multi-user setups.
Tailscale SSH — Tailscale has a built-in SSH feature that handles authentication via Tailscale identity — no sshd configuration needed, no SSH keys to manage. Enable with sudo tailscale up --ssh. Experimental but interesting.
ZeroTier as alternative — similar concept to Tailscale, open source, self-hostable controller. Free for up to 25 devices. Worth knowing as an alternative when Tailscale doesn't fit a use case.
IPv6 investigation — check if Airtel provides IPv6 on your connection. If they do, devices can have real public IPv6 addresses — no NAT, no CG-NAT, direct connections possible. Check with curl -6 ifconfig.me.
VPS jump host — rent a cheap VPS (Hetzner ~₹300/month). Set up WireGuard tunnel from spserver to VPS. Use VPS as a jump host for SSH and public services. More control than Tailscale/Cloudflare, more cost, better for learning.
