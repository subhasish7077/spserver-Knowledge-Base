Filebrowser

A self-hosted web file manager — browse, upload, download, rename and delete server files from any browser anywhere in the world. fileserver.dustyserver.co.in from your phone, no apps required.

01 — Concept
What is Filebrowser

Filebrowser is a lightweight, self-hosted web file manager. It runs as a tiny binary on spserver, listens on port 8080, and serves a clean web interface where you can browse folders, upload and download files, rename items, create directories, and delete files — all from any browser, from anywhere.

Think of it as a self-hosted Google Drive for your server — except it is just a single binary, uses about 10MB of RAM, has zero external dependencies, and stores all its configuration in a single SQLite database file. It requires no Docker, no runtime, no Node or Python — just the binary.

📁
Browse files
Navigate the server filesystem from any browser. Folder tree, file listing, file sizes, modification dates.
⬆️
Upload files
Drag and drop files from your PC to the server. Supports multiple files, shows progress. Works from mobile too.
⬇️
Download files
Download any file or folder (as zip). Pull logs, configs, or project files directly to your local machine.
✏️
Edit text files
Built-in text editor for config files, scripts, HTML. Syntax highlighting for common file types. Edit without SSH.
🔍
Search files
Search by filename across the exposed directory. Quick way to find files without SSH and grep.
👤
Multiple users
Create multiple users with different scopes and permissions. Each user sees only their allowed directories.
ℹ Filebrowser is login-protected. All users must authenticate with a username and password before accessing anything. The admin user is created during setup.
02 — Use case
What we use Filebrowser for on spserver

The root directory for Filebrowser is set to /home/sipu/shared — the same shared folder exposed via Samba. This means the same files are accessible two ways:

At home — use Samba
Full speed LAN access. Drag-drop from Windows File Explorer. Edit files in-place. Native drive letter. Fast — no internet roundtrip.
Away from home — use Filebrowser
Works from any browser anywhere. Upload files from phone. Download a file you need. No apps, no VPN clients, no SSH. Just open fileserver.dustyserver.co.in.

Filebrowser is also useful for quick file edits via the built-in editor — fixing a config file, editing an HTML guide page, tweaking a script — without needing to SSH in and use nano.

01 — Install
Step 1 — Download and install the binary
1
Install Filebrowser binary
▼

Filebrowser is distributed as a single self-contained binary — no package manager, no runtime, no dependencies. The official install script detects your architecture (amd64 in our case) and installs the binary to /usr/local/bin/filebrowser.

bash — on spserver
# Official install script — detects architecture automatically
curl -fsSL https://raw.githubusercontent.com/filebrowser/get/master/get.sh | bash

# Confirm binary installed correctly
filebrowser version
# Should print: Filebrowser v2.x.x

# Confirm binary location
which filebrowser
# Should print: /usr/local/bin/filebrowser
02 — Initialise
Step 2 — Create database and configure root directory
2
Initialise the SQLite database
▼

Filebrowser stores all its configuration — users, settings, permissions — in a single SQLite database file. We keep this in /home/sipu/filebrowser/ to keep it separate from the shared files it manages.

bash
# Create directory for the database
mkdir -p /home/sipu/filebrowser

# Initialise a new Filebrowser database
filebrowser config init \
  --database /home/sipu/filebrowser/filebrowser.db

# Set the root directory — what files Filebrowser exposes
filebrowser config set \
  --root /home/sipu/shared \
  --database /home/sipu/filebrowser/filebrowser.db

# Verify the config was saved
filebrowser config cat \
  --database /home/sipu/filebrowser/filebrowser.db
ℹ The --root path is the top-level directory Filebrowser can access. Users cannot browse above this directory — it acts as a filesystem jail. We set it to /home/sipu/shared so only that folder is exposed, not the entire home directory.
03 — User
Step 3 — Create the admin user
3
Add admin user to the database
▼

Filebrowser requires at least one user to login. The --perm.admin flag grants full admin rights — can create other users, change settings, access all files. This is your main login for the web interface.

bash
# Create admin user
# Replace 'yourpassword' with a strong password
filebrowser users add sipu yourpassword \
  --perm.admin \
  --database /home/sipu/filebrowser/filebrowser.db

# Verify user was created
filebrowser users ls \
  --database /home/sipu/filebrowser/filebrowser.db
# Should list: ID | Username | Scope | Admin
#              1  | sipu     | .     | true
⚠ The default Filebrowser username/password is admin/admin — change it immediately or create your own user as shown above. The web interface is publicly accessible via Cloudflare Tunnel and brute force bots will find it.
04 — Service
Step 4 — Create the systemd service
4
Write and enable the service file
▼

A systemd service ensures Filebrowser starts automatically on boot and restarts on crash. The critical flag here is --address 0.0.0.0 — without it, Filebrowser binds to 127.0.0.1 (localhost only) and becomes completely unreachable from the network. This was the main blocker during setup.

bash
sudo nano /etc/systemd/system/filebrowser.service

Paste this content:

/etc/systemd/system/filebrowser.service
[Unit]
Description=Filebrowser web file manager
After=network.target

[Service]
ExecStart=/usr/local/bin/filebrowser \
  --database /home/sipu/filebrowser/filebrowser.db \
  --port 8080 \
  --address 0.0.0.0
Restart=always
RestartSec=5
User=sipu

[Install]
WantedBy=multi-user.target
bash — enable and start
sudo systemctl daemon-reload
sudo systemctl enable filebrowser
sudo systemctl start filebrowser
sudo systemctl status filebrowser
# Look for: Active: active (running)

# Confirm it is listening on 0.0.0.0:8080
sudo ss -tlnp | grep 8080
# Should show: 0.0.0.0:8080 ← critical, not 127.0.0.1:8080
05 — Firewall
Step 5 — Open port 8080 in UFW
5
Allow port 8080
▼

UFW blocks all incoming by default. Without this rule, Filebrowser is running and bound to 0.0.0.0:8080 — but UFW drops all packets before they reach it. Opening the port allows local network access on port 8080 and also allows cloudflared to forward traffic from the tunnel to localhost:8080.

bash
sudo ufw allow 8080/tcp
sudo ufw status
# Should show: 8080/tcp   ALLOW   Anywhere

# Quick local test — from Windows PC browser
# Open: http://192.168.31.150:8080
# Should see the Filebrowser login page
06 — Public access
Step 6 — Expose via Cloudflare Tunnel
6
Add ingress rule and DNS record
▼

Without this step, Filebrowser is only reachable on the local network. Adding it to the Cloudflare Tunnel makes it accessible at fileserver.dustyserver.co.in from anywhere in the world with HTTPS and DDoS protection.

bash
# Step 1: Add ingress rule to Cloudflare Tunnel config
sudo nano /etc/cloudflared/config.yml
# Add this block BEFORE the catch-all line:
#   - hostname: fileserver.dustyserver.co.in
#     service: http://localhost:8080

# Step 2: Create the DNS CNAME record
cloudflared tunnel route dns spserver fileserver.dustyserver.co.in

# Step 3: Restart cloudflared
sudo systemctl restart cloudflared

# Step 4: Wait 1-2 minutes for DNS propagation
# Step 5: Open https://fileserver.dustyserver.co.in
# Login with sipu / yourpassword
✓ Once live, fileserver.dustyserver.co.in works from your phone on mobile data, your work PC, a friend's laptop — any browser anywhere. Login with your Filebrowser credentials (set in step 3).
01 — Service file
systemd service file — every line explained
/etc/systemd/system/filebrowser.service
[Unit]
Description=Filebrowser web file manager
After=network.target

[Service]
ExecStart=/usr/local/bin/filebrowser \
  --database /home/sipu/filebrowser/filebrowser.db \
  --port 8080 \
  --address 0.0.0.0
Restart=always
RestartSec=5
User=sipu

[Install]
WantedBy=multi-user.target
LineWhat it does
After=network.targetWait for networking to be available before starting — ensures Filebrowser can bind to port 8080 on startup.
ExecStart=...Full path to binary — required for systemd services. The service won't find binaries in PATH the same way your shell does.
--database /home/sipu/filebrowser/filebrowser.dbPoints to the SQLite database containing all users, settings, and permissions. Must exist before starting.
--port 8080Port to listen on. Port 8080 was chosen to avoid conflict with Nginx on port 80. Must match the UFW rule and Cloudflare Tunnel ingress config.
--address 0.0.0.0Critical. Bind to all network interfaces. Without this, Filebrowser binds to 127.0.0.1 (localhost) and is only reachable from the server itself — not from the network or Cloudflare Tunnel.
Restart=alwaysIf Filebrowser crashes or is killed, systemd automatically restarts it.
RestartSec=5Wait 5 seconds before restarting — prevents rapid crash loops.
User=sipuRun as sipu (not root). Filebrowser only needs access to /home/sipu/shared — no root privileges required. Principle of least privilege.
02 — CLI flags
All useful Filebrowser flags
FlagEffect
--database PATHPath to the SQLite database file. Required — all config and users live here.
--port PORTTCP port to listen on. Default is 8080.
--address ADDRESSCritical. IP address to bind to. Use 0.0.0.0 to accept connections from any interface. Default is 127.0.0.1 (localhost only).
--root PATHRoot directory to expose. Users cannot browse above this path. Set during init with filebrowser config set --root.
--log PATHWrite logs to a file instead of stdout. Useful for debugging — --log /var/log/filebrowser.log.
--baseurl PATHServe Filebrowser at a sub-path — e.g. --baseurl /files makes it available at http://host:8080/files. Useful if behind a reverse proxy serving multiple apps on one domain.
--cert PATH --key PATHServe directly over HTTPS using your own certificate. Not needed since Cloudflare handles TLS — use with Flexible or Full Strict SSL modes if needed.

01 — What went wrong
Blockers hit during Filebrowser setup
⚠ Blocker 1 — The big one
Filebrowser running but completely unreachable — browser shows nothing
After installing and starting Filebrowser, sudo systemctl status filebrowser showed it was running. But opening http://192.168.31.150:8080 in the Windows PC browser showed "This site can't be reached" — ERR_CONNECTION_REFUSED. The service was running but accepting no connections from the network.
✓ Root cause and fix
Missing --address 0.0.0.0 — Filebrowser was binding to 127.0.0.1
The default Filebrowser bind address is 127.0.0.1 (localhost only). Without explicitly setting --address 0.0.0.0, Filebrowser only accepts connections from the server itself — not from the network or Cloudflare Tunnel. Running sudo ss -tlnp | grep 8080 confirmed it: 127.0.0.1:8080 — not 0.0.0.0:8080. Fix: add --address 0.0.0.0 to ExecStart in the service file, reload daemon, restart service. Browser immediately connected.
⚠ Blocker 2
Port 8080 accessible locally but not from internet (Cloudflare 502)
After fixing the address issue, Filebrowser was accessible on the local network at http://192.168.31.150:8080. But opening https://fileserver.dustyserver.co.in showed 502 Bad Gateway. Cloudflare Tunnel was running and cloudflared logs showed connection attempts.
✓ Fix
UFW was blocking port 8080
Even though Cloudflare Tunnel connects internally on localhost:8080 (so technically doesn't go through UFW's external rules), the UFW rule was still needed to allow cloudflared's connection. Running sudo ufw allow 8080/tcp immediately resolved the 502 error. Local network access worked because connections from 192.168.31.x are treated as local, but cloudflared's connection was being blocked. Lesson: always open the UFW port for every service, even if it's behind a tunnel.
⚠ Blocker 3
Login works but files are not visible — empty directory listing
After login, the file browser showed an empty directory even though /home/sipu/shared had files in it. The root directory was showing as correctly set to /home/sipu/shared in the Filebrowser settings.
✓ Fix
Service running as wrong user — permission denied reading shared folder
The service file initially had User=root. The /home/sipu/shared directory is owned by sipu with permissions 755 — root can read it, but the files inside had permissions that sipu created (644). The issue was actually subtler: the database was initialised with the root user's context, but the files were owned by sipu. Fix: change service to User=sipu, re-initialise the database as sipu (not sudo), reload and restart. Files appeared correctly.
01 — Issues and fixes
Common Filebrowser problems
Cannot reach Filebrowser — connection refused or timeout
Check in order: (1) Is service running? sudo systemctl status filebrowser. (2) What address is it binding to? sudo ss -tlnp | grep 8080 — must show 0.0.0.0:8080 not 127.0.0.1:8080. If 127.0.0.1 — add --address 0.0.0.0 to ExecStart. (3) Is UFW allowing port 8080? sudo ufw status. (4) Test locally first: curl http://localhost:8080.
fileserver.dustyserver.co.in shows 502 Bad Gateway
Cloudflare Tunnel can't reach Filebrowser. Check: (1) Is Filebrowser running? (2) Is it on port 8080 and bound to 0.0.0.0? (3) Is UFW allowing 8080? (4) Check cloudflared config.yml — is the ingress rule for fileserver.dustyserver.co.in pointing to http://localhost:8080? (5) Check for typos — "localhost" not "locakhost". Restart cloudflared after fixing.
Login fails — "wrong credentials"
The default credentials (admin/admin) may not work if you initialised the database with a different user. Check which users exist: filebrowser users ls --database /home/sipu/filebrowser/filebrowser.db. If the user doesn't exist — add it: filebrowser users add sipu password --perm.admin --database PATH. If you've forgotten the password — delete the user and re-add: filebrowser users rm sipu --database PATH.
Files show up but can't upload — permission denied
Filebrowser is running as a user that doesn't have write permission to the root directory. Check: ls -la /home/sipu/shared — confirm sipu owns it. Check service file User= setting — should be User=sipu. Also check directory permissions: chmod 755 /home/sipu/shared and chown sipu:sipu /home/sipu/shared.
Filebrowser starts then immediately stops
Database file is missing or corrupted. Check: sudo journalctl -u filebrowser --since "5 minutes ago" — look for error messages. Common cause: database path in ExecStart doesn't match where you created it. Verify: ls -la /home/sipu/filebrowser/ — filebrowser.db should exist and be owned by sipu.
How to change the root directory after setup
Stop the service, run: filebrowser config set --root /new/path --database /home/sipu/filebrowser/filebrowser.db, then restart the service. The change takes effect immediately. Note that existing user scopes (if any) may need updating too — in the Filebrowser web UI under Settings → Users.
01 — What's next
Future scope
⟶ Things to explore from here
Multiple users with scoped access — create separate users for different people or use cases. Each user can have a different root scope — e.g. a "guest" user that only sees /home/sipu/shared/public. Set in the Filebrowser web UI under Settings → Users → Add User → Scope.
Cloudflare Access in front of Filebrowser — add email OTP or Google SSO before the Filebrowser login page. Anyone trying to access fileserver.dustyserver.co.in would need to verify their email first — adds a layer before even seeing the login form. Zero Trust → Access → Applications in Cloudflare dashboard.
Expand root directory — change root from /home/sipu/shared to /home/sipu to expose more folders. Or add symlinks inside shared/ that point to other directories — Filebrowser follows symlinks, giving access to other parts of the filesystem through a controlled path.
Nextcloud as upgrade path — Nextcloud is a full Google Drive replacement with sync clients, mobile apps, calendar, contacts, and more. It is much heavier (~500MB RAM) but the natural evolution if file management needs grow. Can run alongside Filebrowser in Docker.
Filebrowser in Docker — run Filebrowser as a Docker container instead of a bare binary. Easier to update (pull new image tag), easier to configure (docker-compose.yml), and fits into the Phase 2 containerisation approach. Official image: filebrowser/filebrowser.
Immich for photos — if storing and managing photos on spserver, Immich is a self-hosted Google Photos alternative. Runs in Docker, has mobile apps for automatic upload, face recognition, album organisation. Good companion service alongside Filebrowser.
