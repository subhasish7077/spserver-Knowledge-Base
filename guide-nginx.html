Nginx Web Server

Serving the website at dustyserver.co.in and acting as the reverse proxy that routes traffic to multiple apps running on the same server — all on different subdomains.

01 — Concept
What is Nginx

Nginx (pronounced "engine-x") is a high-performance web server and reverse proxy. It listens on port 80 (HTTP) and port 443 (HTTPS), receives incoming requests, and either serves files directly or forwards the request to another service. It is the most widely deployed web server in the world, used by Netflix, Airbnb, GitHub, and millions of other sites.

On spserver, Nginx serves two roles simultaneously — it is the web server that hosts the dustyserver.co.in website, and it is the reverse proxy that will route flask.dustyserver.co.in to the Flask Docker container, blog.dustyserver.co.in to a future blog, and so on — all on the same server, all on port 80.

ℹ Nginx uses an event-driven, asynchronous architecture. Unlike older servers that create one thread per connection, Nginx handles thousands of concurrent connections in a single worker process. This makes it extremely efficient even on low-spec hardware like spserver.
02 — Two roles
Web server vs reverse proxy — both at once
Web server role
Nginx reads the request URL, finds the matching file in /var/www/dustyserver/, and sends it back to the visitor. HTML, CSS, images, JavaScript — all served directly. No application code involved.
Reverse proxy role
Nginx reads the incoming hostname, matches it against rules, and forwards the request to another process — Flask on port 5000, a Node app on port 3000, anything. It acts as the middleman between the internet and backend apps.
Visitor requests dustyserver.co.in/index.html Nginx finds /var/www/dustyserver/index.html Nginx serves file directly → visitor sees website Visitor requests flask.dustyserver.co.in/ Nginx reads Host header: flask.dustyserver.co.in Nginx finds server block with server_name flask.dustyserver.co.in Nginx forwards request to http://localhost:5000 Flask processes request → response goes back through Nginx → visitor
03 — Internals
How Nginx processes a request

When a request arrives at Nginx, it goes through a specific decision process to find which server block and location block to use:

HTTP request arrives on port 80 ↓ Nginx checks all server blocks for matching server_name → matches "dustyserver.co.in" ──► serves from /var/www/dustyserver → matches "flask.dustyserver.co.in" ──► proxies to localhost:5000 → no match ──► uses default server (first listed) ↓ Within matched server block, Nginx checks location blocks → location / ──► matches everything (catch-all) → location /api/ ──► only matches URLs starting with /api/ ↓ Request is handled according to the matched location block

The server block selects which virtual host handles the request (based on the domain name). The location block within that server block selects how to handle different URL paths. This two-level routing is what allows Nginx to host multiple sites and route different paths differently — all on one machine.

04 — File structure
Nginx directory structure on Ubuntu
/etc/nginx/ ← Nginx config root ├── nginx.conf ← main config (rarely edited) ├── sites-available/ ← all site configs live here │ ├── default ← default page (we deleted this) │ └── dustyserver ← our site config ├── sites-enabled/ ← symlinks to active configs │ └── dustyserver ← symlink to sites-available/dustyserver ├── conf.d/ ← additional configs (auto-loaded) └── modules-enabled/ ← loaded Nginx modules /var/www/ ← website files └── dustyserver/ ← our website root └── index.html ← homepage /var/log/nginx/ ← log files ├── access.log ← every request logged here └── error.log ← errors and warnings
ℹ The sites-available / sites-enabled pattern is Ubuntu's way of managing Nginx sites. You write config in sites-available/ and create a symlink in sites-enabled/ to activate it. This lets you disable a site by removing the symlink without deleting the config.

01 — Install
Install Nginx and open firewall
1
Install Nginx
▼

The nginx package from Ubuntu's repositories installs a stable version with all features needed for our setup. It also creates the directory structure at /etc/nginx/ and /var/www/html/, and registers itself as a systemd service.

bash
sudo apt install nginx -y

# Enable auto-start on boot
sudo systemctl enable nginx

# Start now
sudo systemctl start nginx

# Check it is running
sudo systemctl status nginx
# Look for: Active: active (running)
2
Open HTTP and HTTPS in UFW
▼

UFW blocks all incoming by default. The Nginx Full app profile opens both port 80 (HTTP) and 443 (HTTPS) in one command. UFW knows about this profile because Nginx registers it automatically during install.

bash
# Allow both HTTP (80) and HTTPS (443)
sudo ufw allow 'Nginx Full'

# Verify
sudo ufw status
# Should show: Nginx Full   ALLOW   Anywhere
02 — Configure
Create the site config file
3
Write the site config in sites-available
▼

Each website or app gets its own config file in /etc/nginx/sites-available/. The filename can be anything — we use the domain name for clarity. This file is not active yet — it becomes active when you symlink it into sites-enabled/.

bash
sudo nano /etc/nginx/sites-available/dustyserver

Paste this config:

/etc/nginx/sites-available/dustyserver
server {
    listen 80;
    server_name dustyserver.co.in www.dustyserver.co.in;

    root /var/www/dustyserver;
    index index.html;

    location / {
        try_files $uri $uri/ =404;
    }
}
03 — Enable
Enable the site and remove the default
4
Create symlink and remove default page
▼

The symlink in sites-enabled/ is what activates the config. Nginx only reads configs from sites-enabled/ — sites-available/ is just storage. We also remove the default Nginx page which would otherwise intercept requests.

bash
# Create symlink to enable the site
sudo ln -s /etc/nginx/sites-available/dustyserver /etc/nginx/sites-enabled/

# Remove the default Nginx welcome page
sudo rm /etc/nginx/sites-enabled/default

# Test config syntax BEFORE reloading — always do this
sudo nginx -t
# Expected: nginx: configuration file /etc/nginx/nginx.conf syntax is ok
#           nginx: configuration file /etc/nginx/nginx.conf test is successful

# Reload Nginx to apply changes (no downtime)
sudo systemctl reload nginx
⚠ Always run sudo nginx -t before reloading. If there is a syntax error in your config, nginx -t shows you exactly where. If you reload with a broken config, Nginx fails to reload and may stop serving requests.

01 — Static site
Static website config — complete and explained
/etc/nginx/sites-available/dustyserver
server {
    listen 80;
    server_name dustyserver.co.in www.dustyserver.co.in;

    root /var/www/dustyserver;
    index index.html index.htm;

    location / {
        try_files $uri $uri/ =404;
    }

    # Cache static assets
    location ~* \.(css|js|jpg|jpeg|png|gif|ico|svg)$ {
        expires 30d;
        add_header Cache-Control "public, immutable";
    }

    # Security headers
    add_header X-Frame-Options "SAMEORIGIN";
    add_header X-Content-Type-Options "nosniff";
}
DirectiveWhat it does
listen 80Accept HTTP connections on port 80. Cloudflare Tunnel sends plain HTTP to Nginx — Cloudflare handles HTTPS on its edge.
server_name dustyserver.co.in www.dustyserver.co.inThis server block handles requests with these Host headers. Both the apex domain and www subdomain go to the same block.
root /var/www/dustyserverThe filesystem path where website files live. Nginx looks here for requested files.
index index.html index.htmWhen a directory is requested (e.g. /), try these filenames as the default file.
location / { ... }Match all URL paths. The / pattern is the catch-all — matches any URL.
try_files $uri $uri/ =404Try to serve: (1) the exact file requested, (2) as a directory with index file, (3) return 404 if neither found.
location ~* \.(css|js|...)$~* means case-insensitive regex match. Match any URL ending in these file extensions.
expires 30dTell browsers to cache these files for 30 days — reduces repeat requests and speeds up the site.
02 — Reverse proxy
Reverse proxy config — forwarding to a backend app

When acting as a reverse proxy for a Flask/Django/Node app, the server block uses proxy_pass instead of root. The proxy_set_header directives pass important metadata to the backend app.

/etc/nginx/sites-available/dustyserver — add a proxy server block
server {
    listen 80;
    server_name flask.dustyserver.co.in;

    location / {
        proxy_pass         http://localhost:5000;
        proxy_http_version 1.1;

        # Pass original request info to backend
        proxy_set_header Host              $host;
        proxy_set_header X-Real-IP         $remote_addr;
        proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # WebSocket support (needed for real-time apps)
        proxy_set_header Upgrade           $http_upgrade;
        proxy_set_header Connection         "upgrade";

        # Timeouts
        proxy_connect_timeout 60s;
        proxy_send_timeout    60s;
        proxy_read_timeout    60s;
    }
}
DirectiveWhat it does
proxy_pass http://localhost:5000Forward the request to Flask running on port 5000. Can also point to another server: proxy_pass http://10.0.0.2:5000.
proxy_http_version 1.1Use HTTP/1.1 for the connection to the backend. Required for WebSocket and keep-alive connections.
proxy_set_header Host $hostPass the original hostname to Flask so it knows which domain was requested. Without this, Flask sees "localhost".
X-Real-IP $remote_addrPass the real visitor IP to Flask. Without this, Flask logs Nginx's IP (127.0.0.1) for every request.
X-Forwarded-ForStandard header for proxy chains. Appends the real IP to any existing forwarded-for chain.
X-Forwarded-Proto $schemeTell Flask whether the original request was HTTP or HTTPS. Flask uses this to generate correct URLs.
Upgrade / ConnectionRequired for WebSocket connections (e.g. ttyd, real-time apps). Passes the upgrade request through the proxy.
03 — Multiple subdomains
Hosting multiple sites on one server

Each subdomain gets its own server block. They can all be in the same config file or in separate files — Nginx loads everything from sites-enabled/. The pattern is identical each time: listen 80, set server_name, then either serve files or proxy_pass.

/etc/nginx/sites-available/dustyserver — complete multi-site config
# Main website
server {
    listen 80;
    server_name dustyserver.co.in www.dustyserver.co.in;
    root /var/www/dustyserver;
    index index.html;
    location / { try_files $uri $uri/ =404; }
}

# Flask app
server {
    listen 80;
    server_name flask.dustyserver.co.in;
    location / {
        proxy_pass http://localhost:5000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}

# Knowledge base docs
server {
    listen 80;
    server_name docs.dustyserver.co.in;
    root /var/www/dustyserver;  # same dir — serves index.html + guides
    index index.html;
    location / { try_files $uri $uri/ =404; }
}

# Future Django app
server {
    listen 80;
    server_name django.dustyserver.co.in;
    location / {
        proxy_pass http://localhost:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
ℹ After editing the config always run sudo nginx -t to validate, then sudo systemctl reload nginx to apply. Reload is zero-downtime — Nginx keeps serving existing connections while workers reload with the new config.

02 — Logs
Nginx log commands
sudo tail -f /var/log/nginx/access.log
Watch incoming requests live — shows IP, method, URL, status code, response size.
sudo tail -f /var/log/nginx/error.log
Watch errors live — first place to check when something isn't working.
sudo tail -100 /var/log/nginx/error.log
Show last 100 lines of error log — good starting point for debugging.
sudo grep "404" /var/log/nginx/access.log
Filter access log for 404 errors — find missing files or broken links.

Find bad gateway errors — usually means the backend app is down.
sudo cat /var/log/nginx/access.log | awk '{print $1}' | sort | uniq -c | sort -rn | head

01 — What went wrong
Blockers hit during Nginx setup
⚠ Blocker 1
Default Nginx page showing instead of our website
After creating the site config and reloading Nginx, opening the browser showed the default "Welcome to nginx!" page instead of our website. The config file was correct but the default site was taking priority.
✓ Fix
Remove the default symlink from sites-enabled
The default Nginx config at /etc/nginx/sites-enabled/default was still active and had a catch-all server block. Since it was listed first, it intercepted all requests. Fix: sudo rm /etc/nginx/sites-enabled/default then sudo nginx -t and sudo systemctl reload nginx. Our site then displayed correctly.
⚠ Blocker 2
nginx -t fails after editing config — permission denied on /var/www/dustyserver
After creating the website directory with mkdir as sipu, sudo nginx -t passed but the site returned 403 Forbidden. The directory existed but Nginx couldn't read the files.
✓ Fix
Set correct permissions on web root directory
Nginx runs as the www-data user. The /var/www/dustyserver/ directory and its files need to be readable by www-data. Fix: sudo chown -R www-data:www-data /var/www/dustyserver and sudo chmod -R 755 /var/www/dustyserver. Alternatively, ensure the directory has execute permission for other users so www-data can enter it.
01 — Issues and fixes
Common Nginx problems
502 Bad Gateway
Nginx is running but the backend app (Flask, etc.) is not responding. Check: (1) Is the app running? sudo systemctl status APP or docker compose ps. (2) Is it on the right port? sudo ss -tlnp | grep PORT. (3) Test from server: curl http://localhost:PORT. (4) Check Nginx error log: sudo tail -20 /var/log/nginx/error.log.
403 Forbidden
Nginx cannot read the requested file or directory. Causes: (1) File permissions wrong — the www-data user can't read the file. Fix: sudo chmod 644 /var/www/dustyserver/index.html. (2) Directory permissions wrong — www-data can't enter the directory. Fix: sudo chmod 755 /var/www/dustyserver. (3) SELinux/AppArmor blocking — less common on Ubuntu.
nginx -t fails — syntax error
A config file has a syntax error. nginx -t shows the file name and line number. Common mistakes: missing semicolons at end of directives, mismatched braces { }, typos in directive names, wrong quotation marks. Fix the error, run nginx -t again, then reload.
Config changes not taking effect after reload
The site config in sites-available/ was edited but the symlink in sites-enabled/ points to the file correctly — changes should take effect on reload. If not: (1) Verify there is no other config overriding this one. (2) Check if nginx.conf includes another directory that has conflicting rules. (3) Try sudo nginx -T to see the full parsed config.
Nginx shows default page instead of my site
The default Nginx config is still active and taking priority. Check: ls /etc/nginx/sites-enabled/ — if default is listed, remove it: sudo rm /etc/nginx/sites-enabled/default. Then sudo nginx -t and reload.
Nginx starts fine but port 80 is already in use
Another process is on port 80. Find it: sudo ss -tlnp | grep :80 — the last column shows the process name. Either stop that process or change Nginx to use a different port. Common culprit: Apache may be installed and running alongside Nginx.
Slow response — Nginx taking too long to respond
If the backend app is slow (Flask, Django), Nginx will wait — the default timeout is 60 seconds. Tune with: proxy_read_timeout 30s in the location block. For static files, slowness usually means the file path is wrong and Nginx is searching. Check root directive matches actual directory.
01 — What's next
Future scope
⟶ Things to explore from here
Nginx as Docker reverse proxy — run Nginx inside a Docker container, alongside your app containers on the same Docker network. Containers talk to each other by name. This is the setup in the Nginx reverse proxy guide (Phase 2).
Gzip compression — add to nginx.conf: gzip on; gzip_types text/plain text/css application/json application/javascript;. Reduces response sizes by 60-80% for text content. Faster page loads with zero code changes.
Rate limiting — protect your apps from abuse: limit_req_zone $binary_remote_addr zone=one:10m rate=10r/s; and limit_req zone=one burst=20;. Limits clients to 10 requests per second with a burst of 20.
Load balancing — distribute traffic across multiple backend instances with the upstream block. Run two Flask workers, Nginx distributes requests between them. Relevant when a single instance can't handle the load.
Nginx caching — cache responses from backend apps in memory or on disk. A database query that takes 100ms on first request returns in microseconds from cache. Configure with proxy_cache_path and proxy_cache.
Traefik as replacement — in Phase 2+ when Docker is the primary deployment method, Traefik auto-discovers containers and configures routing from Docker labels. Less manual config than Nginx. Worth evaluating once Docker is comfortable.
Nginx Unit — Nginx's application server. Runs Python/Node/PHP apps directly without needing Gunicorn or uWSGI. Simpler stack for Flask/Django deployments.
