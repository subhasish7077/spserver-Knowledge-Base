Static IP with Netplan

Giving spserver a permanent, fixed IP address so SSH, Samba, and Cloudflare Tunnel always know where to find it — regardless of reboots.

01 — The problem
Why a static IP is non-negotiable for a server

By default, your router assigns IP addresses automatically using DHCP (Dynamic Host Configuration Protocol). Every device that connects — your PC, phone, server — gets an IP from a pool. The problem is that IP can change. Today spserver is 192.168.31.150, after a reboot it might be 192.168.31.92.

For a desktop or phone this is fine — you don't type IP addresses to use them. But for a server, the IP address is how everything finds it. SSH connections, Samba drive mappings, Cloudflare Tunnel configuration, and any other service that points to the server by IP — all break the moment the IP changes.

Without static IP — DHCP assigns random IPs: Monday: spserver → 192.168.31.150 ← SSH works Tuesday: spserver → 192.168.31.92 ← SSH broken, Samba broken ssh sipu@192.168.31.150 ← connection refused, wrong IP With static IP — always the same: Every day: spserver → 192.168.31.150 ← always works
⚠ A static IP is one of the first things configured on any server — home lab or production. Everything else in this knowledge base depends on spserver always being at 192.168.31.150.
02 — The tool
What is Netplan

Netplan is Ubuntu's network configuration abstraction layer. Instead of configuring the network backend directly, you write a simple YAML file describing what you want — static IP, gateway, DNS — and Netplan translates that into the correct configuration for the backend it uses.

On Ubuntu Server, the backend is systemd-networkd. On Ubuntu Desktop it is NetworkManager. Netplan sits above both and you write the same YAML regardless of what's underneath.

You write: /etc/netplan/01-netcfg.yaml (simple YAML) ↓ Netplan reads and validates the YAML ↓ systemd-networkd (the actual backend that manages the interface) ↓ ethusb0 gets IP 192.168.31.150 — permanently
ℹ Netplan has been the default network configuration tool on Ubuntu since version 18.04. The older /etc/network/interfaces method still works but is deprecated and not recommended on modern Ubuntu.
03 — Concept
Understanding subnet masks — what /24 means

When we write 192.168.31.150/24, the /24 is a subnet mask in CIDR (Classless Inter-Domain Routing) notation. It tells the operating system which part of the IP address identifies the network and which part identifies the device.

/24 means the first 24 bits (the first three numbers) define the network. So all devices with 192.168.31.x are on the same network — they can talk to each other directly without going through a router or the internet.

NotationNetwork rangeMax devicesTypical use
/24 ← our setup 192.168.31.0 – 192.168.31.255 254 Home and small office networks
/16 192.168.0.0 – 192.168.255.255 65,534 Larger corporate networks
/8 10.0.0.0 – 10.255.255.255 16 million Large enterprise, ISP internal
/32 Single IP address only 1 Host-specific routes, firewall rules

In our Netplan config, 192.168.31.150/24 tells the OS — my IP is 150, I'm on the 192.168.31.x network, and anything in that range is reachable directly. Everything else goes through the gateway (192.168.31.1, the router).

04 — Concept
What are nameservers / DNS

DNS (Domain Name System) translates human-readable domain names like google.com into IP addresses that computers actually use. Without DNS configured on your server, you could ping 8.8.8.8 (an IP) but not ping google.com (a domain name).

In the Netplan config we specify 8.8.8.8 (Google's primary DNS) and 8.8.4.4 (Google's secondary). Every time spserver needs to look up a domain — for apt update, curl, cloudflared — it asks these servers.

8.8.8.8 / 8.8.4.4
Google Public DNS. Fast, reliable, globally available. Default choice for most servers. Zero configuration needed beyond adding to Netplan.
1.1.1.1 / 1.0.0.1
Cloudflare DNS. Fastest DNS resolver in the world according to independent benchmarks. Privacy-focused. Valid alternative to Google DNS.
05 — Architecture
Full network diagram
Internet ↕ Airtel CG-NAT (100.220.x.x WAN) ← blocks all incoming ↕ Airtel Modem (192.168.1.1) ↕ Backup Router (192.168.31.1) ← our default gateway ↕ ↕ spserver Your PC 192.168.31.150 192.168.31.7 ethusb0 (USB eth) (DHCP — changes sometimes) static — never changes

The router at 192.168.31.1 is the default gateway — all traffic that isn't destined for the local 192.168.31.x range gets sent there first. The router then decides where it goes.

06 — Alternatives
Other ways to configure a static IP
MethodStatusNotes
Netplan YAML chosen Used Modern Ubuntu standard. Simple YAML, handles everything via systemd-networkd.
/etc/network/interfaces Deprecated Old Debian/Ubuntu method. Still works but deprecated on Ubuntu 18.04+. Mixing with Netplan causes conflicts.
NetworkManager (nmcli) Desktop tool Designed for desktops and laptops switching between WiFi networks. Overkill for a server. Ubuntu Server uses systemd-networkd instead.
DHCP reservation on router Valid alternative Configure the router to always give spserver the same IP based on its MAC address. No server-side config needed. Depends on the router — if you change routers, the reservation is lost.
ip addr add (manual) Temporary only Manually assign an IP via ip command. Does not persist across reboots. Only useful for temporary testing.
01 — Before Netplan
Step 1 — Rename the interface with udev

The USB ethernet adapter had a long auto-generated name like enx00e04c680bc2 based on its MAC address. This needed to be renamed to ethusb0 before writing the Netplan config — because the interface name in the YAML must match the actual interface name exactly.

See the dedicated udev guide for the full rename process. After the rename and reboot, ip link show shows ethusb0 — and that is the name used in Netplan.

ℹ If your interface name is different from ethusb0, replace it everywhere in the Netplan YAML. The name must exactly match what ip link show shows.
02 — Write the config
Step 2 — Create the Netplan YAML file
1
Open the config file
▼

Netplan reads all .yaml files from /etc/netplan/ alphabetically. The file is usually named with a number prefix (01-) to control ordering. The name after the prefix can be anything — we used netcfg to indicate it is the network config.

bash
# Open the Netplan config file for editing
sudo nano /etc/netplan/01-netcfg.yaml

# If the file already exists, check what's there first
cat /etc/netplan/01-netcfg.yaml

# List all files in netplan directory
ls -la /etc/netplan/
2
Write the YAML config
▼

YAML is space-sensitive. Every indent must be exactly 2 spaces — never tabs, never 3 or 4 spaces. A single wrong indent will silently break networking and lock you out of SSH. Always double-check indentation before applying.

/etc/netplan/01-netcfg.yaml
network:
  version: 2
  renderer: networkd
  ethernets:
    ethusb0:
      dhcp4: false
      dhcp6: false
      optional: false
      addresses:
        - 192.168.31.150/24
      routes:
        - to: default
          via: 192.168.31.1
      nameservers:
        addresses:
          - 8.8.8.8
          - 8.8.4.4
⚠ Use Ctrl+X → Y → Enter to save and exit nano. After saving, do not apply yet — test first with sudo netplan try.
03 — Apply safely
Step 3 — Test and apply the config
3
Test the config first (safe)
▼

netplan try applies the config temporarily. If anything goes wrong — SSH drops, you lose network — it automatically reverts after 120 seconds. This is the safe way to test. Only use netplan apply after confirming try worked.

bash
# Test config — auto-reverts in 120 seconds if you don't confirm
sudo netplan try

# If you see "Do you want to keep these settings?" — network is working
# Press Enter to confirm and keep the changes

# If SSH drops after running this — wait 120 seconds and reconnect
# Netplan will have reverted to old settings automatically
4
Apply permanently
▼

Once netplan try confirms everything works, apply permanently. This writes the config to systemd-networkd and persists across reboots.

bash
sudo netplan apply
04 — Verify
Step 4 — Confirm it worked
bash — run these after applying
# Check the IP was assigned correctly
ip addr show ethusb0
# Look for: inet 192.168.31.150/24

# Check the default route is correct
ip route show
# Look for: default via 192.168.31.1 dev ethusb0

# Test router is reachable
ping -c 3 192.168.31.1

# Test internet connectivity (IP, no DNS)
ping -c 3 8.8.8.8

# Test DNS resolution works
ping -c 3 google.com

# View full interface status from systemd-networkd
networkctl status ethusb0
✓ If all three pings succeed — router reachable, internet reachable, DNS working — the static IP is correctly configured and will persist across every reboot.

02 — Line by line
Every line explained
LineWhat it does
network:Root key — everything must be nested under this.
version: 2Netplan config format version. Always 2 on modern Ubuntu. Do not change.
renderer: networkdUse systemd-networkd as the backend. Correct for Ubuntu Server. Desktop uses NetworkManager instead.
ethernets:Config section for ethernet interfaces. WiFi would go under wifis: instead.
ethusb0:The interface name. Must exactly match what ip link show returns. Renamed from the long MAC-based name via udev.
dhcp4: falseDisable IPv4 auto-assignment (DHCP). We are setting our own fixed address instead.
dhcp6: falseDisable IPv6 auto-assignment. Not needed for our setup.
optional: falseThe system waits for this interface to come up before finishing boot. Critical — without this, SSH may not be available immediately after reboot.
addresses:List of IP addresses to assign to this interface.
- 192.168.31.150/24Our fixed IP. The /24 means this is on the 192.168.31.x network. Devices sharing this prefix can communicate directly.
routes:Network routing rules — how traffic should flow.
to: defaultThis rule applies to all traffic not destined for the local network.
via: 192.168.31.1The default gateway — send all non-local traffic to the backup router at this address.
nameservers:DNS configuration — how to resolve domain names to IP addresses.
- 8.8.8.8Google's primary DNS server. First choice for name resolution.
- 8.8.4.4Google's secondary DNS. Used as fallback if 8.8.8.8 is unreachable.

01 — What went wrong
Blockers hit during setup
⚠ Blocker 1
dhclient command not found
After setting up the interface, tried to request an IP manually using dhclient ethusb0. Got "command not found". Tried sudo apt install dhclient — package doesn't exist.
✓ Fix
Use netplan apply instead
Ubuntu 22.04+ removed the old isc-dhcp-client package (which provided dhclient) entirely. It has been replaced by systemd-networkd, which Netplan manages. The correct command is sudo netplan apply — it handles both DHCP and static IP assignment natively. No separate dhclient needed or available.
⚠ Blocker 2
Interface name in YAML didn't match actual name
Initially wrote the YAML with a guessed interface name before confirming the actual name with ip link show. The mismatch meant the config applied but had no effect on the actual interface.
✓ Fix
Always confirm interface name first
Run ip link show first, find the actual interface name, then write the YAML. After the udev rename (separate guide), the name became ethusb0 — which is what the YAML uses. The udev rename must be done and a reboot completed before writing the Netplan config.
01 — Issues and fixes
Common problems
Lost SSH access after applying Netplan
A YAML indentation error caused networking to fail. You can no longer reach the server over SSH. Options: (1) Use Tailscale if it was already installed — connect via Tailscale IP. (2) Physically connect a keyboard and monitor. Once on the server: check the YAML with cat /etc/netplan/01-netcfg.yaml, fix any indentation errors, run sudo netplan try. Prevention: always use netplan try before netplan apply.
YAML parse error — invalid syntax
Running sudo netplan apply returns a YAML error like "mapping values are not allowed here". Cause: tabs used instead of spaces, or wrong indentation level. Fix: open the file with sudo nano /etc/netplan/01-netcfg.yaml, check every indent is exactly 2 spaces, look for any tab characters (shown as wider gaps in nano). Use the copy button in this guide to get a clean version.
IP assigned but cannot ping router
The IP was set but the gateway is wrong. Check ip route show — the default route should show via 192.168.31.1. If it shows a different gateway, update the via line in the YAML and reapply. Also check the ethernet cable is physically connected and the interface is UP: ip link show ethusb0 should show state UP.
Internet works but DNS is broken — can ping 8.8.8.8 but not google.com
The nameservers in the YAML are not being applied. Check if /etc/resolv.conf contains the right DNS servers. Run cat /etc/resolv.conf — should show nameserver 8.8.8.8. If not, the nameservers section of the YAML may have incorrect indentation. Fix indentation and reapply.
Server gets correct IP but changes after reboot
The Netplan config file was saved but netplan apply was never run, so systemd-networkd didn't receive the new config. Or there are multiple Netplan files in /etc/netplan/ and one overrides the other. Check: ls /etc/netplan/. Remove any old files, ensure only 01-netcfg.yaml exists, run sudo netplan apply, then reboot to confirm.
networkctl shows "degraded" or "no-carrier"
"no-carrier" means the ethernet cable is not connected or the switch/router port is not active. Check the physical cable. "degraded" usually means one interface is up but another has an issue — check which one with networkctl status. If the USB adapter is the issue, unplug and replug it.
01 — What's next
Future scope
⟶ Things to explore from here
DHCP reservation on router — as a backup, configure the router to always assign 192.168.31.150 to spserver based on its MAC address. Belt and suspenders approach.
IPv6 configuration — add IPv6 support to the Netplan config. Many ISPs provide real public IPv6 addresses even with CG-NAT. This could eliminate the need for Tailscale and Cloudflare Tunnel for some use cases.
Multiple network interfaces — configure both the USB ethernet adapter and the built-in WiFi (if available) as a failover. If the ethernet drops, the server stays connected via WiFi.
Network bonding — combine two ethernet adapters into one logical interface for redundancy or increased bandwidth. More relevant with multiple physical adapters.
Cloudflare DNS (1.1.1.1) — switch nameservers from Google DNS (8.8.8.8) to Cloudflare DNS (1.1.1.1) for faster resolution and better privacy. Minor change — update the nameservers section and reapply.
VLAN configuration — segment network traffic using VLANs in Netplan. Useful if you add more hardware and want to isolate the server traffic from general home network traffic.
