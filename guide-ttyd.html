ttyd Browser Terminal

A full interactive Linux terminal running inside any browser — no SSH client, no Tailscale app, no setup on the client device. Open terminal.dustyserver.co.in from your phone and you're in.

01 — Concept
What is ttyd

ttyd is a small, open-source tool that runs a terminal emulator inside a web browser. It starts a WebSocket server on a port (7681 in our case), and when you open that URL in a browser, you get a fully functional terminal — exactly like opening a terminal on the server itself. You can run any command, edit files, manage services, tail logs — everything.

The name ttyd stands for tty daemon. A tty (teletype) is the Linux abstraction for a terminal — every terminal session in Linux is a tty. ttyd creates a tty and streams it over WebSocket to the browser using the xterm.js terminal emulator library.

ℹ ttyd has zero dependencies after installation — it is a single binary compiled in C. It uses about 1-2MB of RAM when idle. Even with multiple browser sessions connected, resource usage stays negligible.
02 — Architecture
How ttyd works under the hood
Browser opens: https://terminal.dustyserver.co.in ↓ Cloudflare Tunnel → routes to → localhost:7681 (ttyd) ↓ ttyd serves xterm.js ← JavaScript terminal emulator in browser ↓ WebSocket connection established between browser and ttyd ↓ ttyd spawns: /bin/login (a real Linux login process) ↓ Login prompt appears in browser You type: sipu / your password ↓ Full bash session — authenticated Linux terminal in browser Every keypress → WebSocket → ttyd → tty → shell → output back

The WebSocket connection is bidirectional — keystrokes go in, terminal output comes back. Because ttyd uses xterm.js, you get proper ANSI color codes, cursor movement, and even programs like htop or vim that rely on terminal capabilities.

03 — Critical choice
bash mode vs login mode — never use bash mode publicly

ttyd can be started with either bash or login as the command it runs. This is the most important configuration decision — the security implications are completely different.

ttyd bash — NEVER use publicly
Opens a bash shell directly as whatever user ran ttyd (root if started via sudo). No password is asked. Anyone who opens the URL gets instant root access. Only use for local testing where nobody else can reach the port.
ttyd login — what we use
Spawns /bin/login — a proper Linux login process. Shows a username + password prompt. Uses PAM (Linux auth system). Only authenticated users get a session. The correct choice for any public-facing terminal.
✗ Many online tutorials show ttyd bash — this is fine for local testing on a private port, but catastrophic if exposed publicly. Since our ttyd is accessible at terminal.dustyserver.co.in worldwide, we always use ttyd login.
04 — Comparison
Three ways to access spserver — when to use each
SSH — primary
ssh sipu@192.168.31.150
Best performance. Copy-paste works perfectly. Key auth possible. Needs SSH client. Local network or Tailscale for remote.
ttyd — anywhere, any device
terminal.dustyserver.co.in
Works in any browser — phone, tablet, work PC, library kiosk. No install needed. Slightly higher latency than SSH. Best for quick access from unexpected devices.
Tailscale SSH — remote backup
ssh sipu@100.x.x.x
Works from any network. Bypasses CG-NAT. Emergency access if Cloudflare Tunnel goes down. Requires Tailscale app on client device.
✓ The practical use for ttyd: you're on a different device — a friend's laptop, your tablet, a work machine where you can't install apps — and need to quickly SSH into spserver. Open the browser, go to terminal.dustyserver.co.in, login, done.

01 — Install
Step 1 — Install ttyd
1
Install from Ubuntu package repository
▼

ttyd is available in Ubuntu's official package repository — the simplest and most reliable installation method. It installs the binary and all required libraries.

bash
sudo apt install ttyd -y

# Verify installation and check version
ttyd --version
# Should print: ttyd/x.x.x (libwebsockets/x.x)

# Confirm binary location
which ttyd
# Should print: /usr/bin/ttyd
02 — Test
Step 2 — Test manually before creating service
2
Run ttyd manually and test in browser
▼

Always test a service manually first before turning it into a systemd service. This confirms the binary works, the port is available, and you can see any errors directly in the terminal output. Use Ctrl+C to stop the manual run.

bash — run on spserver
# Run ttyd manually in the foreground
# --writable: allow keyboard input (not just read-only)
# -p 7681: listen on port 7681
# login: run /bin/login (requires username+password)
sudo ttyd --writable -p 7681 login

# Open in browser (from your Windows PC on same network):
# http://192.168.31.150:7681
# You should see a login prompt — type "sipu" then your password

# Press Ctrl+C to stop after confirming it works
⚠ If the browser shows nothing when visiting port 7681, UFW is likely blocking it. Open it first: sudo ufw allow 7681/tcp. Then test again.
03 — Service
Step 3 — Create systemd service
3
Write the systemd unit file
▼

A systemd service makes ttyd start automatically on every boot and restart if it crashes. We run ttyd as root because /bin/login requires root privileges to authenticate users via PAM and switch to the correct user after login.

bash
sudo nano /etc/systemd/system/ttyd.service

Paste this content:

/etc/systemd/system/ttyd.service
[Unit]
Description=ttyd browser terminal
After=network.target

[Service]
ExecStart=/usr/bin/ttyd --writable -p 7681 login
Restart=always
RestartSec=5
User=root

[Install]
WantedBy=multi-user.target
4
Enable, start, and verify
▼
bash
# Tell systemd about the new service file
sudo systemctl daemon-reload

# Enable — auto-start on every boot
sudo systemctl enable ttyd

# Start — run it right now
sudo systemctl start ttyd

# Verify it is running
sudo systemctl status ttyd
# Look for: Active: active (running)

# Confirm it is listening on port 7681
sudo ss -tlnp | grep 7681
04 — Firewall
Step 4 — Open port 7681 in UFW
5
Allow port 7681 through UFW
▼

UFW blocks all incoming by default. Without opening port 7681, ttyd is running but completely unreachable from the network — this was one of the first blockers we hit. The browser shows nothing and gives a "connection refused" or timeout error.

bash
sudo ufw allow 7681/tcp

# Verify the rule is active
sudo ufw status
# Should show: 7681/tcp   ALLOW   Anywhere

# Test from local network browser
# Open: http://192.168.31.150:7681
# Login prompt should appear
05 — Public access
Step 5 — Expose via Cloudflare Tunnel
6
Add ingress rule and create DNS record
▼

Without this step, ttyd is only accessible on the local network. Adding it to the Cloudflare Tunnel makes it publicly accessible at terminal.dustyserver.co.in from anywhere in the world.

bash
# Step 1: Add ingress rule to Cloudflare Tunnel config
sudo nano /etc/cloudflared/config.yml

# Add this block BEFORE the catch-all line:
#   - hostname: terminal.dustyserver.co.in
#     service: http://localhost:7681

# Step 2: Create the DNS CNAME record
cloudflared tunnel route dns spserver terminal.dustyserver.co.in

# Step 3: Restart cloudflared to pick up config changes
sudo systemctl restart cloudflared

# Step 4: Wait 1-2 minutes for DNS to propagate
# Step 5: Open in browser from anywhere:
# https://terminal.dustyserver.co.in
✓ Once live, terminal.dustyserver.co.in works from your phone on mobile data, your work PC, a friend's laptop — any browser anywhere. Login with your sipu username and password.
01 — Service file
systemd service file — every line explained
/etc/systemd/system/ttyd.service
[Unit]
Description=ttyd browser terminal
After=network.target

[Service]
ExecStart=/usr/bin/ttyd --writable -p 7681 login
Restart=always
RestartSec=5
User=root

[Install]
WantedBy=multi-user.target
LineWhat it does
[Unit]Metadata section — describes the service and its dependencies.
Description=ttyd browser terminalHuman-readable name shown in systemctl status output.
After=network.targetWait for networking to be ready before starting. Without this, ttyd might start before the network interface is up and fail to bind to port 7681.
[Service]Service behavior section.
ExecStart=/usr/bin/ttyd --writable -p 7681 loginThe exact command to run. Full path to binary required. --writable enables keyboard input. -p 7681 sets the port. login runs /bin/login — the authenticated login shell.
Restart=alwaysIf ttyd crashes for any reason, systemd automatically restarts it.
RestartSec=5Wait 5 seconds before restarting after a crash — prevents rapid restart loops.
User=rootRequired for login mode. /bin/login needs root to authenticate users via PAM and switch user identity after login. Running as non-root causes PAM authentication failures.
[Install]Defines how systemctl enable wires the service into the boot process.
WantedBy=multi-user.targetStart this service in normal multi-user mode (standard server boot). This is the correct target for network services.
02 — ttyd flags
All useful ttyd flags and options
FlagEffect
--writableAllow keyboard input. Without this, the terminal is read-only — you can see output but can't type. Always include this flag.
-p PORTListen on a specific port. Default is 7681. Change if that port is in use.
-i INTERFACEBind to a specific network interface. Default binds to all (0.0.0.0). Use -i 127.0.0.1 to restrict to localhost only — useful if proxying through Nginx.
-c user:passwordAdd HTTP Basic Auth in front of the terminal. Simple username/password gate before the login prompt even appears. Weak but adds a layer.
-t fontSize=16Set the terminal font size. Can also set fontFamily, theme etc. Multiple -t flags allowed.
-t disableLeaveAlert=trueDisable the browser "are you sure you want to leave?" alert when closing the tab.
-m MAX_CLIENTSLimit simultaneous connections. Default is unlimited. -m 1 allows only one session at a time.
-oOnce mode — exit ttyd after the first client disconnects. Useful for one-shot scripts.
loginThe command to run. This runs /bin/login — requires auth. Can be replaced with any command: bash, htop, python3.

Example — ttyd with basic auth and font size set:

ExecStart example with extra flags
ExecStart=/usr/bin/ttyd --writable -p 7681 \
  -c admin:secretpassword \
  -t fontSize=16 \
  -t disableLeaveAlert=true \
  login
ℹ The -c HTTP Basic Auth flag adds a weak layer of protection — credentials are base64 encoded, not encrypted. Only useful over HTTPS (which we have via Cloudflare). The real security comes from the Linux login prompt that appears after.

01 — ttyd management
Commands for managing ttyd
sudo systemctl status ttyd
Check if ttyd is running — green active means it is listening on port 7681.
sudo systemctl start ttyd
Start the ttyd service.
sudo systemctl stop ttyd
Stop ttyd — closes all active browser terminal sessions.
sudo systemctl restart ttyd
Restart ttyd — use after editing the service file.
sudo systemctl enable ttyd
Enable auto-start on boot.
sudo systemctl daemon-reload
Reload systemd after editing /etc/systemd/system/ttyd.service — always run before restart.
sudo ss -tlnp | grep 7681
Verify ttyd is listening on port 7681. No output = ttyd not running.
sudo pkill ttyd
Kill all ttyd processes — useful if port 7681 is stuck occupied by an old run.
sudo journalctl -u ttyd -f
Follow ttyd logs live — shows connections, disconnections, errors.
sudo journalctl -u ttyd --since "10 minutes ago"
Show ttyd logs from the last 10 minutes.
curl http://localhost:7681
Test ttyd is responding locally — should return the xterm.js HTML page.
ttyd --version
Show installed ttyd version.
ttyd --help
Show all available flags and options with descriptions.
sudo cat /var/log/auth.log | grep ttyd
Check auth log for login attempts through ttyd — shows failed and successful logins.
01 — What went wrong
Every blocker hit during ttyd setup
⚠ Blocker 1
Port 7681 already in use — ttyd won't start
After running sudo ttyd --writable -p 7681 login during testing, pressed Ctrl+C to stop it. Then tried to start the systemd service and it failed — "address already in use". A previous ttyd test was still running in the background.
✓ Fix
sudo pkill ttyd — kill leftover processes
Ctrl+C only sends SIGINT to the foreground process. If ttyd was run with & or in another session, it keeps running. Fix: sudo pkill ttyd kills all ttyd processes. Then sudo ss -tlnp | grep 7681 to confirm the port is free, then start the service normally.
⚠ Blocker 2
Browser shows nothing — port 7681 blocked by UFW
ttyd was running and confirmed listening on port 7681. Opening http://192.168.31.150:7681 in the browser on the Windows PC showed "This site can't be reached" with ERR_CONNECTION_REFUSED or a timeout. Everything on the server looked fine.
✓ Fix
sudo ufw allow 7681/tcp
UFW was blocking port 7681 — the standard pattern for any new service. Fix: sudo ufw allow 7681/tcp. Immediately after, the browser loaded the terminal. This was the moment the diagnostic pattern was established: service running → curl localhost:PORT works → browser doesn't work → UFW is the culprit.
⚠ Blocker 3
Terminal appeared but was read-only — couldn't type anything
The ttyd interface loaded in the browser, the login prompt appeared, but typing had no effect — keystrokes were not reaching the terminal. Clicking in the terminal window did nothing.
✓ Fix
Add --writable flag to ExecStart
ttyd defaults to read-only mode without the --writable flag. The service file was written without it initially: ExecStart=/usr/bin/ttyd -p 7681 login. Fix: add --writable to the ExecStart line, run sudo systemctl daemon-reload then sudo systemctl restart ttyd. Keyboard input then worked immediately.
⚠ Blocker 4
Cloudflare Tunnel 502 — typo in config.yml service URL
After adding the terminal ingress rule to config.yml and restarting cloudflared, opening terminal.dustyserver.co.in showed 502 Bad Gateway. ttyd was running and responding to local curl requests. The tunnel config looked correct at first glance.
✓ Fix
Typo: "locakhost:7681" instead of "localhost:7681"
Careful re-reading of config.yml found service: http://locakhost:7681 — the letters 'a' and 'k' were transposed. cloudflared was trying to connect to a non-existent hostname. Fix: correct the typo to localhost, restart cloudflared, and the terminal immediately became accessible at terminal.dustyserver.co.in.
01 — Issues and fixes
Common ttyd problems
ttyd service fails to start — "address already in use"
Port 7681 is occupied by an old ttyd process or another service. Fix: (1) Kill old ttyd: sudo pkill ttyd. (2) Check what's using the port: sudo ss -tlnp | grep 7681. (3) If another service is using 7681, change ttyd's port in the service file to something else (e.g. 7682). (4) Run sudo systemctl daemon-reload and restart ttyd.
Login fails — "Login incorrect" even with correct password
PAM authentication issue. Possible causes: (1) ttyd is not running as root — User=root must be in the service file. Check: sudo systemctl status ttyd — look for which user it runs as. (2) The user does not have a Linux password set — set one: sudo passwd sipu. (3) Caps lock is on in the browser — browser terminals don't always show this visually.
terminal.dustyserver.co.in shows 502 Bad Gateway
Cloudflare Tunnel can't reach ttyd. Check in order: (1) Is ttyd running? sudo systemctl status ttyd. (2) Is it on port 7681? sudo ss -tlnp | grep 7681. (3) Test locally: curl http://localhost:7681 — should return HTML. (4) Check config.yml for typos — specifically "localhost" vs "locakhost". (5) Restart cloudflared after fixing: sudo systemctl restart cloudflared.
Terminal renders but keyboard input doesn't work
The --writable flag is missing from ExecStart. Check: sudo systemctl status ttyd — look at the ExecStart line in the output. Fix: edit /etc/systemd/system/ttyd.service, add --writable to ExecStart, run sudo systemctl daemon-reload && sudo systemctl restart ttyd.
Terminal session disconnects after a few minutes of inactivity
WebSocket connection timing out due to inactivity. The Cloudflare Tunnel has a default timeout for idle connections. Fix options: (1) Set ClientAliveInterval 60 in /etc/ssh/sshd_config (this applies to SSH sessions in the terminal). (2) Configure Cloudflare Tunnel keepalives. (3) Run a keepalive command in the terminal like while true; do sleep 30; done & before going idle.
htop / vim / ncurses apps look garbled
Terminal size (rows/columns) isn't being communicated correctly. Try: (1) Resize the browser window. (2) Run resize command in the terminal. (3) Set TERM environment: export TERM=xterm-256color. Most apps should work correctly in ttyd's xterm.js terminal — if still garbled, it may be a specific app compatibility issue.
How to check who is currently connected to ttyd
Check active terminal sessions: who or w commands show all logged-in users including their connection source. sudo journalctl -u ttyd -f shows WebSocket connections in real time. sudo cat /var/log/auth.log | grep login shows all login attempts and successes.
01 — What's next
Future scope
⟶ Things to explore from here
Cloudflare Access — add email OTP or Google SSO authentication in front of the terminal before the login prompt appears. Configured in Cloudflare dashboard → Zero Trust → Access → Applications. Anyone trying to open terminal.dustyserver.co.in would need to verify their email first.
HTTP Basic Auth — add the -c username:password flag to ttyd's ExecStart as an additional quick security layer before the Linux login prompt. Simple but adds friction for bots scanning the URL.
fail2ban integration — install fail2ban and configure it to watch /var/log/auth.log for failed ttyd login attempts. After N failures from the same IP, automatically add a UFW deny rule for that IP. Protects against brute force attacks on the login prompt.
Wetty as alternative — Wetty is another browser terminal tool, Node.js-based, with a cleaner UI. Can be run inside Docker. Worth evaluating as an alternative to ttyd for a more polished interface.
Multiple users — create additional Linux users on spserver (e.g. for collaborators). Each user gets their own credentials and home directory. ttyd's login mode handles multiple users correctly — each gets their own isolated session.
Read-only terminal for monitoring — run a second ttyd instance on a different port in read-only mode (without --writable) showing live system stats: ttyd -p 7682 watch -n1 "df -h && free -h && uptime". Expose it as a public dashboard — can't be typed into, just displays data.
Audit logging — use script command or auditd to record everything typed and output in terminal sessions. Compliance-grade session logging for all ttyd connections.
