Flask + MySQL Stack

The full spserver-flask multi-container stack — Flask and MySQL running as separate containers, communicating over Docker's internal network, with persistent data in a named volume. This is Phase 2 complete.

01 — What & Why
Why Flask and MySQL run as separate containers

The multi-container pattern — one container per service — is the foundation of how Docker is used in practice. Flask and MySQL are separated because they have completely different lifecycles, resource profiles, and failure modes. Keeping them separate means you can restart, update, or replace one without touching the other.

Single container — Flask + MySQL together
Running both in one container requires a process manager (supervisord) inside the container. You can't update MySQL without rebuilding the Flask image. The container is large and complex. MySQL data lives inside the container and is lost on rebuild. You can't scale Flask independently of MySQL.
Two containers — Flask + MySQL separated
Each container does one thing. Flask can be rebuilt and redeployed without touching MySQL or its data. MySQL can be upgraded independently. MySQL data lives in a named volume that outlasts both containers. This is the separation of concerns principle applied to infrastructure.
ℹ This guide is the culmination of Phase 2. It brings together the Dockerfile, Docker Compose, Flask app, and Nginx knowledge from the previous guides into a single working production-like stack. The individual pieces are covered in depth in their own guides — this guide focuses on how they fit together and operate as a unit.

03 — Networking
How Flask reaches MySQL — Compose internal DNS

Compose creates a private bridge network for the stack and registers each service's name as a hostname in its internal DNS. From inside the flask-app container, the hostname mysql resolves to the MySQL container's IP automatically. No IP addresses, no host port mapping required between the two services.

Flask connection string: host=mysql port=3306 user=flaskuser password=flaskpass database=flaskdb Inside flask-app container: $ nslookup mysql Server: 127.0.0.11 ← Docker's embedded DNS resolver Address: 172.18.0.x ← mysql container's IP on the bridge network mysql:3306 is not accessible from the host — no ports: mapping in compose Only flask-app (on the same network) can reach it
ℹ The embedded DNS server is always at 127.0.0.11 inside any Docker container. It resolves service names defined in the Compose file to their current container IPs. If a container is restarted and gets a new IP, the DNS record updates automatically — the service name stays constant.
04 — Persistence
Why MySQL data must be in a named volume

A Docker container's filesystem is ephemeral — it exists only as long as the container exists. When a container is removed (by docker compose down, rebuild, or crash recovery), everything written to the container's own filesystem is gone. For a database, this would mean losing all data on every restart.

A named volume exists outside the container lifecycle. Docker creates and manages it at /var/lib/docker/volumes/spserver-flask_mysql-data/ on the host. The MySQL container mounts this directory as /var/lib/mysql inside the container. When the container is recreated, it mounts the same volume — all the data is still there.

docker compose down
Stops and removes containers and the Docker network. The mysql-data volume is not touched. All database data survives. Start again with docker compose up -d and MySQL picks up exactly where it left off.
docker compose down -v
Stops and removes containers, network, and all named volumes. The mysql-data volume is deleted — all database data is permanently gone. Only use this intentionally: to completely reset the stack to a clean state.
01 — Compose file
The full docker-compose.yml — annotated

This is the complete, production-ready Compose file for the spserver-flask stack. Every field is annotated — read the comments to understand the decisions behind each line.

spserver-flask/docker-compose.yml
services:

  flask-app:
    # Build the Flask image from the Dockerfile in this directory
    # Use --build flag when running docker compose up to rebuild after code changes
    build: .

    # Expose Flask to the host on port 5000
    # MySQL has no ports: entry — it is internal-only
    ports:
      - "5000:5000"

    # DB config passed as environment variables
    # Flask reads these with os.environ.get() in app.py
    # DB_HOST must match the MySQL service name below — Compose DNS resolves it
    environment:
      - DB_HOST=mysql
      - DB_PORT=3306
      - DB_NAME=flaskdb
      - DB_USER=flaskuser
      - DB_PASSWORD=flaskpass

    # Wait for MySQL to be healthy before starting Flask
    # depends_on alone only waits for the container to start, not MySQL to be ready
    # condition: service_healthy requires the healthcheck below to pass first
    depends_on:
      mysql:
        condition: service_healthy

    # Auto-restart Flask if it crashes or if spserver reboots
    restart: unless-stopped

  mysql:
    # Official MySQL 8.0 image — stable, well-maintained, widely used
    image: mysql:8.0

    # No ports: entry — MySQL is accessible only within the Docker network
    # Exposing 3306 to the host would allow LAN access to the raw database

    # MySQL official image reads these on first start to initialise the database
    # On subsequent starts (when /var/lib/mysql already has data), these are ignored
    environment:
      - MYSQL_ROOT_PASSWORD=rootsecret
      - MYSQL_DATABASE=flaskdb
      - MYSQL_USER=flaskuser
      - MYSQL_PASSWORD=flaskpass

    # Named volume — Docker manages the storage location on the host
    # /var/lib/mysql is where MySQL stores all its data files
    # This data survives docker compose down — only deleted with down -v
    volumes:
      - mysql-data:/var/lib/mysql

    # Health check — polls mysqladmin to detect when MySQL is actually accepting connections
    # MySQL takes 20–40 seconds to initialise on first start
    # start_period gives it time before failures start counting against retries
    healthcheck:
      test: ["CMD", "mysqladmin", "ping", "-h", "localhost", "-u", "root", "-prootsecret"]
      interval:     10s
      timeout:       5s
      retries:        5
      start_period: 30s

    restart: unless-stopped

# Declare the named volume used by the mysql service
# Without this top-level declaration, docker compose will error
volumes:
  mysql-data:
02 — MySQL container
MySQL container configuration — what each env var does

The MySQL official image reads specific environment variables on first start only — when /var/lib/mysql is empty. These create the initial database, user, and set passwords. On every subsequent start, these values are ignored and the existing data is used.

VariableValue usedWhat it does
MYSQL_ROOT_PASSWORD rootsecret Sets the root user password. Required — MySQL won't start without it. Used in the healthcheck command.
MYSQL_DATABASE flaskdb Creates this database on first start. Must match DB_NAME in the flask-app environment.
MYSQL_USER flaskuser Creates this user with full privileges on MYSQL_DATABASE. Flask connects as this user — not root.
MYSQL_PASSWORD flaskpass Password for MYSQL_USER. Must match DB_PASSWORD in the flask-app environment.
⚠ MySQL only initialises from these env vars when /var/lib/mysql is empty — i.e. when the mysql-data volume is new or has been deleted. If you change these values after the stack has run once, the changes have no effect. To apply new credentials, run docker compose down -v to wipe the volume, then docker compose up -d to reinitialise. This deletes all data.
03 — Flask container
Flask container configuration — connector and connection string

The Flask container is built from the project's Dockerfile using python:3.11-slim as the base image. The key configuration decisions that affect the DB connection:

requirements.txt — the connector that matters
Flask
mysql-connector-python
# mysql-connector-python — pure Python, no system dependencies, installs cleanly
# mysqlclient — C extension, requires pkg-config + libmysqlclient-dev in the image

app.py — how Flask reads the DB config
import os
import mysql.connector
from flask import Flask, jsonify

app = Flask(__name__)

DB_CONFIG = {
    'host':     os.environ.get('DB_HOST',     'localhost'),
    'port':     int(os.environ.get('DB_PORT', '3306')),
    'database': os.environ.get('DB_NAME',     'flaskdb'),
    'user':     os.environ.get('DB_USER',     'flaskuser'),
    'password': os.environ.get('DB_PASSWORD', ''),
}

# DB_HOST is set to 'mysql' in docker-compose.yml
# Compose DNS resolves 'mysql' to the MySQL container's internal IP
# This is why Flask doesn't use localhost or a hardcoded IP

@app.route('/health')
def health():
    try:
        conn = mysql.connector.connect(**DB_CONFIG)
        conn.close()
        return jsonify({'status': 'ok', 'db': 'reachable'})
    except Exception as e:
        return jsonify({'status': 'error', 'db': str(e)}), 500

if __name__ == '__main__':
    # 0.0.0.0 — listen on all interfaces inside the container
    # Without this, Flask binds to 127.0.0.1 and is unreachable from outside
    app.run(host='0.0.0.0', port=5000, debug=False)

Run from the spserver-flask/ directory. The first run builds the Flask image, pulls mysql:8.0, creates the volume and network, starts MySQL, waits for it to pass the healthcheck, then starts Flask. On a fresh system this takes 60–90 seconds.

# NAME                            STATUS

01 — Lifecycle
Full stack lifecycle commands
docker compose up -d --build
Build flask-app image and start all services in background. Use after any code or Dockerfile change.
docker compose up -d
Start the stack using the existing image. Use after a reboot or when nothing has changed.
docker compose down
Stop and remove containers and network. Volume is kept — MySQL data survives.
docker compose down -v
Stop, remove containers, network, and all named volumes.
⚠ Deletes all MySQL data permanently
docker compose restart flask-app
Restart only the Flask container. MySQL keeps running — data and connections unaffected.
docker compose stop
Stop all containers without removing them. Resume with docker compose start.
docker compose ps
Show status of all services. MySQL should show (healthy) when fully ready.
docker compose logs -f
Follow logs from all services live. Ctrl+C stops following — does not stop containers.
docker compose logs -f flask-app
Follow logs from Flask only — requests, errors, print statements.
docker compose logs --tail=100 mysql
Show last 100 lines of MySQL logs — useful for diagnosing startup issues.

01 — What went wrong
Blockers hit while running the Flask + MySQL stack
⚠ Blocker 1
Flask crashed on first startup — "Can't connect to MySQL server on 'mysql'"
The first time the stack was started, Flask came up before MySQL finished initialising. Even with depends_on, Flask started and immediately tried to connect to MySQL — which was still running its first-time setup (creating the database, user, and tables). The connection attempt failed and Flask exited with a MySQL connection error.
✓ Fix
Restart the Flask container once MySQL is healthy — then add a healthcheck
Short-term fix: waited for docker compose ps to show MySQL as (healthy), then ran docker compose restart flask-app. Flask connected successfully on the second attempt. Long-term fix: added the healthcheck block to the MySQL service and changed depends_on to use condition: service_healthy. Now Compose waits until MySQL passes the healthcheck before starting Flask — the crash on first boot no longer happens.
⚠ Blocker 2
pip install failed during build — mysqlclient gcc compilation error
The initial requirements.txt included mysqlclient. The Docker build failed at RUN pip install -r requirements.txt with a long gcc compiler error: mysql_config not found and EnvironmentError: mysql_config not found. The python:3.11-slim base image doesn't include pkg-config or libmysqlclient-dev.
✓ Fix
Replace mysqlclient with mysql-connector-python in requirements.txt
Replaced mysqlclient with mysql-connector-python in requirements.txt. This is a pure-Python MySQL driver that compiles nothing — it installs in the slim image without any system dependencies. The Python API is slightly different (import mysql.connector instead of import MySQLdb) but fully featured for our use case. Rebuilt the image and the pip install completed cleanly.
⚠ Blocker 3
MySQL data wiped unexpectedly — docker compose down -v used by mistake
During a troubleshooting session, ran docker compose down -v intending to just restart cleanly. This deleted the mysql-data volume — all tables and data were gone. The stack restarted with a completely empty database.
✓ Fix
Use docker compose down (without -v) for normal restarts
The -v flag is a destructive operation — it should only be used intentionally when a full database reset is desired. For all normal stop/start cycles, use plain docker compose down. To verify the volume exists before starting: docker volume ls | grep mysql-data. Set up a .aliases or just always double-check the command before running it. In dev, losing data is inconvenient — in production, it would be catastrophic.
⚠ Blocker 4
.bashrc alias for "free" caused division-by-zero error in MOTD on SSH login
After adding a custom alias for free in .bashrc (to format the output differently), SSH logins started showing a Python traceback in the MOTD: ZeroDivisionError: integer division or modulo by zero. The MOTD script was calling free expecting standard output, but the alias was intercepting the call and returning reformatted output that the script couldn't parse.
✓ Fix
Use /usr/bin/free explicitly in scripts — aliases don't apply to non-interactive shells
The MOTD script was using free expecting the standard GNU coreutils output. The alias was only intended for interactive terminal use but it was overriding the command globally. Fix: changed the alias in .bashrc to use a different name (e.g. alias freeh='free -h') instead of overriding free directly. Also updated any custom scripts to call /usr/bin/free with the full path — full paths bypass aliases. Aliases are not expanded in non-interactive shells, but the MOTD was calling free in a context where the alias was active.
01 — Issues and fixes
Common Flask + MySQL stack problems and fixes
Flask exits on startup — MySQL connection refused
MySQL isn't ready yet. Check: (1) docker compose ps — does MySQL show (healthy)? If it shows "starting", wait and check again. (2) docker compose logs mysql — look for "ready for connections" or "mysqld: ready for connections". (3) Once MySQL is healthy, run docker compose restart flask-app. Permanent fix: add a healthcheck to the mysql service and use condition: service_healthy in depends_on — Compose will then wait for MySQL readiness before starting Flask.
Flask can't reach MySQL — "Unknown MySQL server host 'mysql'"
Compose DNS isn't resolving the service name. Check: (1) Is DB_HOST set to mysql (the exact service name in docker-compose.yml)? Verify: docker compose exec flask-app env | grep DB_HOST. (2) Are both services in the same docker-compose.yml? Services in different files are on different networks. (3) Test DNS: docker compose exec flask-app python -c "import socket; print(socket.gethostbyname('mysql'))". If this fails, the services aren't on the same network — check the compose file structure.
ModuleNotFoundError: No module named 'mysql'
mysql-connector-python is missing from requirements.txt or the image wasn't rebuilt after adding it. Fix: (1) Confirm the package is in requirements.txt. (2) Rebuild: docker compose up -d --build. (3) Verify it's in the container: docker compose exec flask-app pip show mysql-connector-python. If it shows "not found", the image is stale — force a rebuild with docker compose build --no-cache flask-app.
MySQL data gone after docker compose down
Was -v used? docker compose down -v deletes named volumes — this is irreversible. Check: docker volume ls | grep mysql — if the volume is missing, the data was deleted. Without a backup, the data is unrecoverable. Going forward: only use docker compose down (no -v) for normal stop/restart cycles. Use docker compose down -v only intentionally for a full reset.
MySQL env var changes not taking effect after restart
MySQL only reads env vars on first start — when /var/lib/mysql is empty. The volume already has data so the env vars are ignored. To apply new credentials: (1) Run docker compose down -v to wipe the volume. (2) Run docker compose up -d to reinitialise from the new env vars. Warning: this deletes all data. In production, use SQL commands (ALTER USER, CREATE DATABASE) to change config without data loss.
Access denied for MySQL user from Flask
Credentials mismatch between flask-app environment and mysql environment. Check: (1) docker compose exec flask-app env | grep DB_ — print Flask's DB config. (2) Compare DB_USER/DB_PASSWORD with MYSQL_USER/MYSQL_PASSWORD in the mysql service. They must match exactly — including case. (3) If credentials were changed after the volume was created: wipe with docker compose down -v and reinitialise. (4) Test the credentials directly: docker compose exec mysql mysql -u flaskuser -pflaskpass flaskdb -e "SELECT 1;".
How to create tables and seed data on first start
Two approaches: (1) Manual — once the stack is up and MySQL is healthy, run docker compose exec mysql mysql -u flaskuser -pflaskpass flaskdb and execute CREATE TABLE and INSERT statements interactively. (2) Automatic — create an init.sql file with all DDL and seed data, then add to the mysql service in compose: volumes: - ./init.sql:/docker-entrypoint-initdb.d/init.sql. MySQL runs any .sql files in /docker-entrypoint-initdb.d/ on first start. On subsequent starts (when the volume already has data), the file is ignored.
01 — What's next
Future scope — Phase 2 complete, Phase 3 ahead
⟶ Immediate improvements to this stack
Move credentials to a .env file — instead of hardcoding DB_PASSWORD=flaskpass in docker-compose.yml, create a .env file in the project root and add it to .gitignore. Docker Compose automatically loads .env for variable substitution: DB_PASSWORD=${DB_PASSWORD}. Credentials stay out of version control — the single most important security improvement for this stack.
Automatic schema initialisation — add an init.sql file to the repo containing all CREATE TABLE statements and seed data. Bind-mount it into MySQL: ./init.sql:/docker-entrypoint-initdb.d/init.sql. On first start, MySQL runs it automatically. Anyone who clones the repo and runs docker compose up -d gets a fully initialised database — no manual SQL required.
Connection retry in Flask — add a startup retry loop in app.py so Flask retries the MySQL connection on failure instead of crashing. Use exponential backoff: try, wait 2s, try, wait 4s, try, wait 8s, up to ~10 attempts. This makes the healthcheck-based wait unnecessary for most restarts and makes the app robust to transient network blips inside Docker.
MySQL backup strategy — the named volume persists data but is on spserver's local disk. If the disk fails, data is lost. Set up a cron job on the host: docker compose exec mysql mysqldump -u root -prootsecret flaskdb > /backups/flaskdb-$(date +%F).sql. Run daily, rotate old backups. For off-site backup, sync the dump files to a remote location via rsync or rclone to a cloud storage bucket.
