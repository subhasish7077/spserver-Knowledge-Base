Docker Compose

Defining and running a multi-container stack — Flask app + MySQL — from a single YAML file. Covers the full lifecycle, named volumes, internal networking, health checks, and why depends_on isn't enough.

01 — What & Why
What Docker Compose is — and why it exists

Docker Compose is a tool for defining and running multi-container Docker applications. Instead of starting each container with a separate docker run command (with all its flags), you describe the entire stack — every service, port, volume, network, and environment variable — in a single YAML file called docker-compose.yml. Then you start everything with one command: docker compose up -d.

The compose file is checked into version control alongside your code. Anyone who clones the repo and runs docker compose up gets the exact same environment — same containers, same config, same networking, same volume structure.

ℹ We use docker compose (no hyphen) — the modern Compose V2 plugin installed as part of Docker Engine. The old docker-compose (with hyphen, Python-based, v1) is deprecated. If your system only has the hyphenated version, upgrade to Docker Engine from the official repository.
02 — Compose vs docker run
Why Compose instead of plain docker run

Consider what it takes to run a Flask app with a MySQL database using only docker run:

bash — docker run approach (fragile)
# Step 1: create a network manually
docker network create spserver-net

# Step 2: create a volume manually
docker volume create mysql-data

# Step 3: start MySQL (remember ALL these flags every time)
docker run -d \
  --name mysql \
  --network spserver-net \
  --volume mysql-data:/var/lib/mysql \
  -e MYSQL_ROOT_PASSWORD=secret \
  -e MYSQL_DATABASE=flaskdb \
  -e MYSQL_USER=flaskuser \
  -e MYSQL_PASSWORD=flaskpass \
  mysql:8.0

# Step 4: start Flask (must remember the same network name)
docker run -d \
  --name flask-app \
  --network spserver-net \
  -p 5000:5000 \
  -e DB_HOST=mysql \
  -e DB_NAME=flaskdb \
  flask-app:latest

# To tear it all down, must remember to stop and remove each container,
# remove the network, and decide what to do about the volume.
Problems with docker run
Long, fragile shell commands that are hard to remember and easy to get wrong. Nothing is version-controlled — the configuration lives only in your shell history. Impossible to share reliably. No single command to bring the whole stack up or down.
Compose fixes all of this
The entire stack — both services, all environment variables, the network, the volume — is declared in one YAML file. Checked into git. Start everything: docker compose up -d. Stop everything: docker compose down. Reproducible on any machine.
03 — YAML structure
The top-level structure of docker-compose.yml

A docker-compose.yml has four top-level keys. Only services is required — the others are optional but used in almost every real setup.

docker-compose.yml — skeleton
# services — the containers to run (required)
services:
  service-name:
    image: some-image:tag    # OR use build: to build from Dockerfile
    ports: [...]
    environment: [...]
    volumes: [...]
    depends_on: [...]

# volumes — declare named volumes used by services (optional)
volumes:
  mysql-data:               # named volume — Docker manages its location

# networks — declare custom networks (optional)
# If omitted, Compose creates a default bridge network automatically
networks:
  backend:

# configs / secrets — advanced, for Swarm mode (not used here)
ℹ The version: key (e.g. version: "3.8") used to be required but is now obsolete in Compose V2. Modern Docker Compose ignores it and may warn about it. Leave it out entirely in new files.
How services communicate

Compose automatically puts all services in the same default bridge network. Services can reach each other using their service name as the hostname — no IP addresses needed.

Host machine
flask-app → port 5000 mapped to host
connects to mysql:3306
mysql → internal only, no host mapping
persists data across restarts
mysql-data volume → /var/lib/mysql

Inside the Flask container, the database host is simply mysql — Compose's internal DNS resolves this to the MySQL container's IP automatically. This is why we set DB_HOST=mysql in the Flask environment.

04 — Field reference
Key fields in a service definition
image
Use a pre-built image
Pulls and runs an existing image from Docker Hub or a registry. image: mysql:8.0 — Compose pulls mysql:8.0 from Docker Hub if not already local.
build
Build from a Dockerfile
build: . tells Compose to build the image using the Dockerfile in the current directory. The --build flag in docker compose up --build forces a rebuild. Without it, Compose reuses the existing image.
ports
Publish ports to the host
Maps host port to container port: "5000:5000" — host:container. Only services that need to be reachable from outside Docker should have ports mapped. MySQL should NOT — it should only be accessible from within the Docker network.
environment
Set environment variables
Pass env vars into the container. Can be a list (- KEY=value) or a map (KEY: value). Used for MySQL credentials, Flask config, connection strings. Sensitive values should use env_file: or Docker secrets instead of hardcoding.
depends_on
Start order dependency
Tells Compose to start this service after another service's container is running. Does not wait for the service inside to be ready — MySQL container running does not mean MySQL is accepting connections. Use healthcheck + condition: service_healthy for real readiness.
volumes
Mount storage into the container
Two forms: named volume (mysql-data:/var/lib/mysql) — Docker manages the storage location, data persists across docker compose down. Bind mount (./app:/app) — mounts a host directory directly, changes are reflected live.
restart
Restart policy
restart: unless-stopped — container auto-restarts on crash or server reboot unless you explicitly stopped it. Other values: no (default), always, on-failure. Essential for production services on spserver.
healthcheck
Readiness probe
Defines a command Docker runs periodically to check if the service is healthy. Other services using depends_on: condition: service_healthy wait until this check passes. Use mysqladmin ping for MySQL to detect true readiness, not just container start.
05 — Storage
Named volumes vs bind mounts — when to use each

Both allow data to persist outside the container's writable layer, but they serve different purposes:

Named volume — mysql-data:/var/lib/mysql
Docker creates and manages the storage location on the host (under /var/lib/docker/volumes/). You never need to know the exact path. Data survives docker compose down and even container removal. The only way to delete it is docker compose down -v or docker volume rm.

Use for: database data, anything you want Docker to manage. Best practice for stateful services.
Bind mount — ./app:/app
Mounts a specific directory from the host filesystem directly into the container. Changes on the host are reflected inside the container immediately — and vice versa. You control the exact path.

Use for: development — mounting source code so changes are live without rebuilding the image. Not recommended for production database storage (permissions, portability issues).
⚠ docker compose down does NOT delete volumes. This is intentional — your database data survives a stack restart. To wipe volumes too, use docker compose down -v. Be careful: this permanently deletes all data in named volumes declared in the compose file.
ScenarioUseWhy
MySQL data persistenceNamed volumeDocker manages it, survives restarts, portable across machines
Live code reloading in devBind mountEdits on host reflect inside container instantly
Nginx config filesBind mountConfig lives in your repo, easy to edit and track with git
App logsNamed volume or bind mountBind mount if you want logs on the host for easy access; named volume if just for retention

02 — Write the compose file
Step 1 — Create docker-compose.yml for Flask + MySQL
1
Write the full docker-compose.yml
▼

Every field is annotated to explain why it is there. Read the comments — they explain the decisions, not just the syntax.

spserver-flask/docker-compose.yml
services:

  flask-app:
    # Build from the Dockerfile in the current directory
    # Use 'image: flask-app:latest' instead if the image is already built
    build: .

    # Map host port 5000 to container port 5000
    # Flask is the only service that should be reachable from the host
    ports:
      - "5000:5000"

    # Environment variables passed into the Flask container
    # DB_HOST=mysql  — "mysql" is the service name, resolved by Compose DNS
    environment:
      - DB_HOST=mysql
      - DB_PORT=3306
      - DB_NAME=flaskdb
      - DB_USER=flaskuser
      - DB_PASSWORD=flaskpass

    # Start flask-app only after mysql is healthy (not just running)
    # condition: service_healthy requires a healthcheck on the mysql service
    depends_on:
      mysql:
        condition: service_healthy

    # Restart automatically if the container crashes or spserver reboots
    restart: unless-stopped

  mysql:
    # Official MySQL 8.0 image from Docker Hub
    image: mysql:8.0

    # NO ports mapping — MySQL is intentionally internal only
    # Only flask-app (on the same Docker network) can reach port 3306
    # Exposing MySQL to the host would be a security risk

    # MySQL official image reads these env vars on first start
    # to initialise the database, root password, user, and password
    environment:
      - MYSQL_ROOT_PASSWORD=rootsecret
      - MYSQL_DATABASE=flaskdb
      - MYSQL_USER=flaskuser
      - MYSQL_PASSWORD=flaskpass

    # Named volume — persists MySQL data across docker compose down
    # Without this, all database data is lost every time the container stops
    volumes:
      - mysql-data:/var/lib/mysql

    # Health check — polls mysqladmin to detect when MySQL is actually ready
    # depends_on alone only waits for the container to start, not MySQL to boot
    # Flask will not start until this check passes (because of condition: service_healthy above)
    healthcheck:
      test: ["CMD", "mysqladmin", "ping", "-h", "localhost", "-u", "root", "-prootsecret"]
      interval: 10s     # check every 10 seconds
      timeout: 5s       # fail if no response within 5 seconds
      retries: 5        # mark unhealthy after 5 consecutive failures
      start_period: 30s # give MySQL 30s to initialise before health checks count

    restart: unless-stopped

# Declare named volumes used by services above
# Docker creates and manages this volume's location on the host
volumes:
  mysql-data:
ℹ Why no ports on MySQL? MySQL running on port 3306 of the host means anyone on the LAN can attempt to connect to your database. Inside Docker, containers on the same Compose network reach each other by service name — Flask can connect to mysql:3306 without the port ever being exposed to the host. Always keep database ports internal.
03 — Start
Step 2 — Start the full stack
2
docker compose up -d
▼

Run from the directory that contains docker-compose.yml. The -d flag runs all containers in the background (detached mode) — without it, the logs stream to your terminal and Ctrl+C stops the entire stack.

bash — run from spserver-flask/
# Build the flask-app image and start all services in the background
docker compose up -d

# What happens:
# 1. Builds flask-app image from Dockerfile (first time)
# 2. Pulls mysql:8.0 from Docker Hub (first time)
# 3. Creates the mysql-data named volume
# 4. Creates the default bridge network for these services
# 5. Starts mysql container
# 6. Waits for mysql healthcheck to pass (up to ~80 seconds)
# 7. Starts flask-app container

# Check that both services are running
docker compose ps
# NAME                    STATUS
# spserver-flask-mysql-1      Up (healthy)
# spserver-flask-flask-app-1  Up
04 — Verify
Step 3 — Confirm the stack is working
bash
# Test Flask is responding on the host
curl http://localhost:5000

# View live logs from all services
docker compose logs -f

# View logs from only flask-app
docker compose logs -f flask-app

# Confirm MySQL container is healthy
docker compose ps
# mysql column should show: Up (healthy)

# Open a shell inside the running MySQL container to inspect the DB
docker compose exec mysql mysql -u flaskuser -pflaskpass flaskdb
# Then run SQL: SHOW TABLES;

# Open a shell inside the Flask container
docker compose exec flask-app /bin/bash
✓ If docker compose ps shows both services as Up and MySQL as (healthy), and curl localhost:5000 returns a response — the stack is fully operational. Flask is talking to MySQL through Docker's internal network.
05 — Rebuilding
Step 4 — Rebuild after Dockerfile or code changes

docker compose up -d by default reuses the existing flask-app image — it does not automatically rebuild when you change the Dockerfile or application code. You must add --build to force a rebuild.

bash — after changing Dockerfile or app code
# Rebuild the flask-app image and restart only changed services
docker compose up -d --build

# Compose is smart: MySQL won't restart if its config hasn't changed
# Only flask-app is rebuilt and restarted

# To stop all services without removing volumes
docker compose down
# Containers and networks are removed, mysql-data volume is kept

# To stop all services AND delete all volumes (wipes database data)
docker compose down -v
# WARNING: mysql-data volume is deleted — all database data is lost

# To restart a single service without rebuilding
docker compose restart flask-app
⚠ If you change the docker-compose.yml (add a new environment variable, change a port, add a volume) you must run docker compose up -d again — Compose will detect the changed config and recreate the affected containers automatically.
01 — Lifecycle
Stack lifecycle commands
docker compose up -d
Create and start all services in the background. Pulls images if needed, creates networks and volumes.
docker compose up -d --build
Same as above but rebuilds images first — required after Dockerfile or source changes
docker compose down
Stop and remove all containers and networks. Named volumes are kept — data is safe.
docker compose down -v
Stop and remove containers, networks, AND named volumes. Permanently deletes database data.
docker compose stop
Stop all containers without removing them. Containers can be restarted with docker compose start.
docker compose start
Start existing (stopped) containers — does not create new ones or rebuild images
docker compose restart
Restart all running services. Useful to apply new environment variables.
docker compose restart flask-app
Restart only the flask-app service — MySQL keeps running uninterrupted
docker compose pull
Pull the latest versions of all images defined in the compose file

01 — What went wrong
Blockers hit while setting up Flask + MySQL Compose
⚠ Blocker 1
Flask crashed on startup — "Can't connect to MySQL server" — depends_on wasn't enough
Stack started successfully but docker compose logs flask-app showed the Flask app crashing immediately with a MySQL connection error. The MySQL container was running but MySQL inside it hadn't finished initialising — it takes 10–30 seconds to set up the database on first run.
✓ Fix
Add healthcheck to MySQL and use condition: service_healthy in depends_on
depends_on alone only tells Compose to start the mysql container before flask-app — it does not wait for MySQL to be accepting connections. Added a healthcheck block to the mysql service using mysqladmin ping, and changed depends_on to use condition: service_healthy. Now Compose waits until MySQL passes the health check before starting Flask. The start_period: 30s gives MySQL time to initialise before failures count against it.
⚠ Blocker 2
pip install failed during docker build — mysqlclient compilation error
Build failed during RUN pip install -r requirements.txt with: error: command '/usr/bin/x86_64-linux-gnu-gcc' failed with exit code 1 and a long stack trace about missing headers. The initial requirements.txt included mysqlclient as the MySQL driver.
✓ Fix
Switch from mysqlclient to mysql-connector-python
mysqlclient is a C extension. It requires pkg-config and libmysqlclient-dev (the MySQL C client library headers) to compile at install time. These system packages are not present in python:3.11-slim. The fix: replace mysqlclient with mysql-connector-python in requirements.txt. This is a pure-Python MySQL driver — no C compilation, no system dependencies, installs cleanly in the slim image. The Python API is slightly different but functionally equivalent for our use case.
⚠ Blocker 3
Stale data after recreating MySQL container — environment variables didn't re-initialise
Changed MYSQL_DATABASE and MYSQL_USER in the compose file, ran docker compose down and docker compose up -d, but the new database and user were not created. The old data and credentials were still there.
✓ Fix
MySQL only initialises from env vars if the data directory is empty — use down -v to reset
The MySQL official image only runs its initialisation (creating the database, user, and setting passwords from env vars) when /var/lib/mysql is empty — i.e., on first start. If the mysql-data volume already exists and has data in it, the env vars are ignored. To fully reset: docker compose down -v (deletes the volume), then docker compose up -d. This wipes all data so only do it in a dev/reset context.
01 — Issues and fixes
Common Docker Compose problems and fixes
Flask exits immediately — MySQL connection refused
MySQL container is running but MySQL isn't ready yet. Check: (1) docker compose ps — does MySQL show (healthy)? If not, the healthcheck hasn't passed yet — wait longer or check MySQL logs. (2) Is depends_on using condition: service_healthy? If it just says depends_on: [mysql], it doesn't wait for readiness. (3) docker compose logs mysql — look for "ready for connections" message. Fix: add a healthcheck + condition: service_healthy. As a temporary workaround, add a retry loop in app.py to retry the DB connection on startup.
Cannot connect to MySQL from Flask — connection refused or unknown host
Flask is trying to connect to the wrong host. Check: (1) Is DB_HOST set to mysql (the service name) — not localhost or an IP? Inside Docker, use the service name. (2) Is the Flask app reading the DB_HOST env var correctly? Print it at startup to confirm. (3) Are both services in the same Compose project (same directory, same compose file)? Services in different projects are on different networks. (4) Run docker compose exec flask-app ping mysql — if it resolves, networking is fine and the issue is in the app code.
Changes to app.py are not reflected after docker compose up -d
Compose reused the existing flask-app image. Fix: always use docker compose up -d --build after changing the Dockerfile or any file that gets COPY'd into the image. Without --build, Compose does not rebuild — it uses whatever image exists from the last build. Confirm the rebuild happened by checking the timestamp in docker images flask-app.
New MySQL env vars not taking effect after docker compose down / up
MySQL only initialises credentials from env vars on first start (when the data directory is empty). If the mysql-data volume already exists, the env vars are ignored. Fix: run docker compose down -v to delete the volume, then docker compose up -d. This wipes all database data — only appropriate in development. In production, use SQL commands to change the database/user instead of changing env vars.
docker compose ps shows service as "Exit 1" or "Restarting"
The container started but the main process crashed. Check logs immediately: docker compose logs flask-app or docker compose logs mysql. Common Flask causes: import error, missing env var, syntax error in app.py. Common MySQL causes: wrong root password format, volume permission issue. If the container restarts in a loop (restart: unless-stopped), the logs may be from the most recent start — check all: docker compose logs --no-log-prefix flask-app.
How to confirm services can reach each other by service name
Run a ping from inside a container: docker compose exec flask-app ping -c 3 mysql. If DNS resolves and packets return, the network is working. If ping is not available in the slim image, try: docker compose exec flask-app python -c "import socket; print(socket.gethostbyname('mysql'))" — if it prints an IP, Compose DNS is working. The issue is then in your app's connection logic, not Docker networking.
01 — What's next
Future scope
⟶ Things to explore from here
env_file instead of environment — move credentials out of docker-compose.yml into a .env file: env_file: .env. The .env file is excluded from git via .gitignore. This keeps secrets out of version control. The .env file in the same directory as docker-compose.yml is also automatically loaded by Compose for variable substitution in the YAML.
Nginx as reverse proxy — next topic — add a third service to the compose file: nginx. Nginx terminates incoming HTTP requests and proxies them to flask-app internally. Users hit port 80 (or 443 with HTTPS) on the host; flask-app port 5000 is no longer mapped directly to the host. Nginx handles TLS, compression, and static file serving.
Connection retry in Flask app — MySQL can take 10–30 seconds to be ready even with healthchecks. A robust app retries the DB connection on startup with exponential backoff instead of crashing. Use a loop with time.sleep or a library like tenacity. This makes the app resilient to transient startup race conditions.
Docker Compose profiles — define different service sets for different environments. A dev profile adds a bind-mounted source directory for live reloading; a prod profile uses the built image with no bind mounts. Start with: docker compose --profile dev up -d.
Multiple compose files — use a base docker-compose.yml plus an override file: docker-compose.override.yml for dev and docker-compose.prod.yml for production. Compose merges them: docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d. Keeps common config in one place while allowing per-environment overrides.
Database initialisation scripts — place .sql files in a directory and bind-mount it to /docker-entrypoint-initdb.d/ in the MySQL container. MySQL runs these scripts on first start (when the data directory is empty) — useful for creating tables, inserting seed data, or setting up grants automatically.
