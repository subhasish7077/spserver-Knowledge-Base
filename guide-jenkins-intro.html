What is Jenkins

Jenkins is an open source automation server — the industry-standard tool for CI/CD. This guide covers what it is, why it exists, how it fits into a DevOps pipeline, and how it runs on spserver behind Nginx.

01 — What & Why
What Jenkins is

Jenkins is an open source automation server written in Java. Its job is to automate repetitive tasks in the software development lifecycle — things that humans would otherwise do manually every time: running tests, building artifacts, packaging Docker images, deploying to servers, sending notifications.

It was originally forked from Hudson in 2011 and has grown into the most widely used CI/CD tool in the industry. It runs on your own infrastructure — not in the cloud — which is one of its key advantages over hosted solutions.

Without Jenkins — manual workflow
Developer pushes code to GitHub. Someone manually SSHs into the server, pulls the latest code, runs tests, builds the Docker image, restarts the container. Every deployment is a manual process. Steps get skipped. Humans make mistakes. Deployments only happen when someone has time.
With Jenkins — automated pipeline
Developer pushes code to GitHub. Jenkins detects the push via webhook. It automatically: pulls the code, runs tests, builds the Docker image, restarts the Flask container — in under two minutes, with no human involvement. Every push triggers the same steps, in the same order, every time.
ℹ Why Jenkins specifically? Jenkins is the industry standard — it is used in enterprise environments including companies like Zscaler. Learning Jenkins on spserver means the skills transfer directly to production environments. GitHub Actions is hosted by GitHub and simpler to start with, but Jenkins is self-hosted, has no usage limits, and is significantly more configurable. The comparison is in the last section of this tab.
02 — CI/CD
CI/CD — what it actually means

CI/CD stands for Continuous Integration / Continuous Delivery (or Deployment). It is a practice, not a tool. Jenkins is the tool that implements the practice.

TermWhat it meansWhat Jenkins does for it
Continuous Integration (CI) Developers merge code frequently — multiple times per day. Every merge is automatically tested to catch integration problems early. Jenkins triggers on every push to GitHub. Runs the test suite automatically. Reports pass or fail back to the developer and GitHub PR.
Continuous Delivery (CD) The codebase is always in a deployable state. After CI passes, the build artifact (Docker image, binary, package) is automatically prepared and staged. Jenkins builds the Docker image after tests pass. Pushes it to a registry or stages it for deployment. The deploy step may still require manual approval.
Continuous Deployment Every code change that passes CI is automatically deployed to production — no manual step. The pipeline goes all the way from push to live. Jenkins restarts the production container automatically after a successful build. No human involvement from push to deploy.

On spserver, we target Continuous Deployment for the Flask app — a push to the main branch triggers Jenkins, which builds the image, runs tests, and restarts the container. The whole process completes in under two minutes.

ℹ Why CI/CD matters beyond automation: The biggest benefit of CI/CD is not speed — it is feedback time. Without CI, a bug introduced on Monday might not be found until Thursday when someone manually tests. With CI, the same bug is caught in two minutes. The earlier a bug is caught, the cheaper it is to fix.

Jenkins is responsible for the middle of this chain — everything from detecting the code push to handing off the running container. The developer owns the code on the left. Nginx, users, and monitoring own the right. Jenkins owns the automation in between.

Each step in the pipeline is defined in a Jenkinsfile — a text file checked into the repo alongside the code. This means the pipeline itself is versioned and code-reviewed, not configured through a UI that only one person has access to.

01 — Master / Agent model
How Jenkins distributes work

Jenkins uses a Controller / Agent architecture (historically called Master / Agent). The Controller is the central Jenkins server — it manages configuration, schedules jobs, serves the web UI, and coordinates agents. Agents are separate machines or processes that actually run the build jobs.

Controller (Master) Serves the web UI. Stores job definitions, build history, plugins, and credentials. Schedules and delegates builds to agents. Runs in Docker as the Jenkins container. All persistent data in the jenkins_home volume.
Built-in node An agent built into the Controller. Runs jobs on the same machine as the Controller. Used on spserver for Phase 3 — the Controller runs jobs locally. Sufficient for a single-server setup.
Remote agents Separate machines connected to the Controller via SSH or JNLP. Each agent can have different OS, tools, or capabilities. Not used in Phase 3 — introduced when multiple build environments are needed (e.g. Linux + Windows builds).
ℹ In production Jenkins setups at companies, the Controller never runs builds directly — it only manages and delegates. Builds run on dedicated agents (often ephemeral Docker containers or cloud VMs that spin up for a build and terminate afterwards). For spserver Phase 3, the built-in node is sufficient — we're running one service on one server.

  --name jenkins \

  -p 50000:50000 \                  # JNLP port for agent connections

  -v /var/run/docker.sock:/var/run/docker.sock \  # allows Jenkins to run Docker commands
  jenkins/jenkins:lts-jdk17

# jenkins/jenkins:lts-jdk17 — the official LTS (Long Term Support) image
# LTS releases are stable and supported for ~12 months
# jdk17 — ships with Java 17, required for modern Jenkins and plugins
⚠ The jenkins_home volume is critical. All Jenkins configuration — jobs, pipelines, credentials, plugin settings, build history — lives in /var/jenkins_home inside the container. Without the volume mount, every container restart wipes the entire Jenkins setup. The volume must always be present. Never run Jenkins without it.

The Docker socket mount (/var/run/docker.sock) deserves special attention:

Why mount the Docker socket?
Mounting /var/run/docker.sock lets the Jenkins container run Docker commands on the host. This is how Jenkins builds Docker images and restarts the Flask container as part of the CI/CD pipeline — it calls docker build and docker compose up from inside the Jenkins container, which execute on the host Docker daemon.
Security implication
Mounting the Docker socket gives the Jenkins container root-equivalent access to the host. Any pipeline that can run code inside Jenkins can run arbitrary Docker commands on the host machine. This is acceptable on a personal lab server — in production, use a dedicated agent with carefully scoped permissions instead.
03 — Plugin system
Why Jenkins uses a plugin architecture

Jenkins core is deliberately lean — it handles scheduling, the UI, and agent communication. Almost everything else is a plugin: Git integration, GitHub webhooks, Docker support, Slack notifications, credentials management, pipeline syntax. There are over 1,800 community-maintained plugins.

This design means Jenkins can integrate with any tool in the DevOps ecosystem without the core becoming bloated. You install only what you need. Plugin updates are independent of Jenkins core updates.

Git plugin
Lets Jenkins clone and poll Git repositories. Required for pulling code from GitHub before a build.
GitHub plugin
Integrates with GitHub API — receives webhooks, updates commit status (✓ or ✗), links builds to PRs.
Pipeline plugin
Enables Jenkinsfile-based pipelines — the core of modern Jenkins. Installed by default with "suggested plugins".
Docker Pipeline
Provides docker.build() and docker.image() steps for use inside Jenkinsfiles. Simplifies Docker operations in pipelines.
Credentials plugin
Stores secrets (SSH keys, passwords, tokens) securely in Jenkins. Referenced by ID in Jenkinsfiles — never hardcoded.
Blue Ocean
A modern visual UI for Jenkins pipelines — shows stage-by-stage progress, logs, and test results. Optional but improves readability.
ℹ During Jenkins setup, the "Install suggested plugins" option installs a curated set of commonly needed plugins — Pipeline, Git, GitHub, Credentials, and others. This is the right choice for a first install. Additional plugins can be added later via Manage Jenkins → Plugins.

Jenkins UI — the main screens

Jenkins is accessed at https://jenkins.dustyserver.co.in. The UI is structured around a left sidebar for navigation, a main content area for jobs and builds, and a top bar for global actions.

Credentials
Manage Jenkins → Credentials
Store secrets securely — GitHub personal access tokens, SSH private keys, Docker Hub passwords. Secrets are referenced in Jenkinsfiles by a credential ID — the actual value is never visible in logs or pipeline code.
Plugins

Install, update, and remove Jenkins plugins. "Available plugins" tab searches the plugin marketplace. "Installed plugins" shows what's active. Restart Jenkins after installing plugins to activate them.
Nodes
Manage Jenkins → Nodes
View and manage build agents (nodes). "Built-In Node" is the Controller itself. Shows executor count, disk space, and connectivity status. Phase 3 uses only the built-in node.
02 — Jobs & builds
Jobs, builds, and where to find the output

In Jenkins, a Job (also called a Project) is the configuration of an automated task — what to do, when to do it, and with what parameters. A Build is a single execution of a Job — one run of the pipeline with a specific result.

TermWhat it isWhere to find it
Job / Project The definition of a pipeline — source code location, trigger, stages to run. Persistent configuration stored in jenkins_home. Dashboard → click the job name. Left sidebar shows configuration, build history, and workspace.
Build One execution of the Job. Has a build number (#1, #2, #3...), a result (SUCCESS, FAILURE, ABORTED), and a duration. Job page → Build History panel on the left. Click a build number to see its details.
Console Output The full log of everything that ran during a build — every command executed, its output, and the final result. The most important debugging tool in Jenkins. Build page → Console Output in the left sidebar. Or click the build number in the history panel.
Workspace The directory on the agent where Jenkins checks out the code and runs the build. Cleaned between builds optionally. Job page → Workspace in the left sidebar. Shows the files on disk from the last build.
Jenkinsfile A text file in the code repository that defines the pipeline stages and steps. Checked in with the code — versioned alongside it. In the GitHub repo root. Jenkins reads it automatically for Pipeline and Multibranch Pipeline jobs.
Build status indicators

Live CPU and memory usage for the Jenkins container — useful during heavy builds

01 — What went wrong
Blockers hit while setting up Jenkins on spserver
⚠ Blocker 1
Jenkins config and jobs lost after container restart — no volume mounted
Started Jenkins with docker run jenkins/jenkins:lts without a volume mount. Set up several jobs and installed plugins. After restarting the container (docker restart jenkins), all jobs, plugins, and configuration were gone — reverted to a fresh install with the initial admin password prompt.
✓ Fix
Always mount a named volume at /var/jenkins_home
Stopped the container, removed it, and recreated with -v jenkins_home:/var/jenkins_home. Went through initial setup again (this time with the volume). After that, restarting or even removing and recreating the container leaves all Jenkins data intact — the volume persists independently of the container. The initial admin password is also stored in the volume: docker exec jenkins cat /var/jenkins_home/secrets/initialAdminPassword.
⚠ Blocker 2
Jenkins accessible at localhost:8090 but not at jenkins.dustyserver.co.in
Jenkins was running and reachable from spserver itself at http://localhost:8090, but accessing https://jenkins.dustyserver.co.in from a browser showed Nginx's default page or a connection error — not Jenkins.
✓ Fix
DNS A record, Nginx server block, and Certbot all need to be in place
Three things were needed: (1) DNS — jenkins.dustyserver.co.in A record at the registrar pointing to spserver's public IP. (2) Nginx server block — /etc/nginx/sites-available/jenkins with server_name jenkins.dustyserver.co.in and proxy_pass http://127.0.0.1:8090, symlinked into sites-enabled/. (3) Certbot — sudo certbot --nginx -d jenkins.dustyserver.co.in for HTTPS. All three were covered in the Nginx Reverse Proxy guide. After all three were in place, the domain worked correctly.
⚠ Blocker 3
Initial admin password — where to find it
On first access to the Jenkins UI, the setup wizard asks for an "Initial Admin Password". The UI shows it is in /var/jenkins_home/secrets/initialAdminPassword — but this path is inside the container, not on the host.
✓ Fix
docker exec to read the file from inside the container
Run: docker exec jenkins cat /var/jenkins_home/secrets/initialAdminPassword. This prints the password directly to the terminal — paste it into the Jenkins UI setup wizard. It is a 32-character hexadecimal string. After completing setup and creating an admin user, this file is no longer needed — but it remains in the volume for reference.

This guide covered what Jenkins is and how it runs on spserver. The remaining topics in Phase 3 go deeper into each layer — from architecture to writing actual pipelines to managing secrets correctly.

Push to GitHub → Jenkins webhook trigger → pull code → run pytest → docker build → docker compose up -d --build → Flask container restarted with new code — all in under two minutes, fully automated. Phase 3 builds the knowledge needed to write and debug this pipeline.
