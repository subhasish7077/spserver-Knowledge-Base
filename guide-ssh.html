SSH Server

Setting up encrypted remote terminal access — the single most important service on any headless server. Once SSH is running, you never need a physical keyboard or monitor again.

01 — Concept
What is SSH

SSH (Secure Shell) is a network protocol that lets you open an encrypted terminal session on a remote machine. Once connected, you get a full interactive shell — exactly as if you were sitting in front of the machine with a keyboard. Every keystroke, every command output is encrypted in transit.

Without SSH, controlling a headless server requires a physically connected monitor and keyboard. SSH eliminates that requirement entirely — you control spserver from your Windows PC, phone, or any device, over any network.

ℹ SSH is not just for servers. It's used by developers, sysadmins, and DevOps engineers constantly — to deploy code, manage cloud instances, configure databases, debug issues. Learning SSH deeply is one of the most high-value skills in the field.
02 — How it works
The SSH connection flow

SSH uses asymmetric cryptography to establish a secure channel before any data is exchanged. Here is what happens when you run ssh sipu@192.168.31.150:

Your PC spserver (sshd) │ │ │── TCP connection to port 22 ──────────────►│ │ │ │◄── Server sends its public key (host key) ──│ │ │ │── Client verifies host key ─────────────────►│ │ (checks ~/.ssh/known_hosts on your PC) │ │ │ │◄──── Encrypted session established ─────────►│ │ │ │── Password or key authentication ───────────►│ │ │ │◄──── Full encrypted terminal session ───────►│

The host key is the server's identity — a unique cryptographic fingerprint. The first time you connect, SSH asks you to verify and save it. Subsequent connections compare against the saved fingerprint — if it changes, SSH warns you (this is the protection against man-in-the-middle attacks).

03 — Authentication
Password vs SSH key authentication

SSH supports two main authentication methods. We currently use password auth — SSH keys are the recommended next step.

Password authentication — current
You type your Linux user password. Simple to set up — no extra steps. The downside: automated bots constantly scan the internet for port 22 and try common passwords (brute force attacks). Mitigated by UFW blocking port 22 externally, and Tailscale for remote access.
SSH key authentication — recommended
A cryptographic key pair. Your PC holds a private key (secret, never shared). The server stores your public key (safe to share). No password to guess or brute force. Also more convenient — no typing on trusted machines. See the Config tab for setup.

Three ways to SSH into spserver

Anywhere via Tailscale

05 — Concept
What is sshd — the SSH daemon

When you install openssh-server, it installs a program called sshd (SSH daemon). The d stands for daemon — a background process that runs continuously. sshd listens on port 22, waiting for incoming connections. When you SSH in, sshd handles the handshake, verifies your identity, and opens a session.

sshd runs on spserver ← started by systemd at boot └── listens on port 22 └── waits for connections └── when connection arrives: ├── performs key exchange ├── verifies identity (password or key) ├── opens a shell session └── keeps session alive until you exit

sshd reads its configuration from /etc/ssh/sshd_config. Any changes to that file require restarting sshd to take effect.

01 — Install
Install and start the SSH server
1
Install openssh-server
▼

The openssh-server package installs sshd — the server-side daemon. The client-side tools (the ssh command you type) come with Ubuntu by default. You only need to install the server package on spserver.

bash — run on spserver
sudo apt install openssh-server -y
2
Enable and start the service
▼

enable makes sshd start automatically on every boot. start starts it right now without rebooting. You need both — enable alone means it only starts next boot, start alone means it stops after the next reboot.

bash
# Auto-start on every boot
sudo systemctl enable ssh

# Start right now
sudo systemctl start ssh

# Verify it is running — look for: Active: active (running)
sudo systemctl status ssh
3
Confirm SSH is listening on port 22
▼

This confirms sshd is actually listening for connections. If this shows nothing, sshd is not running or is listening on a different port.

bash
sudo ss -tlnp | grep ssh
# Expected output: something like
# LISTEN 0  128  0.0.0.0:22  0.0.0.0:*  users:(("sshd",pid=...))
# 0.0.0.0:22 means it's listening on ALL interfaces on port 22
02 — Firewall
Open port 22 in UFW

UFW (Uncomplicated Firewall) blocks all incoming connections by default. Even though sshd is running, connections will be refused until UFW is told to allow port 22.

✗ Critical order warning: Always open port 22 in UFW BEFORE enabling UFW. If you enable UFW with no rules and you are connected via SSH, your session will be immediately cut off and you'll be locked out.
4
Allow SSH through UFW
▼
bash
# Open port 22 for SSH BEFORE enabling UFW
sudo ufw allow 22/tcp

# Now enable the firewall
sudo ufw enable

# Verify the rule is in place
sudo ufw status
# Should show: 22/tcp  ALLOW  Anywhere
⚠ UFW only controls traffic at the server level. Port 22 is still not reachable from the public internet because Airtel CG-NAT blocks all incoming connections. Tailscale is how you reach it remotely.
03 — Connect
Test the connection
5
SSH from your Windows PC (local network)
▼

Windows 10/11 has a built-in SSH client. Open Windows Terminal, PowerShell, or Command Prompt and connect directly — no third-party software needed.

Windows Terminal / PowerShell
# Basic connection — enter password when prompted
ssh sipu@192.168.31.150

# Verbose output — shows each step of the handshake
# Useful for debugging connection issues
ssh -v sipu@192.168.31.150

# Specify port explicitly (useful if port is not 22)
ssh -p 22 sipu@192.168.31.150

First connection will show a fingerprint warning — this is normal. Type yes to save the server's fingerprint to ~/.ssh/known_hosts on your Windows PC. Subsequent connections skip this step.

6
SSH from anywhere via Tailscale
▼

Once Tailscale is installed on both devices (see Tailscale guide), you can SSH from any network using the Tailscale IP. Same command, different IP address.

bash / PowerShell — from anywhere
# Get spserver's Tailscale IP first (run on spserver)
tailscale ip

# SSH using Tailscale IP from any device anywhere
ssh sipu@100.x.x.x   # replace with actual Tailscale IP
04 — Convenience
SSH config file on Windows — shortcuts

Instead of typing the full command every time, create an SSH config file on your Windows PC. This lets you type a short alias instead of the full IP address.

C:\Users\subha\.ssh\config — create this file
# Local network connection
Host spserver
    HostName 192.168.31.150
    User     sipu
    Port     22

# Remote connection via Tailscale
Host spserver-remote
    HostName 100.x.x.x    # your Tailscale IP
    User     sipu
    Port     22

After saving, connect with just:

Windows Terminal
ssh spserver           # local — instead of ssh sipu@192.168.31.150
ssh spserver-remote    # remote via Tailscale
01 — Server config
sshd_config — server-side configuration

The SSH server's main config file lives at /etc/ssh/sshd_config. The defaults are secure enough for getting started — you should only change these settings when you have a specific reason. Always restart sshd after any change.

⚠ Never lock yourself out of SSH while editing this file. Keep your current session open, make changes in a second session, test the new session works, then close the original. If you close the only session and have a broken config — you need physical access.
SettingWhat it controls
Port 22Which port sshd listens on. Default is 22. Some guides suggest changing to a high port (e.g. 2222) to reduce bot noise — minor security benefit but more inconvenient. We keep 22.
PermitRootLogin noImportant. Disables direct login as root. Always leave this as no — login as sipu and use sudo instead. Default on Ubuntu is prohibit-password which also disallows root password login.
PasswordAuthentication yesAllows password-based login. Currently set to yes. When SSH keys are set up, change to no to disable password login entirely — much more secure.
PubkeyAuthentication yesAllows key-based authentication. Default yes — keep this enabled.
AuthorizedKeysFileWhere to look for authorised public keys. Default: ~/.ssh/authorized_keys. No need to change.
MaxAuthTries 6Max password attempts before connection is dropped. Reduce to 3 for slightly better security.
ClientAliveInterval 300Send keepalive packets every 300 seconds to prevent idle sessions from dropping.
X11Forwarding noDisable X11 GUI forwarding — not needed on a server.
bash — after editing sshd_config
# Test config for syntax errors before restarting
sudo sshd -t

# Restart sshd to apply changes
sudo systemctl restart ssh

SSH client config file

The SSH client config on your Windows PC stores connection shortcuts and settings per host. This is separate from sshd_config — it lives on your Windows machine, not on spserver.

03 — SSH keys (next step)
Setting up SSH key authentication

SSH keys are more secure and more convenient than passwords. A key pair consists of a private key (stays on your PC, never shared) and a public key (copied to spserver). The server verifies you hold the private key without you ever sending it.

1
Generate key pair on Windows PC
▼
Windows Terminal / PowerShell
# Generate an Ed25519 key pair (modern, recommended)
ssh-keygen -t ed25519 -C "sipu@spserver"

# Accept default location: C:\Users\subha\.ssh\id_ed25519
# Set a passphrase (optional but recommended for extra security)

# Two files are created:
# id_ed25519     ← private key (NEVER share this)
# id_ed25519.pub ← public key (copy this to server)
2
Copy public key to spserver
▼
Windows Terminal
# Windows ssh-copy-id equivalent
type $env:USERPROFILE\.ssh\id_ed25519.pub | ssh sipu@192.168.31.150 "mkdir -p ~/.ssh && cat >> ~/.ssh/authorized_keys"

Or manually: copy the contents of id_ed25519.pub, SSH into spserver, and paste into ~/.ssh/authorized_keys.

3
Test and disable password login
▼
bash — test key auth works first
# From Windows — should login without password prompt
ssh sipu@192.168.31.150

# Only after confirming key auth works — disable password login on server
sudo nano /etc/ssh/sshd_config
# Change: PasswordAuthentication yes → PasswordAuthentication no
sudo systemctl restart ssh
✗ Never disable password auth without first confirming key auth works. If keys don't work and you disable passwords — you are permanently locked out (no physical access = bricked server).

01 — What went wrong
Blockers hit during setup
⚠ Blocker 1
SSH timed out from public IP — port forwarding set but not working
SSH worked fine on the local network. Tried connecting from a mobile hotspot using the public IP — got "Connection timed out". Port forwarding had been configured on the backup router (192.168.31.1) but the connection still timed out from outside.
✓ Root cause
Double NAT — port forwarding only set on one router
The network has two routers stacked: Airtel modem (192.168.1.1) and backup router (192.168.31.1). Traffic from the internet hits the Airtel modem first — which had no forwarding rule — and dropped the connection before it reached the backup router. Port forwarding must be set on both. This led to discovering CG-NAT (see next blocker).
⚠ Blocker 2
CG-NAT confirmed — port forwarding impossible regardless of configuration
After setting forwarding on both routers, SSH still failed from outside. Checked the Airtel modem WAN IP — showed 100.220.x.x. The 100.x.x.x range is IANA-reserved for Carrier Grade NAT. Airtel does not give spserver a real public IP — hundreds of customers share one real IP through Airtel's own NAT layer.
✓ Fix
Tailscale for remote SSH — bypasses CG-NAT entirely
Installed Tailscale on both spserver and the Windows PC. Both devices reach out to Tailscale's coordination servers — outgoing connections always work behind CG-NAT. They establish a direct encrypted peer-to-peer tunnel. ssh sipu@100.x.x.x (Tailscale IP) now works from any network worldwide. Port forwarding is no longer attempted. See the Tailscale guide for setup.
01 — Issues and fixes
Common SSH problems
ssh: connect to host 192.168.31.150 port 22: Connection refused
sshd is not running on the server. Solutions: (1) If you have physical access — start it: sudo systemctl start ssh. (2) Check if it ever ran: sudo systemctl status ssh. (3) Check UFW: sudo ufw status — make sure port 22 is allowed. Connection refused means the port is closed or sshd is down. Connection timed out means the packet never reached the server (firewall or network issue).
ssh: connect to host 192.168.31.150 port 22: Connection timed out
The connection attempt is not reaching sshd. Causes: (1) UFW is blocking port 22 — check with sudo ufw status. (2) Wrong IP address — verify spserver's IP with ip addr show ethusb0. (3) Connecting from outside home network to local IP — use Tailscale IP instead. (4) Network cable unplugged or interface down.
Permission denied (publickey) — cannot login with password
PasswordAuthentication has been set to no in sshd_config. If you don't have key auth set up, you are locked out. Recover with physical access: open /etc/ssh/sshd_config, change PasswordAuthentication no to yes, restart sshd. Or access via Tailscale if it was set up beforehand.
WARNING: REMOTE HOST IDENTIFICATION HAS CHANGED
The server's SSH fingerprint has changed since you last connected. This happens after an OS reinstall or if the sshd host keys were regenerated. It is also the warning SSH shows for potential man-in-the-middle attacks. If you know you reinstalled the OS — safe to clear: ssh-keygen -R 192.168.31.150 on your Windows PC. Then reconnect and accept the new fingerprint.
SSH session drops / disconnects after inactivity
The connection times out due to inactivity. Fix on the server side — add to /etc/ssh/sshd_config: ClientAliveInterval 60 and ClientAliveCountMax 3. Or fix on the client side — add to ~/.ssh/config on your Windows PC: ServerAliveInterval 60 under the Host block.
Permission denied — ssh key not working
Key authentication failing. Check: (1) Public key is in ~/.ssh/authorized_keys on the server. (2) Permissions are correct: chmod 700 ~/.ssh and chmod 600 ~/.ssh/authorized_keys on the server. (3) PubkeyAuthentication is yes in sshd_config. (4) The right private key is being used — check with ssh -v.
Too many authentication failures
SSH is trying multiple keys automatically from your keyring and hitting the MaxAuthTries limit. Fix: specify the key explicitly: ssh -i ~/.ssh/id_ed25519 sipu@192.168.31.150. Or add IdentitiesOnly yes to the Host block in your SSH client config.
01 — What's next
Future scope
⟶ Things to explore from here
SSH key authentication — generate an Ed25519 key pair, copy the public key to spserver, disable password login. More secure and more convenient. Covered in the Config tab — do this next.
fail2ban — automatically ban IP addresses that fail SSH authentication too many times. Protects against brute force attacks. Install with sudo apt install fail2ban — works out of the box with sensible defaults.
SSH via Cloudflare Tunnel — expose SSH at ssh.dustyserver.co.in through cloudflared. Requires cloudflared on every client. Less useful than Tailscale for personal use but relevant in enterprise contexts.
SSH port knocking — hide port 22 completely. UFW blocks it by default. A sequence of connection attempts to other ports "unlocks" port 22 temporarily. Obscurity-level security — not a replacement for keys and fail2ban.
SSH jump hosts / ProxyJump — connect through one SSH server to reach another. Useful when you have multiple servers and want to use spserver as a gateway to reach them. Config: ProxyJump spserver in client config.
SSH tunnelling / port forwarding — forward local ports to remote services through SSH. Access a database or internal service on spserver as if it were running locally. Already shown in the commands tab — explore more advanced use cases.
Mosh — Mobile Shell. SSH-like but handles network interruptions gracefully. If your mobile connection drops while SSH'd in, Mosh reconnects automatically. Useful for phone access.
