UFW Firewall

Controlling which network traffic is allowed in and out of spserver. UFW is the gatekeeper — everything goes through it before reaching your services.

01 — Concept
What is UFW

UFW stands for Uncomplicated Firewall. It is a frontend — a simplified interface — for managing Linux's packet filtering system. Without a firewall, every port on your server is potentially accessible to anyone on the network. UFW lets you define exactly which traffic is allowed and blocks everything else.

UFW comes pre-installed on Ubuntu but is inactive by default. It sits there doing nothing until you explicitly enable it. This is by design — enabling it before you've opened port 22 would lock you out of SSH.

ℹ A firewall is not the same as security. It controls which ports are reachable — it does not protect against vulnerabilities in the services running on those ports. But it is a fundamental and necessary first layer of defence.
02 — Architecture
UFW vs iptables vs nftables

Linux has a built-in packet filtering system in the kernel. Over the years it has had different interfaces — netfilter, iptables, nftables. UFW sits on top of these and translates your simple commands into the correct low-level rules.

You type: sudo ufw allow 22/tcp ↓ UFW translates into iptables / nftables rules ↓ Linux kernel netfilter ← actual packet filtering happens here ↓ Incoming packet on port 22 → allowed ✓ Incoming packet on port 3306 → dropped ✗
UFW — what we use
Simple human-readable commands. sudo ufw allow 80/tcp is all it takes to open a port. Ubuntu's recommended tool. Pre-installed. Sufficient for 95% of server firewall needs.
iptables / nftables
Raw power — complex syntax, fine-grained control. Used directly when you need stateful inspection, NAT, complex routing, or custom chains. UFW generates iptables rules under the hood — you can view them with sudo iptables -L.
03 — Policies
Default UFW policies — deny in, allow out

When UFW is enabled, it applies default policies to all traffic not matched by a specific rule. The defaults on Ubuntu are sensible for a server:

DirectionDefault policyWhat it means
Incoming DENY All incoming connections are blocked unless you explicitly allow them. Anyone trying to reach a port you haven't opened gets dropped.
Outgoing ALLOW spserver can make outgoing connections freely — apt update, curl, Tailscale, cloudflared all work. No restrictions on what the server initiates.
Forwarding DENY Traffic passing through the server (not destined for it) is blocked. Relevant if spserver were acting as a router — not our use case.

This means after enabling UFW, you must explicitly open a port for each service you want accessible. Miss one and that service becomes unreachable — which is exactly the blocker we hit several times during setup (ttyd, Filebrowser).

05 — Traffic flow
How traffic flows through UFW
Incoming packet arrives on spserver: Packet → port 22 UFW checks rules: 22/tcp ALLOW ✓ → packet reaches sshd → SSH session opens Packet → port 7681 UFW checks rules: 7681/tcp ALLOW ✓ → packet reaches ttyd → browser terminal opens Packet → port 3306 UFW checks rules: no rule found → default DENY → packet dropped silently — no response sent Packet → port 9999 (random) UFW checks rules: no rule found → default DENY → packet dropped — attacker gets no info Outgoing packet from spserver (apt update, cloudflared, tailscale): UFW default outgoing: ALLOW ✓ → all outgoing connections work freely
01 — Critical warning
The order of operations matters enormously
✗ Never run sudo ufw enable before opening port 22. The moment UFW is enabled with default deny-incoming, your current SSH session continues but new SSH connections are blocked. If you disconnect — whether intentionally or due to a network blip — you cannot SSH back in. You need physical access to recover.

The correct sequence every time:

bash — always in this order
# 1. Open port 22 FIRST — before enabling
sudo ufw allow 22/tcp

# 2. Now enable UFW safely
sudo ufw enable

# 3. Verify SSH rule is active
sudo ufw status
02 — Enable
Enable UFW for the first time
1
Check current UFW status
▼

Before enabling, check if UFW is already active and what rules exist. On a fresh Ubuntu Server install, UFW is installed but inactive.

bash
sudo ufw status verbose
# Fresh install shows: Status: inactive
# Already enabled shows: Status: active + list of rules
2
Open SSH port then enable
▼

Allow SSH first — always. Then enable. UFW will warn you that enabling may disrupt existing SSH connections — type y to confirm. Your current session stays alive because UFW doesn't kill existing connections, only blocks new ones.

bash
# Step 1: Open SSH port
sudo ufw allow 22/tcp

# Step 2: Enable UFW (type y when prompted)
sudo ufw enable

# Step 3: Verify
sudo ufw status verbose
03 — Open ports
Opening ports for each service

Every service added to spserver needed its port opened in UFW. Here are all the commands used — in the order services were set up:

3
SSH, Nginx, ttyd, Filebrowser, Samba
▼
bash — all service ports
# SSH — first thing always
sudo ufw allow 22/tcp

# Nginx — 'Nginx Full' opens both port 80 (HTTP) and 443 (HTTPS)
# UFW has pre-built app profiles for common services
sudo ufw allow 'Nginx Full'

# ttyd browser terminal
sudo ufw allow 7681/tcp

# Filebrowser
sudo ufw allow 8080/tcp

# Samba — 'samba' is a pre-built UFW app profile
# Opens ports 139 (NetBIOS) and 445 (SMB)
sudo ufw allow samba
ℹ UFW app profiles are pre-defined rule sets for common services. Nginx Full, samba, OpenSSH are examples. List all available profiles with sudo ufw app list.
4
Docker ports — what NOT to do
▼

Important Docker caveat: Docker bypasses UFW rules by default. When you use ports: in docker-compose.yml, Docker directly adds iptables rules — skipping UFW entirely. This means a Docker container with a mapped port is accessible from the network regardless of UFW rules.

bash — Flask Docker port
# Flask Docker container maps port 5000 to host
# We still open it in UFW for clarity and documentation
sudo ufw allow 5000/tcp

# But be aware: even WITHOUT this UFW rule,
# Docker's port mapping makes it reachable on the local network
# UFW does NOT fully control Docker-mapped ports
⚠ To properly control Docker ports with UFW, edit /etc/docker/daemon.json and set "iptables": false. Then manage all routing through Nginx reverse proxy instead. This is the correct production approach — relevant when adding Flask + MySQL later.

02 — Advanced
Advanced UFW rules
sudo ufw allow from 192.168.31.0/24
Allow all traffic from the local network subnet — useful for Samba, NFS
sudo ufw allow from 192.168.31.7 to any port 22
Allow SSH only from your specific PC's IP — whitelist approach
sudo ufw allow from 100.64.0.0/10
Allow all traffic from Tailscale IP range (100.64.x.x to 100.127.x.x)
sudo ufw limit 22/tcp
Rate-limit SSH — blocks IPs that attempt more than 6 connections in 30 seconds. Basic brute-force protection.
sudo ufw default deny incoming
Set default policy for incoming traffic to deny (already default)
sudo ufw default allow outgoing
Set default policy for outgoing traffic to allow (already default)
sudo ufw logging on
Enable UFW logging — log blocked and allowed connections to syslog
sudo ufw logging medium
Set log verbosity: low / medium / high / full
sudo cat /var/log/ufw.log
View UFW log — shows blocked connection attempts with IPs and ports

# View all iptables rules UFW created

01 — What went wrong
Blockers hit because of UFW

UFW was the cause of several "service not reachable" issues throughout the setup. The pattern was always the same — start a service, test in browser, get nothing. Check UFW. Open the port. Works.

⚠ Blocker 1
ttyd running but browser shows nothing — port 7681 blocked
ttyd was installed and running — sudo systemctl status ttyd showed active. Port was listening — sudo ss -tlnp | grep 7681 showed it. But opening http://192.168.31.150:7681 in the browser showed "can't reach this page" with no response at all.
✓ Fix
sudo ufw allow 7681/tcp
UFW was blocking port 7681. The service was running fine — the firewall was dropping the packets before they reached ttyd. After opening the port, the browser terminal was immediately accessible. Lesson: whenever a service is running but unreachable from the browser, check UFW first.
⚠ Blocker 2
Filebrowser not reachable — port 8080 blocked
Same pattern as ttyd. Filebrowser was running and bound to 0.0.0.0:8080 — confirmed with curl http://localhost:8080 returning HTML. But opening the URL from the browser on the Windows PC showed nothing.
✓ Fix
sudo ufw allow 8080/tcp
Same cause — UFW blocking port 8080. After allowing the port, Filebrowser was immediately accessible from the browser. The key diagnostic here was using curl http://localhost:8080 from the server itself — if that works but the browser doesn't, UFW is almost certainly the issue.
ℹ Diagnostic pattern for "service running but not reachable": (1) Check service status — sudo systemctl status SERVICE. (2) Check what port it's on — sudo ss -tlnp. (3) Test from server itself — curl http://localhost:PORT. (4) If server curl works but browser doesn't — it's UFW. Open the port.
01 — Issues and fixes
Common UFW problems
Locked out of SSH after enabling UFW
You enabled UFW before allowing port 22. Your current session may still be alive but reconnecting is impossible. Immediate options: (1) If Tailscale was already set up — SSH in via Tailscale IP (ssh sipu@100.x.x.x) — Tailscale operates at a different layer and UFW doesn't block it. Once in, run sudo ufw allow 22/tcp. (2) Physical access — connect keyboard and monitor, login locally, run sudo ufw disable or sudo ufw allow 22/tcp. Prevention: always open port 22 before enabling.
Service is running but completely unreachable from network
UFW is blocking the port. Diagnostic steps in order: (1) sudo systemctl status SERVICE — confirm service is running. (2) sudo ss -tlnp — confirm which port it's listening on. (3) curl http://localhost:PORT — test from server itself. If this works, UFW is the issue. (4) sudo ufw status — check if the port has a rule. (5) sudo ufw allow PORT/tcp — open the port and retest.
UFW shows rule exists but port is still blocked
Possible causes: (1) Wrong protocol — tried allow 80 (both TCP+UDP) but service only uses TCP. (2) Conflicting DENY rule takes precedence — sudo ufw status numbered and check rule order. (3) Docker bypassed UFW — Docker container port mappings use iptables directly. (4) Service is binding to 127.0.0.1 not 0.0.0.0 — UFW can't help if the service refuses external connections. Check with sudo ss -tlnp.
Accidentally deleted the wrong UFW rule
Re-add it immediately: sudo ufw allow PORT/tcp. If you deleted the SSH rule and are still connected — add it back before disconnecting. If you've lost SSH and have Tailscale — connect via Tailscale and re-add the rule. UFW persists all changes immediately — no restart needed after adding rules.
ufw status shows "inactive" after reboot
This should not happen — UFW persists its enabled state across reboots. If it does: run sudo ufw enable. If it keeps happening, check if another firewall tool (iptables-persistent, firewalld) is interfering. Also verify: sudo systemctl status ufw — if the UFW service itself is disabled, enable it: sudo systemctl enable ufw.
Port accessible on local network but not from outside
This is expected and correct — UFW allows local network connections but Airtel CG-NAT blocks all incoming from the internet. This is not a UFW problem. External access to services goes through Cloudflare Tunnel (for public services) or Tailscale (for private SSH). UFW only controls what arrives at the server — CG-NAT controls what reaches the server from the internet.
01 — What's next
Future scope
⟶ Things to explore from here
UFW rate limiting for SSH — sudo ufw limit 22/tcp instead of allow. Automatically blocks IPs attempting more than 6 connections in 30 seconds. Simple but effective brute-force protection. Change the current SSH rule to use limit.
IP whitelisting for SSH — restrict SSH to specific IPs only: sudo ufw allow from 192.168.31.7 to any port 22. Then delete the general 22/tcp rule. Strongest local SSH protection — only your PC can SSH in on the local network.
Docker + UFW integration — Docker bypasses UFW by default. For proper control, set "iptables": false in /etc/docker/daemon.json and route all container traffic through Nginx reverse proxy. Required when running databases in Docker that should not be externally accessible.
UFW logging and monitoring — enable sudo ufw logging medium and pipe logs to a monitoring tool. Visualise blocked connection attempts over time using Prometheus + Grafana. See how many bots are scanning your server.
fail2ban with UFW — fail2ban watches log files and dynamically adds UFW deny rules for IPs that fail authentication too many times. Combine with UFW rate limiting for layered SSH protection.
nftables directly — UFW's backend on Ubuntu 22.04+ is nftables. Learning nftables directly gives you more power — stateful rules, sets, maps, verdict chains. Relevant for complex scenarios like running a VPN gateway or multi-homed server.
