Flask App in Docker

Structuring and wiring up the spserver-flask app for Docker — app.py, requirements.txt, MySQL connection via service name, and testing the full stack from a browser on the LAN.

01 — What & Why
What this guide covers

The Dockerfile and Compose guides covered the container infrastructure. This guide covers the Flask application itself — how to write it so it runs correctly inside Docker, how to connect it to MySQL using Compose's internal DNS, and how to read configuration from environment variables instead of hardcoding it.

The app is intentionally simple — a single route that reads a row from MySQL and returns JSON. The goal here is not Flask feature depth but understanding the wiring between the Flask container and the MySQL container, and the patterns (env vars, service name as hostname, bind address) that make a Flask app work correctly in a Docker environment.

Flask on the host (what we're moving away from)
Flask running directly on Ubuntu. Python installed system-wide. MySQL running as a systemd service. Connection string hardcoded with localhost. Works on this one machine, impossible to reproduce elsewhere, no isolation between services.
Flask in Docker (what we're building)
Flask runs in a container built from a Dockerfile. MySQL runs in its own container. They communicate by service name over Docker's internal network. Config comes from environment variables. The whole stack starts with one command.
02 — Project structure
File layout for the spserver-flask project

Every file in the repo has a specific role. The structure is kept flat and minimal — no unnecessary nesting.

spserver-flask/
  ├── app.py Flask application — routes, DB connection, startup config
  ├── requirements.txt pip dependencies — flask, mysql-connector-python
  ├── Dockerfile builds the flask-app image
  ├── docker-compose.yml defines flask-app + mysql services
  ├── .dockerignore excludes venv/, __pycache__, .git from build context
  ├── .gitignore excludes venv/, .env, __pycache__ from git
  └── templates/ Jinja2 HTML templates (if any)
ℹ There is no venv/ in the Docker workflow. Dependencies are installed inside the container by the Dockerfile's RUN pip install step. A local venv is only needed if you want to run the app directly on the host for development — and it must be excluded from the Docker build context via .dockerignore.
03 — Inter-container networking
How Flask reaches MySQL — Docker Compose internal DNS

When Docker Compose starts a stack, it automatically creates a bridge network shared by all services in the compose file. Each service gets a hostname equal to its service name. Docker runs an internal DNS server that resolves these names to the container's IP address.

This means Flask can connect to MySQL using the hostname mysql — the service name defined in docker-compose.yml. No IP addresses, no localhost, no port mapping needed between the two containers.

04 — Configuration
Environment variables — why and which ones

Hardcoding database credentials in app.py is a security risk — credentials end up in the git history and anyone with repo access has the database password. Environment variables keep config out of the code. The values are defined in docker-compose.yml (or a .env file) and read in Python with os.environ.get().

VariableExample valueRead by
DB_HOSTmysqlFlask app — MySQL hostname. Must be the Compose service name, not localhost.
DB_PORT3306Flask app — MySQL port. Almost always 3306.
DB_NAMEflaskdbFlask app — the database to connect to. Must match MYSQL_DATABASE in the mysql service.
DB_USERflaskuserFlask app — the MySQL user. Must match MYSQL_USER in the mysql service.
DB_PASSWORDflaskpassFlask app — the MySQL password. Must match MYSQL_PASSWORD in the mysql service.
FLASK_ENVdevelopmentFlask — enables debug mode and auto-reloader when set to development. Use production on spserver.
⚠ Use os.environ.get('DB_PASSWORD', 'fallback') in Python — not os.environ['DB_PASSWORD']. The get() form returns None (or the fallback) if the variable is missing rather than raising a KeyError that crashes the app at startup. It also makes it obvious what the expected variable names are.

02 — Application code
Step 2 — Write app.py
1
Full app.py — annotated
▼

Every decision in this file has a reason — the annotations explain what would break if you did it differently.

spserver-flask/app.py
import os
import json
import mysql.connector
from flask import Flask, jsonify

app = Flask(__name__)

# Read all DB config from environment variables
# These are set in docker-compose.yml under the flask-app service's environment block
# os.environ.get() returns None if missing — better than KeyError on startup
DB_CONFIG = {
    'host':     os.environ.get('DB_HOST',     'localhost'),
    'port':     int(os.environ.get('DB_PORT', '3306')),
    'database': os.environ.get('DB_NAME',     'flaskdb'),
    'user':     os.environ.get('DB_USER',     'flaskuser'),
    'password': os.environ.get('DB_PASSWORD', ''),
}


def get_db_connection():
    """Open and return a new MySQL connection using the global DB_CONFIG."""
    return mysql.connector.connect(**DB_CONFIG)


@app.route('/')
def index():
    # Basic health check — confirms Flask is up without touching the DB
    return jsonify({'status': 'ok', 'service': 'spserver-flask'})


@app.route('/items')
def get_items():
    # Connect to MySQL, query the items table, return results as JSON
    conn   = get_db_connection()
    cursor = conn.cursor(dictionary=True)  # dictionary=True → rows as dicts, not tuples
    cursor.execute('SELECT * FROM items')
    rows = cursor.fetchall()
    cursor.close()
    conn.close()
    return jsonify(rows)


@app.route('/health')
def health():
    # Deeper health check — confirms DB connectivity
    # Useful for Nginx upstream checks or monitoring
    try:
        conn = get_db_connection()
        conn.close()
        return jsonify({'status': 'ok', 'db': 'reachable'})
    except Exception as e:
        return jsonify({'status': 'error', 'db': str(e)}), 500


if __name__ == '__main__':
    # host='0.0.0.0' — listen on all interfaces inside the container
    # Without this, Flask binds to 127.0.0.1 (loopback only) and is
    # unreachable from outside the container even with port mapping
    app.run(host='0.0.0.0', port=5000, debug=False)
⚠ host='0.0.0.0' is critical. By default, Flask's development server binds to 127.0.0.1 — the container's loopback interface. Traffic coming in through Docker's port mapping arrives on the container's network interface, not loopback. If Flask isn't listening on 0.0.0.0, the port mapping does nothing and curl localhost:5000 from the host gets connection refused.
03 — Database setup
Step 3 — Create the MySQL table

The MySQL container initialises the flaskdb database and the flaskuser automatically on first start (from the compose env vars). But the table structure is not created automatically — you need to create it. The cleanest approach for a small app is to do it once from a MySQL shell.

2
Create the items table inside the running MySQL container
▼

The stack must already be running (docker compose up -d) before running these commands. We exec directly into the MySQL container to run the SQL — no need to expose port 3306 to the host.

bash — run from spserver-flask/
# Open a MySQL shell inside the running mysql container as flaskuser
docker compose exec mysql mysql -u flaskuser -pflaskpass flaskdb

# Inside the MySQL shell:
CREATE TABLE items (
    id   INT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    qty  INT DEFAULT 0
);

INSERT INTO items (name, qty) VALUES ('widget', 10), ('gadget', 5);

SELECT * FROM items;
# Should show both rows

EXIT;
ℹ For a more production-like approach, place init.sql in the project and bind-mount it to /docker-entrypoint-initdb.d/ in the MySQL service — the MySQL image runs scripts in that directory automatically on first start. This is covered in the Future scope tab.

Day-to-day development commands
docker compose up -d --build
Rebuild the flask-app image and restart the stack — use after any change to app.py, requirements.txt, or Dockerfile
docker compose up -d
Start the stack using the existing image — use when code hasn't changed (e.g. after a reboot)
docker compose down
Stop and remove containers. MySQL data volume is kept — data survives.
docker compose down -v
Stop, remove containers, AND delete volumes. Wipes the MySQL database — fresh start on next up.

docker compose logs -f flask-app

docker compose ps
Check both services are Up and MySQL shows (healthy)

01 — What went wrong
Blockers hit while building the Flask + MySQL app
⚠ Blocker 1
curl localhost:5000 — connection refused — Flask not reachable from host
Stack started, docker compose ps showed flask-app as Up, but curl localhost:5000 returned "Connection refused". Flask logs showed it was running — "Running on http://127.0.0.1:5000". Port mapping was correct in the compose file.
✓ Fix
Flask must bind to 0.0.0.0, not 127.0.0.1
Flask's default bind address is 127.0.0.1 — the container's loopback interface. Docker's port mapping sends traffic to the container's network interface (not loopback), so it never reaches Flask. The fix is app.run(host='0.0.0.0', port=5000). After rebuilding and restarting, Flask logged "Running on http://0.0.0.0:5000" and the endpoint was reachable from the host.
⚠ Blocker 2
Flask crashed on first boot — "Can't connect to MySQL" — even with depends_on
After adding the DB connection to app.py and starting the stack, Flask crashed immediately. docker compose logs flask-app showed: mysql.connector.errors.DatabaseError: 2003 (HY000): Can't connect to MySQL server on 'mysql'. The MySQL container was running but MySQL wasn't ready yet — it takes 20–30 seconds to initialise on first start.
✓ Fix
Two-part fix: healthcheck in compose + restart flask-app manually once
Long-term fix: the Compose healthcheck + condition: service_healthy pattern (covered in the Compose guide) prevents Flask from starting before MySQL is ready. Short-term fix during first-time setup: once docker compose ps showed MySQL as healthy, ran docker compose restart flask-app. Flask started successfully because MySQL was fully up by then. The healthcheck is the correct solution — the manual restart is just a workaround while the healthcheck is being set up.
⚠ Blocker 3
ModuleNotFoundError: No module named 'mysql'
Flask started but every request to /items crashed with ModuleNotFoundError: No module named 'mysql'. The module was installed in the local venv but not inside the container.
✓ Fix
mysql-connector-python was missing from requirements.txt
The local venv had mysql-connector-python installed manually via pip but it was never added to requirements.txt. The Dockerfile's pip install -r requirements.txt only installs what's in the file — so the container image was built without it. Fix: add mysql-connector-python to requirements.txt, then docker compose up -d --build to rebuild the image. The module was then available inside the container.
01 — Issues and fixes
Common Flask-in-Docker problems and fixes
Connection refused on curl localhost:5000
Check in order: (1) Is flask-app shown as Up in docker compose ps? If not, it crashed — check docker compose logs flask-app. (2) Is app.run(host='0.0.0.0') set in app.py? If Flask binds to 127.0.0.1, it's unreachable from outside the container. (3) Is ports: "5000:5000" in the compose file under flask-app? Without this, no host port is mapped. (4) Is UFW blocking port 5000 on spserver? Check: sudo ufw status.
mysql.connector error — Can't connect to MySQL server on 'mysql'
Three possible causes: (1) MySQL isn't ready yet — check docker compose ps, wait for mysql to show (healthy), then docker compose restart flask-app. (2) DB_HOST is wrong — it must be mysql (the service name), not localhost or an IP. Verify: docker compose exec flask-app env | grep DB_HOST. (3) The two services are not on the same network — confirm they are defined in the same docker-compose.yml file. Run docker compose exec flask-app ping mysql to test connectivity.
ModuleNotFoundError at runtime — package missing inside container
A Python package is installed in the local venv but not in the container. Fix: add the package to requirements.txt, then rebuild: docker compose up -d --build. The container uses only what's in requirements.txt — your local venv has no effect. Always verify by checking: docker compose exec flask-app pip list to see what's actually installed inside the container.
Code changes not reflected after docker compose up -d
Compose reused the existing image. Always use docker compose up -d --build after changing app.py or any file that's COPY'd into the image. Confirm the rebuild: docker images flask-app — the "CREATED" column should show a recent timestamp. Also confirm the new code is inside the container: docker compose exec flask-app cat /app/app.py.
Access denied for MySQL user — wrong credentials
Flask is connecting to MySQL with the wrong user or password. Check: (1) docker compose exec flask-app env | grep DB_ — print the env vars inside the Flask container. (2) Compare DB_USER and DB_PASSWORD with MYSQL_USER and MYSQL_PASSWORD in the mysql service. They must match exactly. (3) If you changed credentials after the volume was created, the old credentials are still in effect — see the Compose guide blocker about MySQL not re-initialising. Fix: docker compose down -v and docker compose up -d to fully reset.
How to run Flask interactively inside the container for debugging
Stop the running flask-app container (docker compose stop flask-app), then start a one-off container with an interactive shell: docker compose run --rm flask-app /bin/bash. Inside, you can run python app.py directly, or python -c "..." to test individual pieces. All env vars from the compose file are set — it's the same environment as the running container. Exit with exit and the container is removed (--rm).
01 — What's next
Future scope
⟶ Things to explore from here
Gunicorn instead of Flask dev server — app.run() uses Flask's single-threaded development server, which is not production-ready. Replace CMD in the Dockerfile with CMD ["gunicorn", "--bind", "0.0.0.0:5000", "--workers", "2", "app:app"]. Add gunicorn to requirements.txt. Gunicorn handles multiple concurrent requests and restarts workers on crash.
Automatic DB schema initialisation — instead of manually running CREATE TABLE, place init.sql in the project and bind-mount it into MySQL: ./init.sql:/docker-entrypoint-initdb.d/init.sql. MySQL runs all .sql files in that directory on first start. The table is created automatically on docker compose up with a fresh volume. Combine with seed data inserts for a fully automated dev environment.
Connection retry logic in Flask — if MySQL is still initialising when Flask starts, the connection attempt fails. Add a retry loop at startup: try to connect, sleep 2 seconds on failure, retry up to 10 times. This makes the app robust to transient startup races even without a perfect healthcheck. Libraries like tenacity make this clean with decorators.
Nginx reverse proxy — next topic — currently Flask is directly exposed on port 5000. Adding Nginx as a reverse proxy in the Compose file puts Nginx on port 80 (and later 443 with TLS). Nginx forwards requests to Flask internally. Benefits: SSL termination, static file serving, access logs, and Flask is no longer directly reachable from the host network.
Flask application factory pattern — as the app grows, restructure app.py to use a factory function: def create_app(): app = Flask(__name__); ...; return app. This makes the app easier to test (create a test instance with a separate config), supports Flask blueprints for splitting routes into files, and is the standard pattern for non-trivial Flask projects.
Flask-SQLAlchemy for DB access — raw mysql.connector is fine for simple queries but becomes cumbersome as the schema grows. Flask-SQLAlchemy adds an ORM layer: define models as Python classes, query with Python rather than raw SQL, and handle connection pooling automatically. Add Flask-SQLAlchemy and PyMySQL (pure-Python MySQL dialect for SQLAlchemy) to requirements.txt.
